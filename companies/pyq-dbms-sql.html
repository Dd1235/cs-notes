PYQ — DBMS & SQL

4 min read · source: company_wise/pyq_dbms_sql.md

Intuit's SQL battery, D. E. Shaw's schema design, and the storage questions from Flipkart/NetApp — answered with the trade-off language interviewers reward.

What is an ORM, and ORM vs hand-written SQL — trade-offs? Intuit

An Object-Relational Mapper (Hibernate, SQLAlchemy, Django ORM) maps classes↔tables and objects↔rows, generating SQL from object operations, managing identity, sessions/units-of-work, and migrations. For: productivity, type safety, protection from injection by default, portable across engines, CRUD for free. Against: generated SQL can be naive — the flagship failure is the N+1 query problem (1 query for posts, N for authors — fix with eager loading/select_related); opaque performance; leaky abstraction for window functions, CTEs, engine hints. The mature answer: ORM for the 90% CRUD, hand-tuned SQL (most ORMs allow raw escape hatches) for hot reporting/analytics paths — measured with EXPLAIN, not vibes.

Walk normalization 1NF → 5NF. Intuit

Full detail: normalization notes. The ladder, one line each: 1NF — atomic values, no repeating groups. 2NF — no partial dependency on part of a composite key (order-line's product_name depends on product alone → out). 3NF — no transitive dependency (customer's city → pincode chain → out). BCNF — every determinant is a candidate key (3NF's edge cases with overlapping keys). 4NF — no independent multi-valued dependencies stored together (a person's phones × emails must not cross-product in one table). 5NF/PJNF — no lossless decomposition remains possible; join dependencies implied only by keys (rare; cyclic constraint schemas). Close with why we stop: each level trades redundancy/anomaly protection for joins — production schemas live at 3NF/BCNF and denormalize deliberately for read paths.

How do NoSQL databases handle concurrency and integrity — which ACID parts get traded? Intuit

Typical distributed NoSQL (Dynamo-style, Mongo, Cassandra): Durability kept (WALs/replication — WAL notes); Atomicity kept only per-document/row, historically not across items (Mongo added multi-document transactions, at a cost); Isolation weakened — no global serializability; per-key MVCC, optimistic version checks (CAS on a version field), or last-write-wins with vector clocks; Consistency (the C both of ACID and CAP) traded for availability — eventual consistency with tunable read/write quorums (R+W>N for read-your-writes — CAP & replication). The framing that scores: it's BASE vs ACID — push conflict resolution to the application (merge carts, CRDTs) in exchange for partition-tolerant availability and horizontal scale.

Write the SQL to flag duplicate rows with GROUP BY/HAVING. Intuit

-- find the duplicated identities
SELECT email, COUNT(*) AS n
FROM users
GROUP BY email
HAVING COUNT(*) > 1;

-- delete all but the earliest copy (the production follow-up)
DELETE FROM users
WHERE id NOT IN (
  SELECT MIN(id) FROM users GROUP BY email
);

Say why HAVING: WHERE filters rows before grouping; HAVING filters groups after aggregation — a WHERE COUNT(*)>1 is illegal because counts don't exist yet. Bonus points: the window-function version (ROW_NUMBER() OVER (PARTITION BY email ORDER BY id) → delete rn > 1) handles "keep newest by timestamp" more flexibly, and a UNIQUE index is how you prevent recurrence.

Indexing on write-heavy tables — the trade-offs. Flipkart

Every secondary index is another B+ tree that every write must also update: an INSERT into a table with 6 indexes does ~7 tree modifications, with page splits, WAL volume, and lock/latch traffic to match. Reads get O(log n) lookups; writes pay per index. Levers to name: index only what queries prove they need (drop unused — pg_stat_user_indexes); composite indexes serve multiple query shapes with one tree (leftmost-prefix rule); partial indexes (WHERE status='active') shrink the write surface; in Postgres, HOT updates avoid index maintenance only if no indexed column changed — so not indexing a hot-updated column is itself an optimization; extreme write rates → LSM-based engines (RocksDB) that turn random index writes into sequential merges.

Design a schema for ball-by-ball cricket (or any real-time sports) data. D. E. Shaw

Model the event as the atomic fact and everything else as dimensions: tournament, team, player, match (teams, venue, date), innings (match, batting team, order), and the heart — delivery: (match_id, innings_no, over_no, ball_in_over) as a composite natural key, plus bowler_id, striker_id, non_striker_id, runs_off_bat, extras + extra_type, wicket_type + dismissed_player_id (nullable), ts. Normalized (3NF: players/teams referenced, never repeated), append-only — which matches the domain: deliveries happen once and never update. Aggregates (scorecards, player stats) are derived, not stored — or maintained as materialized views refreshed per over. Indexing for the query patterns: (match_id, innings_no, over_no, ball_in_over) clustered/for order, (bowler_id, match_id) and (striker_id, match_id) for player timelines. Scale sentence: partition by tournament/season (sharding notes); real-time feed = the insert stream itself.

Database storage engines vs raw filesystem storage — when does each win? NetApp

A storage engine (InnoDB, RocksDB) layers on top of files: B+ trees/LSM organization, WAL + crash recovery, transactions & MVCC, indexes, buffer-pool caching. The filesystem gives you hierarchical names → byte streams, journaling for its own metadata, and OS page-cache — but no transactions across writes, no secondary indexes, no query language. Use raw files for: large immutable blobs (images, video — store the path in the DB), append-only logs, config. Use an engine when you need atomic multi-item updates, point/range queries by content, or concurrent writers with isolation. The hybrid is the norm: metadata in the database, bulk bytes on the filesystem/object store — because databases handle many small hot rows well and filesystems handle huge sequential streams well.
