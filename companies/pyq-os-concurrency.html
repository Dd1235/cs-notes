PYQ — OS: Concurrency & Sync

5 min read · source: company_wise/pyq_os_concurrency.md

The most-repeated PYQ territory across all companies. Repeat questions merged; every answer carries the follow-up depth.

Critical sections & primitives

What is a critical section? Arista D. E. Shaw

A code region touching shared state that must not be executed by two threads at once — otherwise interleavings produce lost updates (count++ is load-add-store; two threads can both load the same value). The requirements textbooks want named: mutual exclusion, progress (no thread outside the section may block entrants), bounded waiting (no starvation). Protection tools, in escalating order: atomics for single variables, mutexes for compound state, and lock-free CAS designs (how locks work).

Mutex vs semaphore — differences, binary vs counting, when to use which? Arista Intuit Qualcomm

The full essay lives at semaphore vs mutex; the interview core: a mutex is mutual exclusion with ownership — the locker must unlock, enabling priority-inheritance and recursion-detection; a semaphore is a counter of permits — any thread may V(), which is exactly what makes it a signaling mechanism, not just a lock. A binary semaphore (0/1) is not a mutex because it lacks ownership: thread A can P() and thread B can V() — that's a feature when used for cross-thread signaling ("data is ready"), a bug when used for locking. A counting semaphore initialized to N gates a resource pool (N connections, 10 players — the Rubrik playground question). Rule: protect state → mutex; signal events or count resources → semaphore; wait for a condition on protected state → mutex + condition variable.

The Reader–Writer problem with semaphores — and how counts don't underflow/overflow. Qualcomm

State: mutex protecting read_count, and w (binary) gating writers. Reader entry: P(mutex); if (++read_count == 1) P(w); V(mutex); — the first reader locks out writers; exit: P(mutex); if (--read_count == 0) V(w); V(mutex); — the last reader lets them back. Writers just wrap in P(w)/V(w). No underflow/overflow because every P(w) is structurally paired with exactly one V(w) — only the first/last reader touches w, and read_count mutations are inside mutex. Known flaw to volunteer: readers-preferred → writer starvation; fix with a turnstile semaphore both classes pass through. Full recipe also in CS:APP ch 12 notes.

Can you eliminate locking entirely and still be safe? D. E. Shaw ThoughtSpot

Yes, in three escalating senses. (1) Share nothing: partition data per thread, communicate by message passing — no shared mutable state, nothing to lock. (2) Immutable data: concurrent reads of never-written data need no sync. (3) True shared mutation without locks needs hardware atomics: CAS/fetch-add build lock-free stacks, queues, counters (locks internals — and the CAS-increment PYQ in C & low-level). Caveats that impress: lock-free ≠ wait-free (individual threads can retry forever); the ABA problem; and memory-ordering rules still apply — "no locks" never means "no synchronization".

Deadlock

What is a deadlock? The four necessary conditions? Prevention? Arista Intuit

Full page: deadlock. Compressed: threads waiting on each other in a cycle, forever. Conditions (all four required): mutual exclusion, hold-and-wait, no preemption, circular wait. Engineering prevention patterns (Intuit phrasing): global lock ordering (kills the cycle — order by address or ID), acquire-all-atomically (std::scoped_lock(m1, m2) — kills hold-and-wait), trylock with backoff (a form of preemption; beware livelock), timeouts + retry at transaction level (the database strategy — detect and abort a victim). Bonus vocabulary: livelock, starvation, and the banker's algorithm as the avoidance approach nobody ships.

Concurrency concepts

Concurrency vs parallelism, from a scheduling perspective. Amex

Concurrency = structuring work as independently progressing tasks — achievable on one core by interleaving time slices. Parallelism = tasks literally executing simultaneously on multiple cores. A single-core machine running 100 goroutines is concurrent, not parallel; a GPU kernel is parallel, arguably not concurrent. Scheduler view: concurrency is a program property (many runnable tasks), parallelism a hardware property (many execution units). Classic line: concurrency is dealing with many things at once; parallelism is doing many things at once.

What is Python's GIL and how does it constrain multithreading? Amex

The Global Interpreter Lock: CPython's single mutex around the interpreter — only one thread executes Python bytecode at a time, regardless of cores. Consequences: CPU-bound Python threads give ~zero speedup (they time-share the GIL); I/O-bound threads work fine (the GIL is released while blocking on I/O, and by C extensions like NumPy doing native work). Escape hatches: multiprocessing (separate interpreters, real parallelism, IPC cost), native-code libraries, or async I/O for concurrency-without-threads. Current events note: CPython 3.13+ ships an experimental free-threaded (no-GIL) build — worth mentioning, not yet the default in production.

The IPC mechanisms — and when each is chosen. Amex Qualcomm

Full table: IPC notes. Interview compression: pipes/FIFOs for parent-child streams and shell pipelines (built-in backpressure); Unix domain sockets for two-way, multi-client local services (Docker, Postgres); shared memory for the highest bandwidth — zero copies, but you bring your own semaphores; message queues for structured, prioritized messages; signals for one-bit async pokes only. Choosing (Amex's "share transactional state securely"): shared state with integrity → shm + semaphore, or better, a local socket to a single owner process (serialization point = no races); across machines → real sockets, and IPC stops being an OS question.

File locks & descriptor tables

How do file locks work — flock vs fcntl? Arista

Both are advisory: they only constrain processes that also ask — nothing stops an ignorant write(). flock(fd, LOCK_SH|LOCK_EX) locks the whole open file description, is simple, and survives fork sharing. fcntl(fd, F_SETLK, struct flock*) does byte-range locks — the struct carries l_type (RDLCK/WRLCK/UNLCK), l_whence/l_start/l_len (the range), and is the POSIX-portable choice databases use. Classic gotcha: POSIX fcntl locks are per-process — closing any FD to the file drops the process's locks on it. Mandatory locking exists but is effectively dead (removed in modern Linux).

Process table, file descriptor table, global open-file table, inode table — how do they connect? Arista

Four layers: each process's entry in the process table holds its FD table (small integers → pointers); FDs point into the kernel-wide open-file table (one entry per open(), holding offset + flags); those entries point at the inode/v-node table (one per file, inode notes). The relationships explain everything: fork/dup2 copy FD-table pointers → parent and child share one offset (same open-file entry); two separate open()s → two entries → independent offsets; flock locks live on the open-file entry, fcntl locks on the inode per-process. Draw the three-layer picture — it's also in CS:APP ch 12 notes.
