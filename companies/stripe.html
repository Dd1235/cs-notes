Stripe

2 min read · source: company_prep/stripe.md

Payments infrastructure with a famously practical loop: parse real data, integrate a real API, fix real bugs. Almost no classic LeetCode — the trap is assuming that makes it easier.

Pipeline

  1. Practical coding screen — multi-part problem with a product feel: string/JSON parsing, validation, business rules that accrete part by part. Passing part 1 cleanly matters more than reaching part 4.
  2. Onsite: integration round — private repo + API docs (sometimes Stripe's own API), full internet allowed; build something that works end to end.
  3. Onsite: bug squash — unfamiliar codebase, failing tests; find and fix. Calm, methodical reading is the whole skill.
  4. Onsite: system design — real-world failure framing: retries, idempotency, partial outages.

What they actually test

  • Code quality under time pressure — descriptive names, small helpers; temp and x are explicitly noted against you.
  • Reading before writing — both bug squash and integration reward orienting in code you didn't write.
  • Failure-mode thinking — what if the API times out mid-payment? Double-charge? This is where DB transactions/idempotency knowledge shines.
  • Language is your choice — fluency in your language beats breadth.

Revise on this site

ACID · Write-ahead logging · CAP & replication · HTTP evolution · TLS · DNS deep dive · The N+1 problem

Practice prompts (representative, paraphrased)

  1. Parse a currency string like "USD:100,EUR:80;USD:20" into totals per currency — then extend to nested formats twice (that's the multi-part rhythm).
  2. Implement idempotent charge creation: same idempotency key twice must not double-charge — where does the key live, and what are the races?
  3. Take a failing test in any OSS repo and practice narrating your search: reproduce → localize → hypothesize → fix.
  4. Design webhook delivery with at-least-once semantics and consumer dedup.
