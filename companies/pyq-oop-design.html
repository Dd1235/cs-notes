PYQ — OOP & Design

5 min read · source: company_wise/pyq_oop_design.md

D. E. Shaw's OOP battery plus Intuit's design questions, with the follow-up layers answered too.

Compile-time vs runtime polymorphism, with examples. D. E. Shaw

Static (compile-time): the call target is resolved during compilation — function/operator overloading (same name, different parameter lists; resolved by signature) and templates (each instantiation is its own concrete function). Zero runtime cost, no common base needed. Dynamic (runtime): the target depends on the object's dynamic type — virtual functions through a base pointer/reference. Animal* a = new Dog; a->speak(); calls Dog::speak even though the compiler only sees Animal*. Cost: one vtable indirection + lost inlining. Deep pages: function overloading, virtual functions. Sharp follow-up: overload resolution happens on the static type — overloading and overriding compose confusingly (hiding), which is what using Base::f; fixes.

How do virtual functions and vtables work under the hood? D. E. Shaw

Per polymorphic class, the compiler emits one vtable — an array of function pointers, one slot per virtual function, derived classes overwriting the slots they override. Per object, a hidden vptr (installed by each constructor — which is why virtual calls inside constructors don't reach the derived class: the vptr still points at the base vtable mid-construction). A call p->f() compiles to: load vptr → index the vtable slot → indirect call: two loads and a jump. Costs worth naming: +8 bytes per object, defeated inlining and branch prediction pressure — why hot loops avoid virtual dispatch. Diagram + const_cast extras: vtables page, memory layout.

The Diamond Problem — and virtual inheritance. D. E. Shaw

B and C both inherit A; D inherits B and C → D contains two A subobjects: d.a_field is ambiguous (compile error — you must write d.B::a_field), and A's constructor runs twice. Virtual inheritance (class B : virtual public A) collapses the copies: B and C store an offset/pointer to a single shared A, and the most-derived class D constructs A directly (B's and C's A-initializers are ignored — the detail that proves you know it). Cost: extra indirection to reach A's members, more complex object layout. The pragmatic close: diamonds are usually a design smell — prefer interfaces (pure-virtual classes, safe to multiply-inherit) and composition.

When should you avoid inheritance? Composition over inheritance. D. E. Shaw

Inheritance is the strongest coupling in the language: it inherits implementation, exposes protected internals, and is fixed at compile time. Avoid it when the relationship is "has-a"/"uses-a", when you'd inherit just to reuse code, or when the base wasn't designed for extension (fragile base class problem). Composition: hold the capability as a member behind an interface — swappable at runtime (strategy pattern), testable via injection, and no LSP landmines (the classic Square-extends-Rectangle failure: a Square that is-a Rectangle breaks setWidth). Inheritance survives for genuine is-a with a stable, polymorphic interface. Pages: inheritance, design patterns.

Smart pointers — unique, shared, weak — and lifecycle management. D. E. Shaw

RAII applied to heap objects: the pointer's destructor frees the object, so every exit path (including exceptions — exceptions page) cleans up. unique_ptr: sole ownership, zero overhead, move-only — the default. shared_ptr: reference-counted ownership; the control block holds the count, atomically updated (that atomicity is its cost). weak_ptr: non-owning observer of a shared object — lock() yields a shared_ptr or null; it exists to break reference cycles (two shared_ptrs pointing at each other never hit zero — parent holds shared_ptr to child, child holds weak_ptr back). Rules that score: make_unique/make_shared over raw new (exception safety, one allocation); raw pointers/references remain fine as non-owning parameters.

When and why use a private constructor? Intuit

Whenever construction must be controlled: the Singleton (one instance ever), factory-only types (validation before existence), named constructors, and utility classes that shouldn't be instantiated. The clean modern singleton — thread-safe by the C++11 static-local guarantee, no locks needed:

class Config {
    Config() = default;                       // private: nobody else constructs
    Config(const Config&) = delete;           // no copies
    Config& operator=(const Config&) = delete;
public:
    static Config& instance() {
        static Config c;                      // constructed once, thread-safe
        return c;
    }
};

Volunteer the criticism: singletons are global state — hard to test, hidden coupling; prefer passing dependencies explicitly and reserve this for genuinely process-wide facilities (loggers, config).

Design Instagram's data model with clean OOP. Intuit

Structure the answer as entities → relationships → behaviors → scale note. Entities: User (id, handle, profile), Post (id, author, media list, caption, timestamp), Media (URL, type, dimensions), Comment, Like, Follow (follower→followee edge with timestamp — a relationship entity, not a field). Relationships: User 1–N Post; Post 1–N Comment; Like and Follow as association tables (M–N edges carrying data). Behaviors as interfaces: FeedGenerator (strategy: chronological vs ranked — swap without touching entities), NotificationService (observer on Like/Comment/Follow events), media handling behind a StorageBackend interface. The scale sentence interviewers wait for: feed = fan-out-on-write to follower inboxes for normal users, fan-out-on-read for celebrities — the hybrid is the standard answer. Then stop — don't design all of Meta unprompted.

Microservices vs monolith — and when to pivot? Service discovery (Eureka), API gateway. Intuit

A monolith is one deployable: simplest to build, debug, and transact within — the right default for small teams. Microservices split by business capability, buying independent deployment and scaling, fault isolation, per-team autonomy ("Conway alignment") at the price of network failure modes, distributed transactions (sagas/outbox instead of ACID), and operational overhead. Pivot triggers: teams blocking each other's releases; components with wildly different scaling needs; build/test cycles too slow; a bounded context needing a different stack. Service discovery (Eureka/Consul): instances are ephemeral, so each registers itself (heartbeats keep it alive, missed beats evict it) and callers query the registry — client-side load balancing picks an instance; DNS is the low-tech alternative. API gateway: the single front door — routing, authN/Z, rate limiting, TLS termination, request aggregation — so cross-cutting concerns live once, not in every service.
