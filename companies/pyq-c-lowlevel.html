PYQ — C & Low-Level

4 min read · source: company_wise/pyq_c_lowlevel.md

Qualcomm's embedded battery, ThoughtSpot's atomic puzzle, Amazon's stack question. Write-the-code questions come with the code.

Little-endian vs big-endian — and a runtime check in C. Qualcomm

Endianness = byte order of multi-byte values in memory. Store 0x12345678: little-endian (x86, most ARM configs) puts the least significant byte at the lowest address (78 56 34 12); big-endian puts the most significant first (12 34 56 78 — also "network byte order", why htonl/ntohs exist). The classic check — alias an int's first byte:

int is_little_endian(void) {
    unsigned int x = 1;
    return *(unsigned char *)&x == 1;   // LE: low byte first → reads 1
}

char* aliasing is the sanctioned way to inspect representation (char is exempt from strict aliasing). A union { int i; char c[4]; } version is the same idea. Follow-up: sizeof/arithmetic never expose endianness — only memory inspection and I/O do.

How do char/uint8_t interact with endianness? Qualcomm

They don't — endianness is invisible for single-byte types; there's no order to a single byte. It bites only when you reinterpret multi-byte values as byte arrays (or vice versa): casting a uint32_t* to uint8_t* walks bytes in memory order, packets arrive big-endian onto little-endian CPUs, and packed protocol structs need explicit hton*/ntoh* conversion. In embedded work this is the #1 source of "works on my board" bugs — always serialize field-by-field with explicit shifts (b[0] = v >> 24; …), which is endian-independent by construction.

Flip all bits of a 32-bit register in O(1), no loops. Qualcomm

reg = ~reg;      // bitwise NOT — one instruction

The point of the question is recognizing ~ (and not confusing it with logical ! or XOR-by-hand). Related one-liners worth having ready: set bit k x |= 1u<<k, clear x &= ~(1u<<k), toggle x ^= 1u<<k, test (x>>k)&1, lowest set bit x & -x, clear lowest set bit x & (x-1) (the Kernighan popcount trick).

malloc() vs calloc() — mechanics, allocation, overhead. Qualcomm

malloc(n): n bytes, uninitialized (reading is UB). calloc(count, size): count×size bytes, zeroed, with a mandated overflow check on the multiplication (malloc(count*size) silently wraps — a real CVE class). Performance subtlety that scores: for large allocations, calloc is often not slower than malloc+memset — the allocator gets fresh pages from the OS which are already zero (demand-zero pages, CS:APP ch 9), so zeroing costs nothing until first touch. For small reused heap blocks, calloc genuinely memsets recycled memory. Family follow-ups: realloc may move (returns new pointer, copies), free(NULL) is a no-op, and malloc(0) is implementation-defined.

sizeof(arr)/sizeof(int) inside a function taking the array as a parameter? Qualcomm

It computes sizeof(int*)/sizeof(int) = 2 on 64-bit — not the element count. Array decay: in a parameter list, int arr[] and int arr[100] are rewritten to int *arr; the array-ness (and its size) never enters the function. The idiom works only where the actual array is in scope: sizeof(arr)/sizeof(arr[0]) in the defining scope. Fixes: pass the length explicitly, or in C++ take T (&arr)[N] by reference / use std::array/std::span — which preserve the size in the type. This is the single most reliable C interview trap; explaining decay by name is what they're listening for.

Thread-safe increment from a CAS primitive, without mutexes. ThoughtSpot

Given bool f(int *p, int expected)-style compare-and-swap (atomically: if *p matches the old value, store the new one; report success), the safe increment is the CAS retry loop:

void atomic_increment(int *p) {
    int old;
    do {
        old = *p;                    // read current
    } while (!cas(p, old, old + 1)); // retry if someone raced us
}

Each failure means another thread advanced *p between the read and the CAS — re-read and try again; the algorithm is lock-free (someone always makes progress) though not wait-free (one thread can retry indefinitely). If the prompt insists on "loop-free", the honest engineering answer is that a bare read-then-CAS is only safe when contention is otherwise excluded — under real races, correctness requires the retry (or a stronger primitive like atomic fetch_add, which is the hardware's loop-free answer: lock xadd on x86). Say the ABA caveat and you're done (locks internals, and your SPSC-queue project is the extended version of this conversation).

Recursion stack vs iterative loop — memory, cache locality, stack overflow. Amazon

Each recursive call pushes a stack frame (return address, saved registers, locals — memory layout): auxiliary memory O(depth), and depth ~10⁵–10⁶ overflows the default 1–8 MB stack — the recursion-specific crash iteration can't have. Iteration keeps O(1) frames; if you need the structure, an explicit heap stack holds just your data, not full frames — smaller and overflow-free (bounded by heap instead). Cache locality: hot loops touch a compact working set; deep recursion strides through fresh frames (though the top-of-stack itself stays cache-warm — a nuance worth voicing). Completing the answer: tail-call optimization can turn tail recursion into a jump (guaranteed in Scheme, opportunistic in C/C++ at -O2, absent in Python/Java — so "recursion is fine if tail-recursive" is language-dependent), and memoized recursion vs bottom-up DP is the same trade dressed up.
