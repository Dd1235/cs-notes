Rubrik

1 min read · source: company_prep/rubrik.md

Data-security/backup infrastructure. The loop's signature: two system-coding rounds that are pure concurrency — write thread-safe code with real mutexes, not pseudocode.

Pipeline

  1. DSA round — one hard-ish problem (graphs/DP territory), standard format.
  2. System coding round ×2 (the deciders) — implement concurrent primitives and thread-safe structures live: blocking queues, custom locks, schedulers. Compiling, running code expected.
  3. HLD + hiring manager — design with a distributed/storage flavour + projects and fit.

What they actually test

  • Producer–consumer patterns cold — bounded buffers, condition-variable discipline (predicate loops, notify choice).
  • Building primitives, not just using them — "implement a read-write lock", "an atomic queue with your own lock".
  • Resource-constrained thinking — e.g. replicate/process large binaries with limited memory (streaming, chunking).
  • OS vocabulary in follow-ups — processes vs threads, memory management, what a mutex costs.

Revise on this site

Multithreading in C++ · Semaphore vs mutex · Locks internals · Concurrent programming (CS:APP 12) · Deadlock · I/O multiplexing · OS refresher

Practice prompts (representative, paraphrased)

  1. Bounded blocking queue with push/pop, condition variables, no busy-wait — then the follow-up: multiple producers, graceful shutdown.
  2. Job scheduler where jobs have parent dependencies; run everything as early as legal, thread pool underneath.
  3. A "playground" usable by one team at a time, ≤10 players concurrently, teams queue fairly — choose and defend your primitives.
  4. Copy a 2 TB file between machines with 1 GB RAM — chunking, checksums, resume-on-failure.
