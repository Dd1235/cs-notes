PYQ — OS: Memory, Paging & Processes

6 min read · source: company_wise/pyq_os_memory.md

Real previous-year questions, tagged by company, repeats merged. Answers include the follow-up territory interviewers wander into.

Architecture basics

What is the difference between a 32-bit and a 64-bit processor/architecture? Arista

The bit-width is the size of the CPU's general-purpose registers and, crucially, of a pointer. A 32-bit machine can address 2³² bytes = 4 GB of virtual address space per process; 64-bit gives 2⁶⁴ (practically 2⁴⁸/2⁵⁷ used — hardware doesn't wire all 64 lines). Surrounding topics they probe next: wider registers move more data per instruction; more registers in x86-64 (16 vs 8) means fewer stack spills; page-table depth grows (4–5 levels) to cover the space; a 32-bit OS cannot use >4 GB RAM without PAE hacks.

How do you determine if a machine is 32-bit or 64-bit in a single return statement, no variables? Arista

int is64() { return sizeof(void *) == 8; }

Pointer size is the architecture width — that's the whole trick. sizeof is compile-time, so this technically reports what the binary was compiled for — a sharp interviewer will accept sizeof(long) on Linux (LP64) but reject sizeof(int) (see next). Shell version: getconf LONG_BIT.

What is sizeof(int) on 32-bit vs 64-bit? Arista

The trap: 4 bytes on both on mainstream platforms. What changes is sizeof(void*) (4 → 8) and, on Linux/macOS, long (4 → 8, the LP64 model; Windows keeps long at 4 — LLP64). Only sizeof(char) == 1 is guaranteed; the standard gives minimum ranges, not sizes. Follow-up ammo: this is why serializing structs across machines needs fixed-width int32_t/int64_t.

Process vs thread

Difference between a process and a thread — memory layout, address spaces, resource usage? Arista Intuit D. E. Shaw

A process is an isolation unit: own virtual address space, page tables, FD table, credentials (full page). A thread is an execution unit inside one: own stack, registers, PC — but shares code, globals, heap, and FDs with siblings. Consequences: thread creation/switch is cheaper (no page-table/TLB switch — see CS:APP ch 9 notes); processes fail independently while one crashing thread kills the whole process; processes communicate via IPC, threads via shared memory + locks. Multiprocessing vs multithreading (D. E. Shaw Amex phrasing): choose processes for fault isolation and security boundaries (Chrome tabs), threads for cheap sharing and low-latency communication (server request pools).

Can the execution of one thread affect other threads? Arista

Yes — that's both the feature and the bug. Legitimately: through shared data, signaling (condition variables), and shared FDs. Illegitimately: data races on shared state, one thread's segfault killing the process, a thread holding a lock stalling everyone (or dying while holding it), unbounded allocation exhausting the shared heap, and cache-level interference (false sharing — memory layout notes). "Local" stack variables are only private by convention — any thread can write another's stack through a leaked pointer.

Virtual memory & paging

What is virtual memory? Arista

The abstraction that each process sees a private, contiguous address space, decoupled from physical RAM. The MMU + page tables translate virtual → physical per access. It buys: isolation (you can't even name another process's memory), overcommit (allocate more than RAM; touch faults it in), sharing (libraries mapped once), and RAM as a cache for disk. Deep dive: CS:APP ch 9 notes.

What is paging and how does it function? Arista Intuit

Chop virtual space into fixed pages (4 KB) and RAM into same-size frames; a per-process page table maps page → frame, and any page can land in any frame. Translation: virtual address = VPN + offset → page-table lookup (cached by the TLB) → PPN + offset. Because mapping is per-page, there's no need for contiguous physical memory — which is exactly what kills external fragmentation (below).

Internal vs external fragmentation — and how paging addresses them? Intuit Qualcomm

External: free memory exists but in scattered pieces too small to use — a disease of variable-size contiguous allocation (segmentation, malloc arenas). Internal: waste inside an allocated unit — you got a 4 KB page but used 1 byte. Paging eliminates external fragmentation for RAM (any frame fits any page) at the price of some internal (average half a page per allocation region). "Virtual memory fragmentation" (Qualcomm): the virtual space itself can fragment too — a 32-bit process doing many mmaps may fail a large allocation despite enough total free space; 64-bit's vast space is the practical cure. Inside malloc, the allocator fights external fragmentation with coalescing and size-class bins (CS:APP ch 9).

What are page faults? Count the faults for a reference string under FIFO. Arista

A page fault = access to a page with no valid frame mapping; the kernel loads it (or kills you — segfault). Counting technique, worked: string 1,2,3,4,1,2,5,1,2,3,4,5, 3 frames, FIFO evicts by load order:

1✗ 2✗ 3✗ | 4✗(evict 1) 1✗(evict 2) 2✗(evict 3) | 5✗(evict 4) 1✓ 2✓ | 3✗(evict 5? no—evict 1) ...
Result: 9 faults with 3 frames, 10 with 4 frames — Belady's anomaly.

Method: track the queue, mark hit/miss per access, count misses — do it in a table, out loud. The kicker fact: more frames can mean more faults under FIFO; LRU can't do that (page replacement notes).

Demand paging and copy-on-write during process creation — step by step. NetApp

fork() copies page tables, not pages: both processes' entries point at the same frames, all marked read-only. A write by either → protection fault → kernel copies that one page, remaps writer to the copy, restores write permission — copy-on-write. Demand paging is the same lazy philosophy at exec/mmap: map regions with no frames at all; first touch faults the page in from the binary/file/zero-pool. Net effect: fork of a 2 GB process is milliseconds, and fork+exec never copies the 2 GB at all.

What exactly happens on a page fault — interrupt to mapped page? NetApp

  1. MMU can't translate → raises a fault, CPU traps to the kernel's fault handler with the faulting address and cause. 2. Kernel checks the process's memory-map metadata (Linux: VMAs): address not in any region → SIGSEGV; valid but not present → legitimate fault. 3. Locate the content: file page → page cache or disk read; anonymous swapped page → swap; first-touch heap → zero page; COW → copy the frame. 4. If RAM is full, evict a victim first (replacement policies, write back if dirty). 5. Update page table + TLB, return from trap — the instruction re-executes, unaware anything happened.

Which hardware translates virtual addresses, and how does the TLB fit? Qualcomm

The MMU (memory-management unit), on the CPU die. Every access must be translated; a multi-level page-table walk costs several memory reads, so the MMU caches recent translations in the TLB (~dozens–thousands of entries). TLB hit → translation is ~free; miss → hardware walks the tables; page-not-present → fault (above). Context switches used to flush the TLB (why process switches cost more than thread switches); modern CPUs tag entries with ASIDs to avoid full flushes. Huge pages (2 MB) exist mostly to stretch TLB reach.
