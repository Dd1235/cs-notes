PYQ — Cloud, Containers & ML

6 min read · source: company_wise/pyq_cloud_ml.md

NetApp's infrastructure set, Flipkart's architecture questions, Amazon's ML trio — the answers that show you understand the machinery, not the buzzwords.

Containers & orchestration

How does Docker sandbox applications — namespaces and cgroups? NetApp

A container is not a VM — it's a normal Linux process wearing two kernel features. Namespaces control what it can see: PID (its own process tree, it thinks it's PID 1), NET (own interfaces/routing table), MNT (own filesystem view), UTS (hostname), IPC, and user (root inside ≠ root outside). Cgroups control what it can use: CPU shares, memory caps (exceed → the container's own OOM kill), I/O bandwidth, PID counts. No guest kernel, so startup is process-fast and density is high; the flip side — containers share the host kernel, so a kernel exploit escapes all of them (why gVisor/Firecracker exist for hostile multi-tenancy). One process view: ps on the host sees container processes; the namespace just renumbers them inside.

How does Docker layer images — OverlayFS? NetApp

Images are stacks of read-only layers (one per Dockerfile instruction, content-addressed, shared across images — ten containers from one image store the base once). At runtime, OverlayFS merges them: lowerdir = the image layers, upperdir = a per-container writable layer, presented as one filesystem. Reads fall through to the highest layer containing the file; writes trigger copy-up (copy the file to the writable layer, modify there); deletes write a whiteout marker masking the lower file. Consequences worth volunteering: writing a 1 GB file inside a container copies it up first (slow — why databases use volumes, which bypass overlay entirely); layer order affects build-cache hits; docker commit just freezes the upper layer into a new read-only one.

What is a Kubernetes Pod — networking and volumes inside it? NetApp

The pod is k8s's atomic scheduling unit: one or more containers that always co-schedule on one node and share a network namespace — one IP per pod, containers reach each other on localhost (which is why two containers in a pod can't bind the same port). Shared volumes mount into all containers of the pod — the mechanism behind sidecar patterns (app writes logs to the shared volume; the shipping sidecar reads them). Under the hood a paused "infra" container holds the namespaces; app containers join it. Pod-to-pod: every pod gets a cluster-routable IP (CNI plugins implement the flat network); Services put a stable virtual IP + load balancing in front of ephemeral pods — pods are cattle, their IPs die with them.

How do cloud load balancers use health checks to distribute traffic? NetApp

Each backend is probed on a defined check — TCP connect, or better HTTP GET /healthz expecting 200 — every N seconds; fail M consecutive → ejected from rotation, pass K → readmitted (thresholds prevent flapping). Distribution across healthy targets: round robin, least-connections, or consistent hashing for cache affinity. The design depth that scores: liveness vs readiness (k8s terms — liveness "restart me", readiness "don't send traffic yet", e.g. during warm-up or draining); deep vs shallow checks (checking your database in /healthz lets one DB blip eject every backend — cascading failure; keep LB checks shallow, alert on deep ones); connection draining on planned removal; and slow-start ramping for newly readmitted instances.

Architecture

Hot vs cold storage tiers — choosing by access pattern. Flipkart

Tiering trades latency against cost per GB: hot (SSD/standard object storage — ms access, premium price) for actively served data; warm/infrequent-access (cheaper per GB, retrieval fee) for occasional reads; cold/archive (Glacier-class — minutes-to-hours retrieval, ~10× cheaper) for compliance and backups. The decision inputs: access frequency and predictability, retrieval SLA, and the often-missed retrieval + early-deletion fees — cold storage read often is more expensive than hot. Production pattern: lifecycle rules age data down tiers automatically (orders: hot 90 days → warm 1 year → archive 7 years), with the metadata/index always hot so you can find things without touching the cold bytes.

Architect a payment orchestrator with retries, dropouts, and state mismatches. Flipkart

The whole problem is exactly-once effect on top of at-least-once delivery. Core pieces: a payment state machine (CREATED → AUTHORIZED → CAPTURED / FAILED / REFUNDED) persisted transactionally; idempotency keys on every mutating call (same key → same result, no double charge — the Stripe page practices this); async retries with exponential backoff + jitter off a durable queue, never in-request; the outbox pattern (state change + outgoing message committed in one ACID transaction, a relay publishes after) so state and messages can't diverge; and reconciliation — a periodic sweep comparing internal state against the gateway's, because timeouts leave you not knowing whether the charge landed (query, don't guess). Fraud check sits as a scoring step before authorization — synchronous rules + async ML scoring, with manual-review queues for the gray zone.

ML quickies

Accuracy vs precision vs recall vs F1 — and when each matters. Amazon

From the confusion matrix: Accuracy = (TP+TN)/all — meaningless under imbalance (99% "no fraud" predictor is 99% accurate and useless). Precision = TP/(TP+FP) — of my alarms, how many were real; maximize when false positives are costly (spam filter eating real mail, arrest decisions). Recall = TP/(TP+FN) — of the real cases, how many I caught; maximize when false negatives are costly (cancer screening, fraud detection — investigate a few extra rather than miss one). F1 = harmonic mean of P and R — a single number when both matter; harmonic, so one bad side tanks it. Complete the answer with the lever: the classification threshold trades P against R; precision-recall curves (not ROC) are the honest picture under heavy imbalance.

Transformer architecture — and what Q, K, V actually do. Amazon

Pipeline: tokens → embeddings + positional encoding (attention itself is order-blind) → stacked blocks of multi-head self-attention + feed-forward, each wrapped in residual connections + layer norm → output head. Attention's mechanics: each token's vector is projected three ways — Query ("what am I looking for"), Key ("what I can be found by"), Value ("what I contribute"). Scores = QKᵀ/√d (the √d keeps softmax gradients sane) → softmax → weights over all tokens → weighted sum of Values becomes the token's new representation. The retrieval analogy lands well: Q is your search phrase, K the documents' titles, V their contents — attention is a soft lookup. Multi-head = several projection sets attending to different relationship types in parallel; causal masking (zeroing future positions) is what makes GPT-style decoders generative. Why it won: attention is O(n²) but fully parallel across the sequence — no recurrent bottleneck — so it scales with hardware.

Decision tree vs random forest — bagging and splitting. Amazon

A decision tree greedily picks, at each node, the feature+threshold that best purifies the split (Gini impurity or entropy/information gain), recursing until stop criteria. One deep tree: interpretable, and a notorious overfitter — high variance, tiny data changes reshape it. A random forest attacks that variance with two randomizations: bagging (each tree trains on a bootstrap sample — drawn with replacement, ~63% unique rows, the rest "out-of-bag" giving free validation) and feature subsampling (each split considers only a random subset, ~√p features — otherwise every tree grabs the same dominant feature and trees stay correlated; decorrelation is the whole point). Predictions: majority vote / average across hundreds of trees — variance averages away, bias stays modest. Trade: loses single-tree interpretability (feature importances partially compensate); gradient boosting (XGBoost) is the sequential, bias-attacking cousin that usually edges it on tabular benchmarks.
