HTTP/1.1 → 2 → 3

2 min read · source: cn/http_evolution.md

One protocol, three generations, each killing the previous generation's bottleneck. The recurring villain: head-of-line blocking, murdered twice, at two different layers.

HTTP/1.1 — one request at a time per connection

Keep-alive reuses the TCP connection (vs 1.0's connect-per-request), but each connection is a strict request→response pipeline. A slow response blocks everything queued behind it — application-level head-of-line blocking. (Pipelining tried to fix this and failed — responses had to return in order; browsers never shipped it on.)

Browser workaround: ~6 parallel TCP connections per host. Site workarounds you should name: domain sharding, sprite sheets, JS bundling — all protocol workarounds masquerading as best practices.

HTTP/2 — multiplexing over one TCP

One TCP connection, many independent streams: requests/responses chopped into frames, interleaved, reassembled. Slow response no longer blocks others — plus header compression (HPACK) and server push (now deprecated — fine to say so).

The catch: TCP underneath is one ordered byte stream (TCP notes). One lost packet stalls ALL streams until retransmission — the blocking moved down a layer: transport-level head-of-line blocking. On lossy networks H2 can lose to H1's six naive connections.

HTTP/3 — QUIC, over UDP

Fix TCP's HOL by leaving TCP: QUIC runs over UDP and reimplements reliability per-stream in userspace, with TLS 1.3 baked in.

  • Lost packet stalls only its own stream — the villain finally dies.
  • Handshake: TCP+TLS's 2–3 RTTs collapse to 1 RTT (0-RTT on resumption).
  • Connection ID instead of the (IP, port) 4-tuple → survive switching Wi-Fi→cellular without reconnecting.
  • Deployment reality: userspace evolution (no kernel/middlebox ossification), but UDP-hostile networks fall back to H2 — which is why every H3 client keeps a TCP fallback.

The table you draw

H1.1 H2 H3
Transport TCP TCP QUIC/UDP
Parallelism 6 connections streams, 1 conn streams, 1 conn
HOL blocking at app layer at TCP layer per-stream only
Setup RTTs (with TLS) 2–3 2–3 1 (0 resumed)

Drill

  1. One image request times out on H1 vs H2 vs H3 — what happens to the other requests in each?
  2. Why did H2 make bundling/sharding anti-patterns overnight?
  3. Why must QUIC live in userspace over UDP rather than as TCP v2? (Middleboxes ossified TCP; UDP passes through; iteration without kernel upgrades.)

Stripe API discussions and system-design rounds touch this constantly.
