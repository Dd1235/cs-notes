TCP - Transmission Control protocol

Pros:

  • acknowledgement
  • guaranteed delivery
  • connection based , need to first establish a conn
  • congestion control (packets get delayed as servers and routers get overwhelmed)
  • ordered packets, one packet can go through china, one through the pacific and so on.

Cons:

  • larger packets
  • need more bandwidth , if you have like 10 Gigabit bandwidth doesnt matter
  • slower than UDP
  • stateful , if your server can restart and client can continue on then you're stateless
  • server memory (dos) server can stackup on useless conn and use up memory

UDP - User Datagram protocol

Cons:

  • no acknowledgement
  • no guaranteed delivery , only checksum
  • connectionless (don't know your client)
  • no congestion control
  • no ordered packets
  • security

Pros:

  • smaller packets
  • less bandwidth
  • faster than tcp
  • stateless

TCP internals

4 tuple -> two ip, two ports -> unique tcp conversation
stream index: 0

sequence numbers - when I send you the tcp syn, I will send you the intial sequence number, wireshark will zero this out

I get back an ack with sequence number incremented once.
For most SDE/backend interviews, if you know the material below deeply, you'll be ahead of 90% of candidates.

1. Why TCP exists

IP gives:

Host A ----> Host B

but does not guarantee:

  • delivery
  • ordering
  • duplicate removal
  • flow control
  • congestion control

TCP adds all of these.

Think:

Application
    ↓
TCP
    ↓
IP

TCP's job:

Turn an unreliable packet network into a reliable ordered byte stream.

2. TCP is a BYTE STREAM

This is extremely important.

Many people think TCP sends messages.

It does not.

Application:

send("hello");
send("world");

Receiver might get:

helloworld

or

hel
loworld

or

hello
world

TCP guarantees:

hello arrives before world

not message boundaries.

UDP preserves packet boundaries.

4. Three-Way Handshake

Before data can flow:

Client:

SYN
seq = x

Server:

SYN ACK
seq = y
ack = x+1

Client:

ACK
ack = y+1

Connection established.

sequenceDiagram
    participant C as Client
    participant S as Server
    Note over C,S: both sides closed
    C->>S: SYN (seq = x)
    Note right of S: server allocates state<br/>(SYN queue)
    S->>C: SYN-ACK (seq = y, ack = x+1)
    C->>S: ACK (ack = y+1)
    Note over C,S: ESTABLISHED — data can flow
    C->>S: HTTP request (seq = x+1)

Why 3 and not 2?

Need both sides to confirm:

  1. I can hear you.
  2. You can hear me.

Otherwise old duplicate packets could create fake connections.

5. Sequence Numbers

Every byte has a sequence number.

Suppose:

seq = 1000

Send:

HELLO

5 bytes.

Receiver expects:

1005

next.

Example:

1000 H
1001 E
1002 L
1003 L
1004 O

ACK:

ack = 1005

means

I've received everything up to 1004.

6. Acknowledgements

TCP uses cumulative ACKs.

Received:

1000-1999

ACK:

2000

means:

Everything before 2000 arrived.

Not:

I only received 1999.

7. Retransmissions

Suppose:

Packet 1 arrives
Packet 2 lost
Packet 3 arrives

Receiver:

ACK 1000
ACK 1000
ACK 1000

repeatedly.

Sender sees duplicate ACKs.

Eventually retransmits missing packet.

This is how reliability works.

8. Sliding Window

TCP doesn't wait after every packet.

Terrible:

send
wait
send
wait

Instead:

send 100 packets
then wait

Window:

[sent but unacked]

Example:

Window size = 5
1 2 3 4 5

ACK for 1 arrives:

2 3 4 5 6

window slides.

Hence:

Sliding Window Protocol

9. Flow Control

Problem:

Sender = fast
Receiver = slow

Receiver buffer:

64 KB

If sender keeps blasting data:

buffer overflow

Receiver advertises:

Window Size = 8192

Meaning:

I can accept 8KB more.

Sender obeys.

Interview answer:

Flow control protects the receiver.

10. Congestion Control

Additive Increase Multiplicative Decrease.

Different problem.

Flow control:

protect receiver

Congestion control:

protect network

Imagine:

1 million senders

all transmitting at full speed.

Routers melt.

TCP slows itself down.

cwnd

Actual sending limit:

Interviewers love this.

Closing requires four-way handshake.

Server:

ACK

Later:

Client:

ACK

Reason:

Each direction closes independently.

TCP is full duplex.

    participant S as Server

UDP

  • 8 byte header instead of 20 byte like in tcp

Think of UDP as the postal service version of networking.

TCP says:

"I will make sure every letter arrives, in order, without duplicates."

UDP says:

"Here's the letter. Good luck."

That sounds simple, but there's still quite a bit happening under the hood.

1. Application writes data

Suppose your game server does:

sendto(sockfd, "hello", 5, ...);

The kernel receives 5 bytes:

h e l l o

and builds a UDP packet around them.

2. UDP Header is added

UDP adds only an 8-byte header:

0      15 16     31
+--------+--------+
| SrcPort| DstPort|
+--------+--------+
| Length |Checksum|
+--------+--------+

Example:

Source Port      = 50000
Destination Port = 53
Length           = 13
Checksum         = ...

where:

13 = 8 bytes header + 5 bytes data

Packet now becomes:

UDP Header
    src=50000
    dst=53

Payload:
    hello

3. IP Layer wraps it

UDP itself cannot route packets.

The IP layer wraps the UDP segment:

IP Header
UDP Header
Payload

Example:

Src IP = 192.168.1.10
Dst IP = 8.8.8.8
Protocol = 17 (UDP)

Protocol number 17 tells the receiver:

"This payload belongs to UDP."

4. Find MAC Address (ARP)

The machine now asks:

"How do I reach 8.8.8.8?"

Usually through the router.

If it doesn't know the router's MAC:

Who has 192.168.1.1?

Router replies:

I do.
MAC = AA:BB:CC:DD:EE:FF

ARP cache gets updated.

5. Ethernet Frame Created

Network card receives:

Ethernet Header
IP Header
UDP Header
Payload

Example:

Dst MAC = Router MAC
Src MAC = My MAC

Frame goes onto the wire.

6. Routers Forward It

Each router only looks at:

IP Header

Specifically:

Destination IP
TTL

The UDP contents are mostly ignored.

Router:

Receive frame
↓
Remove Ethernet header
↓
Inspect IP
↓
Choose next hop
↓
Create new Ethernet frame
↓
Forward

Notice:

MAC changes every hop

but

Source/Destination IP remain unchanged

until the final destination.

7. Receiver Gets Packet

At Google's DNS server:

NIC receives frame.

Kernel strips:

Ethernet
IP

and sees:

Protocol = UDP

So it hands payload to UDP.

UDP sees:

Destination Port = 53

and delivers it to the DNS process listening on port 53.

What UDP DOESN'T Do

UDP never:

  • acknowledges packets
  • retransmits packets
  • guarantees ordering
  • detects duplicates
  • performs congestion control
  • maintains a connection

There is no:

SYN
ACK
FIN

like TCP.

No connection state.

Every packet is independent.

Example: Packet Loss

Suppose sender transmits:

Packet 1
Packet 2
Packet 3
Packet 4

and Packet 3 disappears.

Receiver sees:

1
2
4

UDP simply delivers:

1
2
4

to the application.

No complaint.

No retransmission.

The application must decide what to do.

Games often just ignore the loss.

DNS often retries.

QUIC implements reliability itself on top of UDP.

Why Use UDP Then?

Because all the TCP machinery costs time.

TCP must maintain:

  • sequence numbers
  • ACKs
  • retransmission timers
  • congestion windows
  • receive windows
  • connection state

UDP avoids all of that.

A UDP send can be roughly:

Application
   ↓
UDP
   ↓
IP
   ↓
NIC

Very little bookkeeping.

When you type:

nslookup google.com

typically:

Imagine your laptop has:

  • Chrome

The IP address identifies the machine:

The port identifies the application:

The kernel performs a lookup like:

This is one of the most common networking interview questions.

The short version:

MTU = Maximum size of an IP packet on a link

MSS = Maximum TCP payload in one TCP segment

MSS is derived from MTU.

MTU (Maximum Transmission Unit)

MTU is a property of the network link.

For Ethernet, the common value is:

1500 bytes

Meaning:

IP Header
+ TCP/UDP Header
+ Data
<= 1500 bytes

must fit into one Ethernet frame.

If it doesn't fit:

2000-byte IP packet

then either:

  • fragmentation happens (bad)
  • packet gets dropped and sender learns to use smaller packets

Example:

MTU = 1500

IP Header = 20 bytes
TCP Header = 20 bytes

Remaining payload:
1500 - 20 - 20
= 1460 bytes

MSS (Maximum Segment Size)

MSS applies only to TCP.

It tells:

"What's the largest chunk of application data I should put in one TCP segment?"

For a normal Ethernet path:

MTU = 1500

IP Header = 20
TCP Header = 20

MSS = 1460

Visualization:

|<------------ MTU = 1500 ----------->|

+----------+-----------+-------------+
| IP Header|TCP Header | TCP Payload |
| 20 bytes | 20 bytes  | 1460 bytes  |
+----------+-----------+-------------+

The payload part is MSS.

Why does TCP care about MSS?

Suppose application writes:

10000 bytes

TCP will split:

1460
1460
1460
1460
1460
1460
1240

instead of generating huge packets.

How does MSS get chosen?

During the TCP handshake:

SYN

contains an MSS option.

Example:

Client:
MSS = 1460

Server:
MSS = 1460

Each side tells the other:

"Please don't send me TCP payloads larger than this."

What about UDP?

UDP does not have MSS.

You can do:

sendto(..., 5000);

and create a 5000-byte UDP datagram.

But then IP may need to fragment it.

This is one reason large UDP packets are often avoided.

Common interview numbers

Ethernet:

MTU = 1500

IPv4:

IP Header = 20 bytes
TCP Header = 20 bytes

MSS = 1460

IPv6:

IP Header = 40 bytes
TCP Header = 20 bytes

MSS = 1440

Easy way to remember

Think of a shipping box 📦.

MTU = size of the entire box.

+-------------------+
| headers + payload |
+-------------------+

MSS = usable space after TCP/IP headers are already occupying part of the box.

Box size (MTU)
    -
Header space
    =
Payload space (MSS)

So in interviews:

MSS = MTU - IP Header - TCP Header

(ignoring optional headers).

That's the formula people expect you to know.

HTTP/2 and QUIC

  • http is a stateless protocl
  • hypertext transfer protocol
  • http 1.0 - establish a connection over tcp, then immediately ask for a file, get it, shut down tcp connection
  • http 1.1 - pipelining, send over your png file, and other file, multiple requests one after other in a single tcp stream.
  • head of line blocking at the http layer. Can work around by opening a bunch of tcp connections.
  • http 2 - set up streams on top one tcp connection. each stream is like a tcp connection of its own. allows you to multiplex. multiple requests over one tcp connection.
  • each request gets a stream id. Everything is chopped into frames and interleaved on the wire. Each individual image may finish later than the first image in http 1.1 But image 2 starts immediately, css/js don't wait behind giant images. total page load is much better.
  • in http 1.1, 5 mb image, 100 kb cs, 100 kb js, instead of interleaving, in 1.1 the image starts and everything waits.
  • there is SETTINGS_MAX_CONCURRENT_STREAMS sent bt the server.
  • but tcp is still a byte stream. if packet b is lost, packet c cannot be delivered, until packet b is retransmitted.
  • so lost packet can stall every stream, tcp level head of line blocking
  • http 3 uses quic andd udp. quic has independent streams. if image stream packet lost only that stream waits. css js html stream can continue.

Interview summary:

HTTP/1.1 often needed multiple TCP connections for parallelism.
HTTP/2 uses one TCP connection and multiplexes many logical streams via frames.
Frames from different streams are interleaved on the wire.
Streams are limited by SETTINGS_MAX_CONCURRENT_STREAMS.
HTTP/2 removes HTTP-layer HOL blocking but still suffers TCP-level HOL blocking.
HTTP/3/QUIC removes that remaining TCP HOL blocking by providing independent streams over UDP.

  • quic is the newest transport layer protocol

Most important HTTP concept.

Meaning:

1. HTTP Caching

Imagine loading YouTube.

   |

   |

   |

Wasteful.

Instead:

   |

No network request.

Can store, but must validate before using.

Never store.

Useful for:

Bank statements
Medical records

Just:

Still valid.

Huge bandwidth savings.

4. HTTP/1.1 HOL Blocking

Suppose:

Image1
Image2
Image3

over one TCP connection.

TCP guarantees ordering.

If packet 5 lost:

1 2 3 4 X 6 7 8

TCP holds:

6 7 8

until 5 arrives.

Application sees nothing after 4.

This is called:

Head-of-Line Blocking

HTTP/2 solved application-level HOL blocking:

multiple streams.

But all streams still share:

one TCP connection

Loss affects everyone.

5. Why HTTP/2 Still Suffers

Imagine:

Stream B -> css
Stream C -> js

All multiplexed.

TCP packet carrying image data lost.

TCP must preserve order.

Therefore:

CSS waits
JS waits
Image waits

even though only image packet was lost.

This is transport-layer HOL blocking.

7. Why UDP?

Because TCP is implemented in kernel.

Cannot easily modify.

QUIC:

Implemented in user space

Runs over UDP.

Gives Google freedom to innovate.

8. QUIC Streams

Key idea:

Each stream has independent ordering.

Imagine:

Connection
 ├─ Stream 1 (image)
 ├─ Stream 2 (css)
 └─ Stream 3 (js)

Packet carrying image lost.

Only:

Stream 1

waits.

CSS continues.

JS continues.

This eliminates transport-layer HOL blocking.

Example

TCP:

Packet:
1 2 3 X 5 6

Need:

1 2 3 4 5 6

before delivering.

Everyone waits.

QUIC:

Stream A:
1 2 X 4

Stream B:
1 2 3 4

Stream B can continue.

Only Stream A pauses.

  • ACKs
  • retransmissions
  • congestion control

inside QUIC itself.

Effectively:

Reliable TCP-like transport

built atop UDP.

The Interview Summary

If someone asks:

Why HTTP/3?

Answer:

HTTP/2 multiplexes streams over a single TCP connection. Packet loss in TCP causes transport-layer head-of-line blocking because TCP delivers bytes in order. HTTP/3 uses QUIC over UDP, where streams are independently managed. Loss in one stream does not block others, reducing latency and improving performance on lossy networks.

If they ask:

How does QUIC remain reliable over UDP?

Answer:

QUIC implements acknowledgements, retransmissions, congestion control, flow control, encryption, and stream multiplexing in user space on top of UDP.

That answer alone is usually enough to impress in a backend/networking interview. 🚀

One subtle detail many candidates miss: QUIC does not eliminate packet loss. It eliminates unnecessary waiting caused by unrelated streams sharing TCP's single ordered byte stream. That distinction is exactly what interviewers are looking for.

Excellent questions. These are exactly the questions that get into the design tradeoffs of QUIC.

If QUIC is in user space, how is it safe?

Remember, "user space" doesn't mean "bypass the OS".

The OS still controls:

  • memory protection
  • process isolation
  • network access
  • scheduling

QUIC is just a program using UDP sockets.

Chrome
  |
QUIC implementation
  |
UDP socket
  |
Kernel
  |
NIC

Chrome cannot suddenly access another process's memory.

It still calls:

sendto()
recvfrom()

through the kernel.

The kernel remains the security boundary.

Why move transport logic out of the kernel?

Updating TCP is painful.

To improve TCP:

Linux kernel update
Windows kernel update
Router compatibility
Middlebox compatibility

takes years.

With QUIC:

Update Chrome
Update Firefox
Update Server

done.

Google can ship a new congestion algorithm next week.

No kernel upgrade needed.

Does QUIC implement flow control?

Yes.

Actually two kinds.

Stream flow control

Per stream.

Stream 1 -> 10 MB image
Stream 2 -> CSS

You may want:

Allow Stream 2
Pause Stream 1

without affecting everything.

Connection flow control

Across all streams combined.

Example:

Connection:
  Stream A = 50 MB
  Stream B = 20 MB

Maybe receiver only wants:

100 MB

buffered.

QUIC enforces that.

Very similar to TCP receive windows.

Does QUIC implement congestion control?

Yes.

Otherwise internet would melt. 🔥

QUIC implements things like:

  • Cubic
  • BBR
  • Reno

depending on implementation.

Same idea as TCP:

Network congested
↓
Slow down

Does QUIC have a slower connection than TCP?

Usually the opposite.

Connection establishment is often faster.

TCP + TLS 1.3:

TCP handshake
TLS handshake

roughly:

1 RTT + 1 RTT

(implementation dependent)

QUIC:

Handshake
Encryption negotiation

combined.

Often:

1 RTT

or

0 RTT

for repeat connections.

So startup latency is often lower.

Is QUIC connection-oriented?

Yes.

This surprises people because UDP itself is connectionless.

UDP:

send packet
send packet
send packet

No concept of a connection.

QUIC builds one.

QUIC connection:

Connection ID
Streams
Flow control
ACKs
State

all maintained by QUIC.

So from an application perspective:

QUIC is connection-oriented.

Is QUIC a byte stream protocol?

This is where it differs from TCP.

TCP:

Hello World

becomes:

byte 0
byte 1
byte 2
...

TCP only understands bytes.

QUIC understands streams.

Connection
   |
   +-- Stream 1
   +-- Stream 2
   +-- Stream 3

Each stream is independently ordered.

So QUIC is not merely:

one giant byte stream

like TCP.

It's more accurate to think:

QUIC provides multiple reliable ordered byte streams within one connection.

Does QUIC always guarantee reliability?

No.

This is a very important design point.

QUIC supports different delivery semantics.

For HTTP/3:

Reliable
Ordered

because web pages need correctness.

But QUIC also supports things like:

  • unreliable datagrams
  • partially reliable delivery

through extensions.

Example:

Voice chat.

Suppose packet 100 is lost.

Would you rather:

Option A

Wait 200ms for retransmission.

or

Option B

Skip packet 100 and play packet 101.

Real-time applications usually want B.

QUIC can support that.

TCP cannot.

Do we choose which guarantees to add?

Exactly.

Think of UDP as:

Raw ingredients

and QUIC as:

A transport toolkit

You can build:

Reliable ordered stream

or

Unreliable datagrams

or

Multiplexed streams

over the same connection.

HTTP/3 chooses:

Reliable
Ordered
Multiplexed
Encrypted

because that's what web traffic needs.

A comparison interviewers love:

Feature TCP QUIC
Reliable Yes Yes
Ordered delivery Yes Yes
Congestion control Yes Yes
Flow control Yes Yes
Multiplexed streams No Yes
Kernel implementation Yes Usually
User-space implementation No Yes
Connection migration No Yes
HOL blocking between streams Yes No
Runs over UDP No Yes

The key mental model is:

QUIC is not "UDP but faster."

It is closer to:

"A modern transport protocol that reimplements most of TCP's good properties, adds stream multiplexing and connection migration, and ships in user space over UDP so it can evolve much faster."
