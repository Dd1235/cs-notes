I/O Multiplexing — select, poll, epoll

6 min read · source: cn/3.md

  • IO multiplexing

  • IO with many sources

  • server

  • read / receive from socket is a blocking operation

  • thread per client does not scale

  • io monitoring calls: select/poll/epoll api

  • pass the file descriptors, when there is a data, some of the sockets are ready for io

This is one of the most common OS/networking interview topics.

The problem they solve is:

"I have 10,000 sockets. How do I know which ones have data available without creating 10,000 threads?"

The Problem

Suppose a server has:

Client1
Client2
Client3
...
Client10000

connected.

Each connection has a socket FD:

4
5
6
...
10003

You want to know:

Which socket is ready to read?

without doing:

read(fd1);
read(fd2);
read(fd3);
...

because:

  1. Most sockets are idle.
  2. Blocking on one socket prevents checking others.

Naive Solution: One Thread Per Connection

Thread 1 -> Client 1
Thread 2 -> Client 2
Thread 3 -> Client 3
...

Works for small numbers.

Problems:

  • memory
  • context switches
  • scheduler overhead

10k connections becomes painful.

I/O Multiplexing

Idea:

One thread
Many sockets

Ask kernel:

"Tell me which FDs are ready."

Kernel waits.

When something becomes ready:

socket 57
socket 108
socket 902

Kernel returns those.

select()

Oldest solution.

select(...)

You give:

fd 4
fd 5
fd 6
...
fd N

Kernel checks them.

Conceptually:

Application
   |
select()
   |
Kernel scans
   |
Ready FDs returned

Example

fd_set readfds;

FD_SET(sock1, &readfds);
FD_SET(sock2, &readfds);

select(...);

Kernel modifies set:

sock2 ready

Problems with select()

1. O(n) Scan

Every call:

Check fd1
Check fd2
Check fd3
...
Check fd10000

Even if only one socket is active.

2. Copy Overhead

Every call:

User -> Kernel
Kernel -> User

copies FD sets.

3. FD Limit

Usually:

FD_SETSIZE

often:

1024

Classic interview point.

poll()

Improved version.

Uses:

poll(...)

instead of bitsets.

Instead of:

fd_set

you provide:

struct pollfd[]

Example

pollfds[0].fd = sock1;
pollfds[1].fd = sock2;

Kernel fills:

POLLIN
POLLOUT

etc.

Advantages:

  • no 1024 limit
  • cleaner API

But...

poll() still has O(n)

Every call:

Scan all FDs

Again.

For:

100,000 sockets

still expensive.

Why O(n) Is Bad

Suppose:

100,000 sockets

Only:

1 socket

receives data.

Kernel still scans:

100,000 entries

every time.

Wasteful.

epoll()

Linux solution.

This is what interviewers usually care about.

Idea:

Don't ask the kernel to scan every FD every time.

Instead:

Register interest once.

Create epoll instance:

epoll_create1(...)

Register FDs:

epoll_ctl(...)

Example:

socket 1
socket 2
socket 3

added once.

Wait:

epoll_wait(...)

Kernel returns only active sockets.

Conceptually

select/poll:

Every call:
scan all sockets

epoll:

Register once

Kernel tracks readiness

Return only ready sockets

Why epoll Is Faster

Suppose:

100,000 sockets

Only:

socket 987

has data.

select/poll:

scan 100,000
find 1

epoll:

ready list already maintained

returns:

socket 987

directly.

This is often described as:

O(number_of_ready_fds)

instead of:

O(total_fds)

(technically a bit more nuanced, but good enough for interviews).

Internal Picture

select:

User
 |
fd list
 |
Kernel scans
 |
return

every call.

epoll:

Register FD once
       |
Kernel stores interest list
       |
Ready events added internally
       |
epoll_wait returns ready list

        RL --> W
    end

Level Triggered vs Edge Triggered

Very common follow-up.

Level Triggered (default)

If data exists:

Socket buffer:
HELLO

you'll keep getting notifications.

Readable
Readable
Readable
Readable

until data consumed.

Example:

HELLO

Read only:

HE

Remaining:

LLO

Kernel continues notifying.

Easy to use.

Edge Triggered

Notify only on transition.

No Data
  ↓
Has Data

one notification.

Example:

HELLO

Kernel:

Readable!

once.

You read:

HE

leave:

LLO

No further notification.

You must drain socket completely.

Interview answer:

Edge-triggered reduces notification overhead but requires non-blocking sockets and careful handling to avoid missing data.

Example Server

Nginx:

1 worker
   |
epoll_wait()
   |
thousands of connections

Node.js

Under Linux:

libuv
  ↓
epoll

Redis

Uses:

epoll

on Linux.

Complexity Summary

Mechanism FD Limit Scan Cost
select ~1024 O(n)
poll No practical limit O(n)
epoll Huge O(ready_fds)

Interview Answers

Why is poll better than select?

  • No FD_SETSIZE limit
  • Cleaner API

But still O(n).

Why is epoll better than poll?

Because epoll registers FDs once and the kernel maintains a ready list, avoiding a full scan of all descriptors on every wait.

Why does Nginx use epoll?

Because it can efficiently handle tens or hundreds of thousands of mostly idle connections with a small number of threads.

What is I/O Multiplexing?

I/O multiplexing allows a single thread to monitor many file descriptors and be notified when any become ready for I/O, avoiding one-thread-per-connection designs.

That's usually enough depth for Google, Rubrik, DE Shaw, Databricks, or similar backend interviews.

  • epoll data structure in kernel, red black tree for ready fd.

  • call epoll_ctl once per socket, add each to my epoll instance

  • give back each of these that are intersted

  • interest list is red black tree

  • ready list is a doubly linked list

  • default: level trigerred vs edge triggered

  • epoll_wait -> for each event, dispatch event, and handle

  • nginx (reverse proxy) , redis, node.js libuv runtime, haproxy (l4/l7 load balances) all run on epoll_wait()

  • every epoll_wait() read() write() is a syscall

  • io_uring - ring buffers in memory, even the syscalls you optimize
