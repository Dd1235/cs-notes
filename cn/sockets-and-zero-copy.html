Normally:

No.

Only one listening socket can own a given:

(IP address, Port, Protocol)

tuple.

For example:

TCP 0.0.0.0:8080

If Process A has already bound it:

bind(..., 0.0.0.0:8080)

then Process B trying the same thing gets:

EADDRINUSE
Address already in use

Why?

When a packet arrives:

192.168.1.10:8080

the kernel must know:

Which process should receive this?

If multiple processes owned the same listening socket, the answer would be ambiguous.

But there are exceptions

1. Different protocols

TCP and UDP are separate namespaces.

This is perfectly legal:

Process A:
TCP  :8080

Process B:
UDP  :8080

because:

(TCP, 8080)

and

(UDP, 8080)

are different.

2. Different IPs

Suppose machine has:

127.0.0.1
192.168.1.100

Then:

Process A:
127.0.0.1:8080

Process B:
192.168.1.100:8080

is legal.

The complete key is:

(IP, Port, Protocol)

not just port.

3. SO_REUSEPORT

Modern Linux supports:

setsockopt(fd,
           SOL_SOCKET,
           SO_REUSEPORT,
           ...);

Now multiple processes can bind:

0.0.0.0:8080

simultaneously.

Example:

Worker 1
Worker 2
Worker 3
Worker 4

all listen on:

:8080

Kernel distributes incoming connections among them.

Why use this?

Multi-core scalability.

Instead of:

One process accepts
then hands work to others

you get:

Kernel
   |
   +--> Worker 1
   +--> Worker 2
   +--> Worker 3

directly.

Nginx and other high-performance servers use techniques like this.

4. Fork after listening

Very common.

Parent:

socket()
bind()
listen()

then:

fork();
fork();
fork();

All children inherit the listening socket.

Parent
  |
fork
  |
+---- Worker A
+---- Worker B
+---- Worker C

All share the same kernel socket object.

This is not multiple independent binds.

It's multiple processes sharing one listening socket.

What happens after accept()?

Suppose client connects:

Client
   |
Server:8080

Kernel creates a connected socket identified by:

(src IP, src port,
 dst IP, dst port)

Example:

(10.0.0.5, 52000,
 192.168.1.100, 8080)

Another client:

(10.0.0.6, 53000,
 192.168.1.100, 8080)

These are different connections.

That's why thousands of clients can connect to the same server port simultaneously.

Interview answer

Can two processes listen on the same IP/port?

Usually no, because the kernel allows only one bind for a given (IP, port, protocol) tuple.

Exceptions:

  1. Different protocols (TCP vs UDP)
  2. Different IP addresses
  3. SO_REUSEPORT
  4. Shared listening socket inherited via fork()

A related interview question:

If only one process can listen on port 80, how does Nginx use multiple worker processes?

Answer:

One listening socket is created, then workers inherit it (or use SO_REUSEPORT), and the kernel distributes incoming connections among them.

A socket is a kernel-managed communication endpoint exposed to a process through a file descriptor. It encapsulates network state such as the protocol (TCP/UDP), local and remote addresses, ports, buffers, and connection state. Processes interact with sockets using system calls like socket(), bind(), listen(), accept(), connect(), send(), and recv().

This is a very common systems/backend interview topic because it touches:

  • OS
  • networking
  • kernel/user space
  • performance

Why bring file data into user space if the application isn't modifying it?

For a static file server:

  ↓

Not exactly.

Data still moves:

 ↓

No CPU-mediated copies between kernel and user buffers.

Zero extra copies.

Without DMA

Write byte

With DMA

Interview Definition

A concise answer:

Zero copy is an optimization that avoids unnecessary copying of data between kernel space and user space. Instead of moving bytes through application buffers, the kernel uses techniques such as sendfile(), mmap(), and DMA to transfer data directly between storage, memory, and network devices, reducing CPU overhead and memory bandwidth usage.

Interview Follow-up

If asked:

"How does sendfile() achieve zero copy?"

Answer:

Instead of read() followed by write(), the application asks the kernel to transfer data directly from the file cache to the socket. The data never enters user-space buffers, eliminating kernel-to-user and user-to-kernel copies.

One subtle thing many candidates miss

When people say:

Zero Copy

they usually mean:

Zero CPU copies

not

Zero movement of bytes

The bytes still travel:

Disk → RAM → NIC

The optimization is that the CPU doesn't repeatedly copy those bytes between buffers. That's where the big performance win comes from.
