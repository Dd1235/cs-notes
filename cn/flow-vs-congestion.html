Flow Control vs Congestion Control

6 min read · source: cn/4.md

This is one of the most common networking interview topics. A lot of candidates memorize buzzwords like "sliding window" and "slow start" but can't explain why they exist.

The key is:

Flow control protects the receiver.

Congestion control protects the network.

Those are completely different problems.

Big Picture

TCP Solution

Receiver tells sender:

Sliding Window

Core interview concept.

Suppose sender can have:

5 packets

outstanding.

[1][2][3][4][5]

sent.

ACK for packet 1 arrives.

Window slides:

[2][3][4][5][6]

Hence:

Sliding Window Protocol

Without it:

Send packet
Wait ACK
Send packet
Wait ACK

Very slow.

With sliding windows:

Send many packets
Wait later

Pipeline stays full.

TCP maintains:

Congestion Window.

This is:

How Does TCP Detect Loss?

Two major ways.

Timeout

Packet sent.

No ACK.

Wait.

Wait.

Wait.

Timeout.

Packet lost.

Very expensive.

Triple Duplicate ACK

Much more important.

Suppose sender sent:

1
2
3
4
5

Packet 3 lost.

Receiver gets:

1
2
4
5

TCP delivers in order.

So receiver repeatedly says:

ACK 3
ACK 3
ACK 3

Meaning:

Still waiting for byte 3.

Three duplicate ACKs:

ACK3
ACK3
ACK3

Sender infers:

Packet 3 probably lost.

Fast Retransmit

Immediately retransmit.

No timeout needed.

Huge performance improvement.

Fast Recovery

Instead of:

Reset to 1

which is expensive,

TCP assumes:

Network mostly okay.

Reduce window somewhat and continue.

Why Does TCP Need Congestion Control?

Imagine 100 senders.

Without congestion control:

Everyone sends faster
↓
Router drops more packets
↓
Everyone retransmits
↓
More packets
↓
More congestion

Network collapse.

This happened on the early Internet.

Interview Question

What is the difference between flow control and congestion control?

Perfect answer:

Flow Control Congestion Control
Protect receiver Protect network
Receiver driven Sender driven
Uses rwnd Uses cwnd
Prevent buffer overflow Prevent router overload

Window Formula

Most important equation:

\text{Send Window}=\min(\text{rwnd},\text{cwnd})

Meaning:

Sender cannot exceed:

  • receiver capacity (rwnd)
  • network capacity (cwnd)

Modern TCP

Interviewers may ask:

Is TCP still AIMD today?

Mostly yes conceptually.

But modern variants include:

  • TCP Reno
  • TCP New Reno
  • TCP Cubic (Linux default)
  • BBR (Google)

What is BBR?

Traditional TCP:

Loss => congestion

BBR:

Measure bandwidth
Measure RTT
Estimate bottleneck

Instead of waiting for loss.

Google uses it heavily.

30-Second Interview Answer

Flow control prevents overwhelming the receiver using the advertised receive window (rwnd). Congestion control prevents overwhelming the network using the congestion window (cwnd). TCP uses a sliding window and can have up to min(rwnd, cwnd) bytes in flight. Congestion control starts with slow start, grows exponentially, transitions to congestion avoidance with additive increase, and reacts to packet loss using multiplicative decrease, fast retransmit, and fast recovery.
