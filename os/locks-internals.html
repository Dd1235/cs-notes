How Locks Actually Work

2 min read · source: os/locks_internals.md

"Use a mutex" is the answer at the app layer. One layer down, the interview question becomes: build the mutex.

Why disabling interrupts isn't it

Single-CPU textbooks disable interrupts around critical sections. Real answer: doesn't work on multicore (other CPUs keep running), and userspace can't be trusted with the interrupt switch. Mentioning this earns points.

Attempt 1 — spinlock via test-and-set

The hardware gift: atomic read-modify-write instructions.

while (test_and_set(&lock->flag) == 1)
    ;   // spin: someone else holds it
// critical section
lock->flag = 0;

test_and_set returns the old value and writes 1 — atomically. If you got 0 back, you won the lock. Correct, but spinning burns a full CPU while waiting; on one core, a spinner can waste its whole slice waiting for a holder who can't run.

Attempt 2 — compare-and-swap (CAS)

CAS(addr, expected, new): write new only if current value equals expected; report success. Strictly more powerful — the building block of lock-free structures (your SPSC queue notes are CAS's cousins). A CAS loop replaces "lock, modify, unlock" with "read, compute, try to swap, retry on failure".

Attempt 3 — stop spinning, sleep

Spinlocks waste cycles when hold times are long. The fix: on contention, put the waiter to sleep on a queue and wake it at unlock. That's a blocking mutex.

Linux futex = the hybrid everyone actually ships: try a userspace atomic first (uncontended lock = zero syscalls, nanoseconds); only on contention call futex_wait to sleep in the kernel. Fast path cheap, slow path fair.

Analogy: spinlock = standing at a busy bathroom door jiggling the handle; mutex = writing your name on the whiteboard and sitting down; futex = jiggle once, then write your name.

When to spin anyway

Multicore + critical sections shorter than a context switch (a few hundred ns) → spinning wins; sleeping costs two switches. Long or unpredictable hold times → block. Say the trade-off, name pthread_spinlock vs pthread_mutex, done.

Reference: OSTEP ch 28 — Locks.

Drill

  1. Implement a spinlock given only CAS. Now add exponential backoff — why does it help?
  2. Why is an uncontended futex lock syscall-free, and why does that matter for real programs? (Most locks are usually uncontended.)
  3. Ticket lock vs test-and-set lock — what fairness problem does the ticket fix?
