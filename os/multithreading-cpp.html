Join and Detach with Joinable

  • once a thread is tarted we wait for it to finish t1.join()
  • double join will result into program termination
  • if needed we should check thread is joinable before join

constructing std::thread calls pthread_create(...), then pthread_join(thread_handle, nullptr), os keeps metadata for every thread in thread contorl block, when worker finishes, os marks the thread as terminated, wakes any thread waiting in join. without joining, os thread objects owns as os thread, destroying thread object will leave os thread orphaned.

join can only be called once because after t.join() the thread no longers exists, resources are reclaimed.

hence do if(t1.joinable()) t1.join()

Detach - detach newly created thread from parent thread

Check if a thread is joinable and only detach if its detachable.

A thread starts its life in a joinable state ----> there is an actual os thread running and thread object is owner of that thread. You can release that ownership either using join() or detach(). Join -> wait till work() finishes, os resources are cleaned up, t no longer owns a thread, and joinable becomes false.

After detach(), t stops owning the thread, thread continues running independently, "go do your thing, I won't track you"

Mutex

  • race conditions is a situation where two or more threads/process change a common data at the same time
  • if there is a race condition, protected section is called critical section
  • mutex is used to avoid race condition
  • we use lock(), unlock() on mutex to avoid race condition.

try_lock

try_lock() returns immediately instead of blocking. On success returns true else returns false. Hence its non-blocking

  • if its called by the thread that has the same mutex, then its a deadlock with undefined behaviour.
  • want to lock same mutex again, use recursive mutex.

timed-mutex

try oto acquire a lock for a limited amount of time instead of waiting forever.

try_lock() doesn't wait at all, try_lock_for(duration), try_lock_until(time_point)

std::recursive-mutex

  1. which threads owns
  2. how many times that thread has locked it
// std::mutex m;
std::recursive_mutex m;

void f(){
    m.lock(); // count = 1
    g(); // g also tries to lock m
    m.unlock(); // count = 0
}

void g(){
    m.lock(); // deadlock => // count = 2
    m.unlock(); // count = 1
}

Most experienced C++ engineers avoid it when possible because it can hide design problems.

  • same thread can lock multiple times
  • must unlock same number of times
  • stores owner + rec count

"Only when re-entrant locking is genuinely needed. Most of the time I'd rather redesign the locking structure so that a regular mutex suffices, because it's simpler and easier to reason about."

lock-guard

  • it is a small RAII wrapper around a mutex
  • template function
contructor -> lock the mutex
destructor -> unlock the mutex
std::mutex m;

void work() {
    std::lock_guard<std::mutex> lg(m);

    // critical section

} // lg destroyed here, m.unlock() automatically called
void f() {
    std::lock_guard<std::mutex> lg(m);

    throw std::runtime_error("oops");
}

no deadlock

lock_guard is the simplest RAII mutex wrapper. It acquires a mutex in its constructor and releases it in its destructor, guaranteeing exception-safe unlocking and preventing many common synchronization bugs.

promise and future in c++ threading

#include <future>
#include <thread>
#include <iostream>

int main() {
    std::promise<int> p;
    std::future<int> f = p.get_future();  // linked to p

    std::thread t([&p] {
        // do work...
        p.set_value(42);                  // producer sets the value
    });

    int result = f.get();                 // blocks until value is ready
    std::cout << result << "\n";          // 42
    t.join();
}

std::promise and std::future in C++

These two form a one-shot communication channel between threads: one end sets a value, the other waits for it.

The Core Mental Model

Think of it like a vending machine ticket system:

  • promise = the machine that will dispense the result
  • future = the ticket you hold, redeemed when the result is ready
  • The shared state = the slot connecting both
#include <future>
#include <thread>
#include <iostream>

int main() {
    std::promise<int> p;
    std::future<int> f = p.get_future();  // linked to p

    std::thread t([&p] {
        // do work...
        p.set_value(42);                  // producer sets the value
    });

    int result = f.get();                 // blocks until value is ready
    std::cout << result << "\n";          // 42
    t.join();
}

Key Operations

promise<T>

Method Effect
get_future() Returns the associated future (call only once)
set_value(v) Stores value, unblocks any waiting get()
set_exception(eptr) Stores an exception instead of a value
set_value_at_thread_exit(v) Defers notification until thread exits

future<T>

Method Effect
get() Blocks until ready, returns value (or rethrows exception). Callable only once — invalidates the future.
wait() Blocks until ready, doesn't consume
wait_for(duration) Timed wait, returns future_status
valid() Whether this future holds a shared state

Exception Propagation

The promise/future channel transparently tunnels exceptions across thread boundaries:

std::promise<int> p;
std::future<int> f = p.get_future();

std::thread t([&p] {
    try {
        throw std::runtime_error("oops");
    } catch (...) {
        p.set_exception(std::current_exception());  // marshal exception
    }
});

try {
    int v = f.get();   // rethrows on the calling thread
} catch (const std::exception& e) {
    std::cout << e.what() << "\n";  // "oops"
}
t.join();

std::async — The High-Level Wrapper

Most of the time you don't need to manage a promise manually. std::async creates the promise/future pair for you:

std::future<int> f = std::async(std::launch::async, [] {
    return expensive_computation();
});

// ... do other work ...

int result = f.get();  // blocks if not done yet

std::launch::async forces a new thread. std::launch::deferred runs lazily on get(). Omitting the policy lets the runtime decide.

shared_future — Multiple Consumers

A future is move-only and get() is destructive (one consumer only). If you need multiple threads to observe the same result, share it:

std::shared_future<int> sf = p.get_future().share();

// Both threads can call sf.get() safely
std::thread t1([sf]{ std::cout << sf.get() << "\n"; });
std::thread t2([sf]{ std::cout << sf.get() << "\n"; });

Broken Promise

If a promise is destroyed without set_value or set_exception, future::get() throws std::future_error with broken_promise. This is the C++ equivalent of a dangling callback.

When to Use What

Scenario Tool
Fire-and-get a single async result std::async
Producer/consumer with explicit thread control promise + future
Multiple threads reading one result shared_future
Ongoing stream of values promise/future is wrong — use a queue or channel

The key limitation to internalize: this is a one-shot mechanism. A promise can only be satisfied once. For repeated signaling, you want condition_variable; for a stream of values, a concurrent queue.

aync -> eager, deferred -> lazy

std::launch::async is eager. Task starts immediately on a new thread whether or not you ever get a get(). Happens in parallel with your current thread right now.

std::launch::deferred nothing runs at all until you call f.get() or f.wait(). At a point, the function runs synchronously on the calling thread -- no new thread is every created. It's basically a deferred function call dressed up as a future.

latency hiding - work that takes times, and instead of blocking your thread waiting for it, you want to overlap that works wtih others things you could be doing. This is what async allows us to do.
