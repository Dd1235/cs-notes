IPC — Pipes, Shared Memory, Signals

2 min read · source: os/ipc.md

Processes are isolated on purpose; IPC is the set of doors the kernel cuts into the walls. Know each door's shape and when you'd use it.

The menu

Mechanism What it is Best for Gotcha
Pipe ` ` one-way byte stream, kernel buffer parent↔child, shell pipelines
Named pipe (FIFO) pipe with a filesystem name unrelated processes, same host still one-way, blocking opens
Unix domain socket socket API, kernel-local two-way, multiple clients (Docker, Postgres, X11) more setup than a pipe
Shared memory (mmap/shm) same physical pages in both address spaces highest bandwidth, zero copies you bring the synchronization
Message queue kernel-managed discrete messages structured, prioritized messages size limits, less used now
Signal async one-bit poke (SIGKILL, SIGTERM…) notifications, kill, Ctrl-C almost nothing is safe inside a handler

The two philosophies

Everything above is either message passing (pipes/sockets/queues — kernel copies data, kernel provides the safety) or shared state (shm — no copies, no safety). The trade is exactly the threads-vs-processes trade in miniature: convenience+isolation vs speed+footguns. Shared memory + semaphores is the classic pairing — the fastest door and the least supervised.

Pipes mechanically

pipe(fd) gives fd[0] read / fd[1] write; fork(); each side closes the end it doesn't use. Shell a | b = pipe + two forks + dup2 stdout/stdin + exec. Bounded kernel buffer (~64 KB) means a full pipe blocks the writer — free backpressure, and the reason yes | head terminates: head exits, yes gets SIGPIPE.

Signals in three sentences

A signal is the kernel setting a flag; delivery runs your handler between instructions, whenever the process is next scheduled. Handlers may only call async-signal-safe functions — no printf, no malloc (deadlock risk if interrupted mid-malloc). The grown-up pattern: handler sets a volatile sig_atomic_t flag; the main loop notices — or signalfd/self-pipe to feed signals into your epoll loop.

Analogy: a signal is a sticky note slapped on your monitor mid-keystroke; you glance, finish the keystroke, deal with it. Writing an essay on the sticky note (complex handler logic) is how bugs happen.

Drill

  1. Implement a | b in C: which FDs does each process close and why does leaving the write end open hang the reader?
  2. Two processes share an mmap'd counter — what breaks without a semaphore, and where does the semaphore live? (In the shared region.)
  3. Why is SIGKILL unblockable and why do containers send SIGTERM first?

Reference: pipes/FIFOs in OSTEP ch 5; FD tables in CS:APP ch 12 notes.
