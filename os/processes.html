Processes — Lifecycle, fork & Friends

2 min read · source: os/processes.md

A process is a running program plus the OS's bookkeeping about it. The bookkeeping is the interview material.

Program vs process

A program is a dead file on disk. A process is that file brought to life: address space, registers, PC, open files, credentials. Ten processes can run one program — same recipe, ten separate cakes.

The OS tracks each in a PCB (process control block): PID, state, saved registers, page-table pointer, open-FD table, scheduling info.

The state machine

stateDiagram-v2
    [*] --> Ready: fork()
    Ready --> Running: scheduled
    Running --> Ready: timer preemption
    Running --> Blocked: I/O or wait()
    Blocked --> Ready: I/O completes
    Running --> Zombie: exit()
    Zombie --> [*]: parent wait()s

Blocked ≠ Ready is the distinction interviewers fish for: a blocked process costs nearly nothing — it isn't schedulable until its event fires.

The Unix trio

  • fork() — clone me. Returns twice: child's PID in the parent, 0 in the child. Cheap thanks to copy-on-write (see CS:APP ch 9).
  • exec() — replace my program, keep my PID and open FDs (that FD survival is how shells wire up redirection before exec'ing).
  • wait()/waitpid() — reap a finished child and collect its exit status.

Shell in one line: fork() → child does exec(cmd) → parent wait()s.

Zombies and orphans

  • Zombie: exited child whose parent hasn't wait()ed. Only the PCB entry survives — holding the exit status. A few are normal; thousands = parent bug (fd/PID leak).
  • Orphan: parent died first; child is adopted by init/systemd (PID 1), which dutifully reaps it.

Analogy: a zombie is a finished exam waiting to be collected; an orphan is a student whose teacher left — the principal takes over collection.

Context switch

Timer interrupt → kernel saves registers into PCB A → picks next via scheduler → restores PCB B → returns to userspace as B. Costs: microseconds of register work, plus the real cost — cold TLB and caches. Thread switches within a process skip the page-table swap; that's why they're cheaper.

Reference: OSTEP ch 4–5 (free) — the cleanest 30 pages on this topic anywhere.

Drill

  1. fork() is called once — why does it return twice? What's in each return?
  2. Why must exit status live in the kernel after death? (Parent may ask later; the address space is gone.)
  3. for(int i=0;i<3;i++) fork(); — how many processes? (8.)
