Virtual Memory — CS:APP Ch. 9

3 min read · source: os/self_notes/ch9.md

Didn't finish the chapter, but read the relevant parts for interviews — stopped at implicit free lists. The summary below covers what interviews actually ask from ch. 9.

The one-sentence version

Every process sees a private, contiguous virtual address space; the MMU + page tables translate virtual pages to physical frames on the fly, and the OS uses that indirection for isolation, caching, sharing and lazy allocation.

Address translation

graph LR
    CPU["CPU: virtual address"] --> TLB{"TLB hit?"}
    TLB -- yes --> PA["physical address"]
    TLB -- no --> PT["walk page table"]
    PT --> V{"page present?"}
    V -- yes --> PA
    V -- no --> PF["page fault → OS"]
    PF --> DISK["load page / zero page / COW copy"]
    DISK --> PT
    PA --> MEM["L1/L2/L3 → DRAM"]
  • Virtual address = VPN + offset; page table maps VPN → PPN, offset passes through. Pages are typically 4 kB.
  • The TLB is a small cache of translations inside the MMU; a miss costs a page-table walk (multi-level on x86-64: 4 levels).
  • Page fault is not an error — it's the OS's hook: page on disk (swap it in), page never touched (hand out a zeroed frame), COW write (copy the frame), or a genuine bad access (SIGSEGV).

What the indirection buys

  • Isolation — a process cannot even name another process's memory.
  • VM as a cache for disk — DRAM holds the working set; the rest lives on disk until faulted in.
  • Copy-on-write fork — parent and child share frames marked read-only; the first write faults and copies just that page. This is why fork of a huge process is cheap.
  • mmap — map a file (or anonymous memory) into the address space; page faults do the reads for you. Shared libraries are mapped once, shared by all processes.
  • Overcommit / lazy allocation — malloc of 1 GB touches nothing until you write to it.

Thrashing

If the working set exceeds physical RAM, the machine spends its time paging instead of computing — throughput falls off a cliff. Fix: less memory, more RAM, or better locality.

Dynamic memory allocation (where I stopped)

  • The heap allocator (malloc/free) manages the heap region; the goals are throughput and low fragmentation (internal = padding inside a block, external = free memory scattered in unusable pieces).
  • Implicit free list — every block has a header with size + allocated bit; walking the heap means jumping header to header. Placement: first fit / next fit / best fit; freeing wants coalescing (boundary tags allow O(1) merging with the previous block).
  • Beyond that: explicit free lists, segregated fits (what real allocators do) — to read on a second pass.

Drill

  1. Walk a load instruction: TLB miss, page fault, everything — the full slow path.
  2. Why is fork + exec cheap despite fork "copying" the address space?
  3. Internal vs external fragmentation — one example each.
  4. What actually happens on malloc(1 << 30)?
