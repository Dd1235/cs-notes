Deadlock

2 min read · source: os/deadlock.md

Four conditions, three strategies, one practical answer (lock ordering). This page is deliberately the shape of the interview answer.

The four conditions (all must hold)

  1. Mutual exclusion — resources aren't shareable.
  2. Hold and wait — threads hold some locks while waiting for others.
  3. No preemption — you can't forcibly take a lock away.
  4. Circular wait — a cycle: A waits for B's lock, B waits for A's.

Memory hook: "Mutual Hold, No Circle". Break any one and deadlock is impossible.

The canonical bug

// thread 1            // thread 2
lock(m1);              lock(m2);
lock(m2);   // waits   lock(m1);   // waits  → cycle

Three strategies

Prevention — design so a condition can't arise. The industrial answer: global lock ordering (kills circular wait — everyone acquires m1 before m2, always). Others: acquire all locks atomically (kills hold-and-wait; std::scoped_lock(m1, m2) does deadlock-avoiding all-or-nothing acquisition), or use trylock + back off (a preemption of sorts — beware livelock: two threads politely retrying forever).

Avoidance — admit requests only if a safe state remains. Banker's algorithm in three sentences: processes declare max needs upfront; before granting, the OS checks a completion order still exists for everyone; if not, the requester waits. Beautiful, rarely used — real systems don't know max needs upfront.

Detection & recovery — let it happen, find cycles in the waits-for graph, kill/rollback a victim. Databases do exactly this (deadlock detector + abort one transaction — ties to concurrency control); OSes mostly don't bother (the "ostrich algorithm").

Deadlock's cousins

  • Livelock — nobody blocks, nobody progresses (hallway dance).
  • Starvation — system moves, one thread never wins (readers flooding a writers lock — see CS:APP ch 12).

Drill

  1. Transfer A→B and B→A concurrently, locking both accounts — write the deadlock, then fix it by ordering on account ID.
  2. Which of the four conditions does scoped_lock break? Which does lock ordering break?
  3. Why do databases choose detection while kernels choose "ostrich"? (DBs can cheaply roll back a transaction; kernels can't roll back arbitrary threads.)

Reference: OSTEP ch 32 — Common Concurrency Problems.
