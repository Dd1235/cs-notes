Rapid Refresher

3 min read · source: website/content/os_refresher.md

A last-pass revision sheet distilled from the OS notes. Concurrency is the highest-yield interview area here — drill the mutex/semaphore and threading rows hardest.

Process vs thread

Process Thread
Address space own shared with siblings
Owns pages, FDs, signal handlers stack + registers + PC
Create/switch cost heavy light
Failure isolation strong one bad thread kills all
Talk via IPC (pipes, sockets, shm) shared memory (needs sync)

Three ways to serve many clients (CS:APP ch. 12)

  1. Multiprocess — fork per client: isolated, expensive.
  2. I/O multiplexing — one thread, select/poll/epoll: efficient, event-state machine complexity.
  3. Multithreading — thread per client or pool: shared state, needs locks.

C++ threading toolbox

  • Create: std::thread t(fn, args...) (function ptr, lambda, functor); always join() or detach().
  • std::mutex + std::lock_guard (RAII, scope-locked) / std::unique_lock (movable, works with condvars).
  • std::condition_variable — wait for a predicate: cv.wait(lk, []{ return ready; }) (loop handles spurious wakeups).
  • std::async / std::future — get a value out of a thread without manual plumbing.
  • std::atomic<T> — lock-free counters/flags; beware: atomicity ≠ ordering of surrounding code.

Mutex vs semaphore — the essay in four lines

A mutex is mutual exclusion with ownership: the locker must unlock.
A semaphore counts permits: any thread may signal.
Binary semaphore ≠ mutex — the ownership rule is the difference.
Use mutex to protect state; use semaphore to signal events or bound a resource pool.

Deadlock

  • Four conditions (all required): mutual exclusion, hold-and-wait, no preemption, circular wait.
  • Practical fix: global lock ordering (break circular wait), or std::scoped_lock(m1, m2) which locks atomically.
  • Livelock: everyone retries politely forever. Starvation: someone never wins.

Virtual memory (CS:APP ch. 9)

  • Each process sees a private, contiguous address space; the MMU maps virtual pages → physical frames via page tables.
  • TLB caches translations; miss → page-table walk; not-present → page fault → OS loads from disk/zero page.
  • Benefits: isolation, overcommit, mmap, copy-on-write fork, shared libraries mapped once.
  • Thrashing: working set exceeds RAM — the system pages instead of computing.

Scheduling ladder

FIFO (convoy effect) → SJF/STCF (optimal turnaround, needs clairvoyance, starves) → RR (great response, bad turnaround) → MLFQ (learns from behaviour: use full slice → demote; priority boost fixes starvation). Linux CFS/EEVDF: least virtual runtime runs next.

Deadlock in four words

Mutual exclusion, hold-and-wait, no preemption, circular wait — break one (practical: global lock ordering, or std::scoped_lock). Livelock = moving but stuck; starvation = system moves, you don't.

Storage quick hits

  • Inode = file metadata + block pointers; a directory maps names → inode numbers; hard link = same inode, symlink = path in a file.
  • Journaling = WAL for metadata: intent → commit → checkpoint; replay on crash.
  • Page replacement: LRU (temporal locality) approximated by clock/second-chance; FIFO suffers Belady's anomaly, LRU can't (stack property).
  • IPC: pipes (stream, backpressure), unix sockets (two-way), shared memory (fast, bring your own semaphore), signals (flag only — handlers do almost nothing).

Scheduling in one paragraph

Preemptive schedulers time-slice runnable threads across cores (Linux CFS: fairness by virtual runtime). Blocked ≠ runnable: a thread waiting on I/O or a lock costs almost nothing — which is why thread-per-connection dies from memory and context switches, not CPU.

Drill questions

  1. Producer-consumer with a bounded buffer: which primitives and where exactly do they go?
  2. Why must cv.wait take the predicate (or sit in a while loop)?
  3. Two threads increment a shared int 1M times each without sync — what's the range of outcomes and why?
  4. What actually happens on a page fault, step by step?
  5. Why is a context switch between threads of one process cheaper than between processes? (no page-table/TLB switch)
  6. When is detach() acceptable? What bug does forgetting join() cause?
