Concurrent Programming — CS:APP Ch. 12

3 min read · source: os/self_notes/ch12.md

Read it but did not take detailed notes; this is the main learning, expanded just enough to revise from.

Three ways to write a concurrent server

  1. Multiprocess — fork per client. Separate address spaces: no accidental sharing, no races — but sharing anything on purpose needs IPC, and processes are heavy.
  2. I/O multiplexing — one process, select / poll / epoll over many descriptors, an event loop with explicit state machines. Efficient and race-free, but the logic gets inverted and complex (see CN notes on epoll).
  3. Multithreading — threads share the address space: cheap sharing, cheap creation, and every shared variable is now a potential race.

The threads memory model

  • Each thread: own stack, registers, program counter, thread ID.
  • Shared by all: code, globals, heap, open file descriptors.
  • The subtle bit: stacks are not protected — any thread can reach another thread's stack through a pointer. "Local" is a convention, not an enforcement.

Semaphores — P and V

P(s): wait until s > 0, then decrement. V(s): increment, wake a waiter. Binary semaphore used as a lock = mutex.

Producer–consumer (bounded buffer) — the exact recipe:

insert:  P(slots)  P(mutex)  add item to buffer  V(mutex)  V(items)
remove:  P(items)  P(mutex)  remove an item      V(mutex)  V(slots)

Order matters: taking mutex before slots/items deadlocks — you'd sleep on a full/empty buffer while holding the lock the other side needs.

Readers–writers — maintain two semaphores (mutex for the counter, w for writers) and one read_count integer: the first reader in does P(w), and the last reader out does V(w); writers just wrap their section in P(w)/V(w). This is the readers-favored version — writers can starve.

Thread safety and reentrancy

  • A function is thread-safe if it behaves correctly under concurrent calls; reentrant if it uses no shared state at all (reentrant ⊂ thread-safe).
  • Classic unsafe example: rand() — it keeps hidden static state between calls. Fix: rand_r(&seed) where the caller owns the state.
  • Unsafe classes from the book: functions that don't protect shared variables, keep state across calls, return pointers to statics, or call any of the above.
  • Lock and copy: to safely use a function returning a pointer to a static, lock, copy the result out, unlock.

Deadlock

  • Deadlock from semaphores: two threads P() two semaphores in opposite orders — each holds one, waits for the other. Visible as a "deadlock region" in the book's progress graphs.
  • Mutex lock-ordering rule: every thread acquires locks in the same global order → circular wait is impossible.

File descriptor tables (review from ch. 10)

Per-process descriptor table → kernel-wide open file table (holds the offset; shared after fork, which is why parent and child advance the same file position) → v-node table (one per file). Two open()s of the same file = two offsets; fork or dup2 = shared offset.

Drill

  1. Write producer-consumer from memory — the four semaphore ops on each side, in order, and justify the order.
  2. Why can readers starve writers in the readers-writers solution above? How would you flip the preference?
  3. Is strtok thread-safe? Why not, and what class of unsafety is it?
  4. Draw the descriptor-table picture after fork() and explain why the offset is shared.
