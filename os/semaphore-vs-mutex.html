Semaphore vs Mutex

4 min read · source: os/semaphore_vs_mutex.md

A mutex is for mutual exclusion.

A semaphore is for counting available permits/resources.

They look similar because both can block threads, but their meaning is different.

1. Mutex

A mutex protects a critical section so that only one thread enters at a time.

#include <iostream>
#include <thread>
#include <mutex>
using namespace std;

mutex m;
int counter = 0;

void increment() {
    for (int i = 0; i < 100000; i++) {
        lock_guard<mutex> lock(m);
        counter++;
    }
}

int main() {
    thread t1(increment);
    thread t2(increment);

    t1.join();
    t2.join();

    cout << counter << '\n';
}

Here:

lock_guard<mutex> lock(m);

means:

lock mutex
do work
unlock automatically when scope ends

A mutex usually has ownership:

Thread A locks it
Thread A should unlock it

Thread B should not unlock a mutex locked by Thread A.

2. Semaphore

A semaphore has a counter.

A thread can:

acquire()  -> decrement permit count, or block if count is 0
release()  -> increment permit count, waking a blocked thread if needed

Example: allow at most 3 threads inside.

#include <iostream>
#include <thread>
#include <semaphore>
#include <vector>
using namespace std;

counting_semaphore<3> sem(3);

void worker(int id) {
    sem.acquire();

    cout << "Thread " << id << " entered\n";
    this_thread::sleep_for(chrono::seconds(1));
    cout << "Thread " << id << " leaving\n";

    sem.release();
}

int main() {
    vector<thread> threads;

    for (int i = 0; i < 10; i++) {
        threads.emplace_back(worker, i);
    }

    for (auto& t : threads) {
        t.join();
    }
}

At most 3 threads can be inside worker at once.

Core difference

Feature Mutex Semaphore
Main purpose Protect shared data Control access to N resources
Count Basically 0/1 locked state Counter: 0, 1, 2, 3, ...
Ownership Usually owned by locking thread No strict ownership
Unlock/release by another thread? Should not Allowed
Typical use Critical section Producer-consumer, bounded queues, resource pools
C++ type std::mutex std::counting_semaphore, std::binary_semaphore

Important conceptual difference

A mutex says:

Only one thread may touch this shared thing at a time.

A semaphore says:

There are K permits. A thread needs one permit to proceed.

So a mutex is about protecting data.

A semaphore is about signalling / limiting concurrency / counting resources.

Binary semaphore vs mutex

A binary semaphore has only 0 or 1 permit.

binary_semaphore sem(1);

This can look like a mutex:

sem.acquire();
// critical section
sem.release();

But it is still not exactly the same.

Difference

With a mutex:

same thread that locks should unlock

With a semaphore:

one thread can acquire, another thread can release

That makes semaphores useful for coordination.

Example:

#include <iostream>
#include <thread>
#include <semaphore>
using namespace std;

binary_semaphore ready(0);

void producer() {
    cout << "Producing data\n";
    ready.release(); // signal consumer
}

void consumer() {
    ready.acquire(); // wait until producer signals
    cout << "Consuming data\n";
}

int main() {
    thread c(consumer);
    thread p(producer);

    p.join();
    c.join();
}

Here the consumer blocks until the producer calls release().

This is not a mutex-style use case. This is signalling.

Producer-consumer example with semaphores

Suppose a bounded queue has capacity N.

You can use:

emptySlots semaphore initialized to N
filledSlots semaphore initialized to 0
mutex for protecting the actual queue

Pseudo-code:

enqueue(x):
    emptySlots.acquire();  // wait for space
    mutex.lock();
    queue.push(x);
    mutex.unlock();
    filledSlots.release(); // one more item available

dequeue():
    filledSlots.acquire(); // wait for item
    mutex.lock();
    x = queue.front();
    queue.pop();
    mutex.unlock();
    emptySlots.release();  // one more empty slot available

Notice the roles:

semaphores -> track how many slots/items exist
mutex      -> protects the queue data structure itself

This is the cleanest way to remember it.

When to use which?

Use a mutex when you have shared data:

map<string, int> freq;
mutex m;

and multiple threads modify it.

Use a semaphore when you want to represent permits:

only 5 database connections allowed
only 3 downloads at once
bounded queue has 10 empty slots
consumer waits until producer creates an item

Interview-level summary

A mutex is a lock used for mutual exclusion. It protects a critical section and is owned by the thread that locks it. A semaphore is a counter-based synchronization primitive. Threads acquire and release permits. A binary semaphore can behave like a lock, but unlike a mutex, it does not necessarily have ownership and can be used for signalling between threads.
