Filesystems — Inodes & Journaling

2 min read · source: os/filesystems.md

Two ideas carry every filesystem interview: "a file is an inode, a name is a pointer to one" and "crash consistency is WAL wearing a different hat".

The inode

An inode holds everything about a file except its name: size, permissions, timestamps, link count, and — the heart — pointers to the data blocks (direct pointers, then indirect/double-indirect blocks for big files; that multi-level scheme is why max file sizes exist).

A directory is just a file whose contents are name → inode number pairs. Path resolution /home/dee/notes.md = walk inodes: root's table → home's inode → its table → dee → its table → notes.md. (This is why deep paths cost reads and why the kernel caches lookups — the dcache.)

Hard links vs symlinks

  • Hard link: another name → same inode. Link count++; data dies only when count hits 0 and no process holds it open. Two equal names, no "original".
  • Symlink: a tiny file containing a path. Can dangle, can cross filesystems.

Interview classic: rm a 10 GB log a running process still has open — why does disk space not come back? (Link count 0, open count 1 — the inode lives until close. Fix: truncate, don't unlink.)

Crash consistency

Appending a block touches three things: data block, inode, block bitmap. Crash between writes = inconsistency (e.g. bitmap says allocated, inode never points to it — leaked block; worse orderings corrupt files).

Journaling = write-ahead logging, same idea, different layer:

graph LR
    W["write intent to journal<br/>(what I'm about to do)"] --> C["journal commit record"] --> P["checkpoint: apply to real<br/>inode/bitmap/data locations"] --> F["free journal entry"]
    C -.->|"crash after commit?<br/>replay journal"| P
    W -.->|"crash before commit?<br/>discard, old state intact"| X["no-op"]

Modes: data journaling (everything through the journal — safe, slow) vs metadata journaling (ext4 default: journal metadata only, write data first so metadata never points at garbage).

Name-drops that land

ext4 (journaling), XFS (large files), copy-on-write filesystems ZFS/Btrfs/APFS (never overwrite in place — snapshots fall out for free), and log-structured filesystems as the LSM-tree ancestor.

Reference: OSTEP ch 40 — File System Implementation & ch 42 — Journaling.

Drill

  1. ls -l shows link count 3 for a file — draw the picture.
  2. Why can't hard links cross filesystems? (Inode numbers are per-filesystem.)
  3. Exactly which crash windows does metadata journaling leave open, and why is that accepted?
