C++ Casts

6 min read · source: oops_cpp/self_notes/17.md

In C we lose information and there is lack of clarity of semantics. In C++ we can perform a fresh inference of type with or without change of value.

  • static_cast performs all conversions allowed implicitly. Not long those with pointers to classes, but als a void* to any pointer type. Convert integeres, floating-ponit values and enum types to enum types

  • no checks are performed during run-time to guarantee that the object being converted is in fact a full object of the destination type.

  • not only upcast but allso downcast.

  • can explicitly call a single argument constructor or a conversion operator, the user define cast. convert to rvalue references, convert enum class values to integers or floating point, convert any type to void.

  • instead of i = d; do i = static_cast(d) - okay, no warning from compiler

double* pd = &d;

i = pd; // implicit, error, not okay, wouldn't happen, why do this? assign double* to an int? serialization deserialization, cannot write down pointer values, same pointer address won't occur, need some indicative reference numbers
i = static_cast<int>(pd); // static_cast - error
i = (int)pd; // c cstyle, okay, risky, use reinterpret_cast

we get compiler error on static cast because it does not allow conversion on unrelated types. Like pointer to a double to an int. c-style cast works because it works in every case.

static_cast will work with downcasting but its better to not. dynamic_cast is runtime checked. Requires at least one virtual method (polymorphic types)

class Base  { public: virtual ~Base() {} };
class Dog   : public Base {};
class Cat   : public Base {};

Base* b = new Dog();

// pointer version: returns nullptr on failure
Dog* d = dynamic_cast<Dog*>(b);   // succeeds — d != nullptr
Cat* c = dynamic_cast<Cat*>(b);   // fails   — c == nullptr

if (d) { /* safe to use */ }

// reference version: throws std::bad_cast on failure
try {
    Dog& ref = dynamic_cast<Dog&>(*b);   // ok
    Cat& bad = dynamic_cast<Cat&>(*b);   // throws
} catch (const std::bad_cast& e) { ... }

class Window { public: virtual void onResize();...};
class SpecialWindow : public Window {
public:
    virtual void onResize(){
        static_cast<Window>(*this).onResize();
        // this doesn't work! you create a temporary object which is a copy and then call onResize on that!
        // do special window specific stuff
    }
}

instead Window::onResize();

static_cast with unrelated classes? If class A has a constructor that takes a const reference to class B, in that case static_cast(my_b_object) will work.

class A{
    ...
public:
    // cast B to A
    A(const& B){
        ...
    }
    operator int(){
        cout << "A::operator int()\n"; return A();
    }
    // cast A to B
    operator B()
}

this would make static_cast(a_object) or (B)a;

  • make sure you don't end up using the temporary object like the window case.

reinterpret_cast

  • converts any pointer type to any other pointer type (static can only upcase or downcast, don't downcast ever)
  • can also cast pointers to or from integer types.

class A {};
class B {};

int main(){
    int i = 2;
    double d = 3.7;
    double* pd = 7d;

    i = pd; // error
    i = reinterpret_cast<int>(pd); // okay
    i = (int)pd; // c style okay
    cout << pd << " " << i << endl;

    A *pA;
    B *pB;

    pA = pB; // error
    pA = reinterpret_cast<A*>(pB) // okay
    pA = (A*)pB; // C stle, okay
}

applications - serialization, deserialization, packetization, depacketization, etc.

Dynamic Cast

  • can only be used with pointers and references to classes or with void*

  • ensure result of the type conversion points to a valid complete object of hte destination pointer type

  • naturally includes pointer upcase (pointer_to_derived to pointer_to_base)

  • can also downcast polymorphic classes if-and-only-if pointed object is a valid complete object of the target type

  • down_cast would only work if its valid. for unrelated_cast it would be invalid other than when you have null or void*

  • downcast dispatch needs a polymorphic hierarchy.

  • can also be used iwth references and that is interesting.

  • can take a reference to B, and convert it to a reference to A, so upcast, and that's okay. but use static cast.

  • downcast on both pointer and reference

  • now A &rA3 = b; here its a b class object ( the derived class)

  • there is no "null reference", so throws an exception if you try to do illegal casting using references. bad-cast exception.

  • can't downcast reference illegally. This computation cannot be proceeded.

  • can also get bad-cast error for unrelated cast.

  • so why does dynamic type need polymorphic hierarchy? in c++ once you have created an object, how to know which class? its just a bit pattern. Looking at that object don't know if its a pair of int or complex type etc.

  • when there is a polymorphic hierarchy, each class has a vfunction table that is unique for that class.

  • that object will have a pointer to the vtable

  • that pointer is the identity, which class it came from.

  • that info is used by dynamic_cast.

typeid operator here

  • there is a typeid operator is used where the dynamic type of a polymorphic object must be known and for a static type identification.

  • polymorphic object - Objects of a class type that declares or inherits at least one virtual function are polymorphic objects.

  • lvalid expression

  • typeid oeprator can be applied ona type or an expression

  • operator== operator!= checks whether the objects refer to the same type

  • name implementation defined name of the type

  • won't work for non polymorphic type

  • works for polymorphic type only as it uses RTTI vtable. run time type information

  • if polymorphic object is bad bad_typeid exception

A a;
cout << typeid(a).name() << ": " << typeid(&a).name() << endl; // static
A *p = &a;
cout << typeid(p).name() << ": " << typeid(*p).name() << endl; // dynamic

B b; // derived from A
cout << typeid(b).name() << ": " << typeid(&b).name() << endl; // static
p = &b;
// same, typeid of p and deref p, we have dynamic
A &r1 = a; A& r2 = b;
// typeid and name, we get classA: classB

If its non-polymorphic hierarchy, it will not dynamically understand that derefencing base class pointer is actually pointing to a derived class. Derived class has no vtable so system as runtime does not print class Derived when you get typeid of the dereferenced object. Unless there is no vtable no way for compiler to tell the type of the object, if my base class pointer is pointing to derived class object.
