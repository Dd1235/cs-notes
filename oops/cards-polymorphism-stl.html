Polymorphism, Casts, Templates & STL Flashcards

10 min read · source: oops_cpp/flashcards/03_polymorphism_casts_templates_stl.md

A001

Tags: inheritance

Prompt: When is public inheritance appropriate?

Answer: When the derived type is substitutable for the base type: an is-a relationship where code expecting a base can correctly use a derived object.

A002

Tags: inheritance

Prompt: Public/protected/private inheritance changes base member access how?

Answer: Public inheritance keeps public as public and protected as protected. Protected inheritance makes public/protected base members protected. Private inheritance makes public/protected base members private.

A003

Tags: inheritance, design

Prompt: Inheritance vs composition: when prefer composition?

Answer: Prefer composition for has-a, uses-a, or implemented-using relationships, and when substitutability is not required.

A004

Tags: overloading, overriding, hiding

Prompt: Define overloading, overriding, and hiding.

Answer: Overloading: same name with different parameters in the same scope. Overriding: derived virtual function replaces base virtual behavior. Hiding: a derived declaration with the same name hides the base overload set.

A005

Tags: overriding

Prompt: Why use override?

Answer: It makes the compiler verify that a derived function actually overrides a virtual base function, catching signature mistakes and accidental hiding.

A006

Tags: name-hiding, edge-case

Prompt: In Base* p = new Child; cout << p->x;, if both classes have data member x, which x is used?

Answer: Base::x. Data members are resolved using static type, not virtual dispatch. Variables are hidden, not overridden.

A007

Tags: binding

Prompt: Static type vs dynamic type?

Answer: Static type is the declared compile-time type. Dynamic type is the actual most-derived type of the object a pointer/reference refers to at runtime.

A008

Tags: binding, virtual

Prompt: What conditions are needed for dynamic dispatch?

Answer: A virtual function in the base, a call through a pointer/reference to base, and a dynamic object whose most-derived type may differ from the static type.

A009

Tags: vtable

Prompt: Explain the common vtable/vptr model.

Answer: Each polymorphic class has a vtable of virtual function pointers. Each polymorphic object stores a hidden vptr pointing to its class vtable. A virtual call uses that table at runtime.

A010

Tags: vtable, costs

Prompt: Common costs of virtual dispatch?

Answer: One hidden vptr per polymorphic object in common implementations, indirect-call overhead, and reduced optimization opportunities.

A011

Tags: virtual-destructor

Prompt: When must a base destructor be virtual?

Answer: If objects may be deleted through a base pointer/reference-owning interface. Without a virtual destructor, delete base_ptr_to_derived is undefined behavior.

A012

Tags: virtual-destructor, correction

Prompt: Does a missing virtual destructor cause object slicing?

Answer: No. It causes undefined behavior when deleting a derived object through a base pointer. Slicing is when a derived object is copied/passed by value into a base object.

A013

Tags: abstract-class

Prompt: What makes a class abstract?

Answer: It has at least one pure virtual function that remains pure in that class, so objects of that class cannot be instantiated.

A014

Tags: pure-virtual, edge-case

Prompt: Can a pure virtual function have a body?

Answer: Yes. Define it outside the class and call it with qualified syntax such as Base::f() from derived code for common behavior.

A015

Tags: abstract-class

Prompt: What happens if a derived class does not override all inherited pure virtual functions?

Answer: The derived class remains abstract and cannot be instantiated.

A016

Tags: slicing

Prompt: What is object slicing?

Answer: Copying a derived object into a base object by value discards the derived subobject. Virtual dispatch on that sliced base object cannot reach derived behavior.

A017

Tags: slicing

Prompt: How do you avoid slicing in polymorphic code?

Answer: Pass/store polymorphic objects by reference, pointer, or smart pointer, not by base value.

A018

Tags: casts

Prompt: Match casts: static_cast, dynamic_cast, const_cast, reinterpret_cast.

Answer: static_cast: intended compile-time conversions. dynamic_cast: runtime-checked casts in polymorphic class hierarchies. const_cast: add/remove const/volatile. reinterpret_cast: low-level pointer/integer or unrelated pointer reinterpretation.

A019

Tags: casts, static-cast

Prompt: What can static_cast do with class types?

Answer: It can upcast, perform unchecked downcasts, invoke explicit single-argument constructors or conversion operators, cast to rvalue references, convert enum values, and convert to void.

A020

Tags: casts, static-cast, edge-case

Prompt: Why is static_cast<Window>(*this).onResize(); inside SpecialWindow::onResize() wrong when trying to call the base method?

Answer: It creates a sliced temporary Window object and calls the method on that temporary. Use Window::onResize(); to call the base implementation on the current object.

A021

Tags: casts, dynamic-cast

Prompt: What does dynamic_cast<Derived*>(basePtr) return on failure? What about dynamic_cast<Derived&>(baseRef)?

Answer: Pointer dynamic_cast returns nullptr on failure. Reference dynamic_cast throws std::bad_cast because references have no null value.

A022

Tags: casts, dynamic-cast

Prompt: Why does checked downcasting require a polymorphic hierarchy?

Answer: Runtime type information is available through polymorphic machinery such as vtables/vptrs. Without polymorphism, the runtime cannot reliably know the most-derived class from a base expression.

A023

Tags: casts, dynamic-cast

Prompt: Is dynamic_cast a design smell?

Answer: Often. It can mean the base interface is missing needed virtual behavior. It is acceptable for boundary checks, optional downcasts, or frameworks where runtime type discovery is explicit.

A024

Tags: casts, const-cast

Prompt: What is the only C++ cast that can remove constness or volatility?

Answer: const_cast.

A025

Tags: casts, const-cast, edge-case

Prompt: When is modifying through a const_cast undefined behavior?

Answer: When the original object is truly const. It is only safe to remove constness from a non-const object viewed through a const-qualified pointer/reference.

A026

Tags: casts, const-cast

Prompt: Can const_cast change an object's actual type?

Answer: No. It changes cv-qualification on pointer/reference expressions; it does not transform object representation or dynamic type.

A027

Tags: casts, reinterpret-cast

Prompt: Why is reinterpret_cast dangerous?

Answer: It can produce a pointer/reference/integer view that does not actually represent a valid object of the target type. Using the result can violate alignment, aliasing, and lifetime rules.

A028

Tags: casts, c-style-cast

Prompt: Why avoid C-style casts in C++?

Answer: They hide intent by trying several cast categories. C++ casts document whether you mean checked downcast, const removal, compile-time conversion, or low-level reinterpretation.

A029

Tags: rtti, typeid

Prompt: What does typeid(expr) report for polymorphic vs non-polymorphic class expressions?

Answer: For polymorphic glvalue expressions, it reports dynamic type. For non-polymorphic expressions, it reports static type.

A030

Tags: rtti, typeid

Prompt: What are std::type_info::name(), operator==, and operator!= used for?

Answer: name() gives an implementation-defined type name. == and != compare whether two type_info objects represent the same type.

A031

Tags: rtti, typeid, edge-case

Prompt: When can typeid(*p) throw std::bad_typeid?

Answer: When p is a null pointer to a polymorphic type and the expression dereferences it for typeid.

A032

Tags: multiple-inheritance

Prompt: What is the diamond problem?

Answer: A derived class receives duplicate base subobjects through two inheritance paths, creating ambiguity and duplicated state.

A033

Tags: multiple-inheritance

Prompt: What does virtual inheritance solve in a diamond hierarchy?

Answer: It shares one common virtual base subobject instead of duplicating it along each inheritance path.

A034

Tags: multiple-inheritance, design

Prompt: What kind of multiple inheritance is usually safer?

Answer: Multiple inheritance of small, stateless interfaces. Multiple stateful concrete bases require strong justification.

A035

Tags: exceptions

Prompt: What problem do exceptions solve compared with return-code error handling?

Answer: They separate normal control flow from error propagation, prevent ignored return codes from silently losing errors, and let stack unwinding clean up RAII objects.

A036

Tags: exceptions

Prompt: Synchronous vs asynchronous exceptions/events in the course notes?

Answer: Synchronous errors occur at a planned program point, such as throwing on invalid stack pop. Asynchronous events are unexpected external interrupts/signals.

A037

Tags: exceptions

Prompt: What is stack unwinding?

Answer: When an exception propagates, stack frames are exited and destructors for fully constructed local objects are called.

A038

Tags: exceptions

Prompt: Why catch exceptions by const&?

Answer: It avoids copying, preserves dynamic type without slicing, and allows catching standard exception hierarchies polymorphically.

A039

Tags: exceptions, edge-case

Prompt: Why should more specific catch clauses appear before base-class catches and catch(...)?

Answer: Catch matching is tried top-to-bottom. A base catch such as catch (B) can catch D first, making later catch (D) unreachable.

A040

Tags: exceptions

Prompt: What does bare throw; do inside a catch block?

Answer: It rethrows the currently handled exception without creating a new exception object.

A041

Tags: exceptions

Prompt: Exception safety levels?

Answer: No guarantee: object may be corrupted. Basic: invariants hold and no leaks. Strong: operation succeeds or state is unchanged. No-throw: operation will not throw.

A042

Tags: exceptions, raii

Prompt: Why does RAII reduce the need for C-style goto cleanup?

Answer: Resources are owned by objects whose destructors run automatically on every exit path, including exceptions, so shared manual cleanup labels are unnecessary.

A043

Tags: templates

Prompt: What is a function template?

Answer: A pattern describing how to generate functions for concrete template arguments, e.g. template<class T> T max(T a, T b).

A044

Tags: templates

Prompt: template<class T> vs template<typename T>?

Answer: For type template parameters, they are equivalent. typename is also needed to disambiguate dependent nested type names.

A045

Tags: templates, typename

Prompt: Why write typename T::value_type x; in a template?

Answer: Because T::value_type is a dependent name. Without typename, the compiler may parse it as a static member/value expression rather than a type.

A046

Tags: templates, specialization

Prompt: Why specialize Max<char*> instead of using generic x > y?

Answer: Generic pointer comparison compares addresses, not string contents. A specialization can use strcmp.

A047

Tags: templates, arrays, edge-case

Prompt: Why does template<class T> T Max(T arr[]) not know the array length?

Answer: The array parameter decays to T*. To preserve size, take a reference: template<class T, size_t N> T Max(const T (&arr)[N]).

A048

Tags: templates, class-template

Prompt: Why can Stack<char>* ps; be declared with only a forward declaration, but Stack<char> s; cannot?

Answer: A pointer declaration only needs to know that the type exists. An object definition needs the complete class layout and constructors/destructor.

A049

Tags: templates

Prompt: What is a non-type template parameter?

Answer: A compile-time value parameter, such as template<int N> or template<class T, size_t N>, not a type.

A050

Tags: templates, polymorphism

Prompt: What is parametric polymorphism?

Answer: Code is written generically in terms of parameters such as types or compile-time values, and the compiler instantiates concrete versions.

A051

Tags: stl

Prompt: What four concepts does the STL separate?

Answer: Containers store data, iterators navigate ranges, algorithms operate on iterator ranges, and function objects/lambdas customize behavior.

A052

Tags: stl, containers

Prompt: Choose the container: contiguous dynamic array, double-ended queue, linked list, ordered map, hash map, heap adaptor.

Answer: vector, deque, list, map, unordered_map, priority_queue.

A053

Tags: stl, vector-list

Prompt: Why is vector often faster than list even for many traversals?

Answer: vector has contiguous storage and excellent cache locality. list has pointer chasing and poor locality despite cheap relinking.

A054

Tags: stl, iterators

Prompt: Why does std::sort work on vector but not list?

Answer: std::sort requires random-access iterators. vector provides them; list only provides bidirectional iterators and has its own list::sort.

A055

Tags: stl, iterators

Prompt: List iterator categories from weakest to strongest.

Answer: Input/output, forward, bidirectional, random access, contiguous.

A056

Tags: stl, erase-remove

Prompt: Explain the erase-remove idiom.

Answer: remove_if moves kept elements forward and returns the new logical end; erase physically removes the trailing elements: v.erase(remove_if(v.begin(), v.end(), pred), v.end());.

A057

Tags: functors

Prompt: What is a functor?

Answer: An object that overloads operator() and can be called like a function, often while storing state.

A058

Tags: functors

Prompt: Why can a functor outperform a function pointer?

Answer: Its concrete type is known to templates, so calls are often inlineable and it can store state without type-erasure overhead.

A059

Tags: callables

Prompt: Compare function pointer, functor, lambda, and std::function.

Answer: Function pointer stores no state and has limited inline potential. Functor stores state and is inline-friendly. Lambda is compiler-generated functor with captures. std::function is type-erased and convenient but adds overhead.

A060

Tags: functors, deprecated

Prompt: Is inheriting std::binary_function required for STL functors?

Answer: No. It is deprecated/removed in modern C++. Just define operator() with the needed signature.

A061

Tags: policy

Prompt: What is policy parameterization?

Answer: Passing behavior as a parameter, such as a comparator to sort, so one algorithm supports multiple policies without hardcoded branching.

A062

Tags: virtual, base-call

Prompt: How do you call a base implementation from an override?

Answer: Use qualified syntax inside the derived method, e.g. Base::f();, not a cast that creates a base temporary.
