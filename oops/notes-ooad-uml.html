OOAD & UML

6 min read · source: oops_cpp/generated_notes/04_ooad_uml.md

Why OOAD Exists

Software complexity comes from:

  • Problem-domain complexity.
  • Development-process complexity.
  • Change pressure.
  • Discrete behavior with many possible states.
  • Human cognitive limits.

OOAD manages this by using:

  • Decomposition: break the system into parts.
  • Abstraction: focus on essential properties.
  • Hierarchy: organize details at multiple levels.
  • Models: reason about the system before code.

Complex Systems

Common traits of complex systems:

  • Hierarchical structure.
  • Relative primitives: what is primitive at one level may be complex below it.
  • Separation of concerns.
  • Repeated patterns.
  • Stable intermediate forms.

Design implication: build stable, testable parts with clear interfaces.

Object Model

Major elements:

Element Meaning
Abstraction capture essential behavior and properties
Encapsulation hide representation and protect invariants
Modularity divide into cohesive, loosely coupled units
Hierarchy organize through inheritance, aggregation, or layers

Minor elements:

Element Meaning
Typing enforce valid operations on objects
Concurrency support simultaneous activities
Persistence object state outlives a process

Abstraction vs Encapsulation

Abstraction is about what a thing is from the user's perspective.

Encapsulation is about hiding how it is implemented.

class Stack {
public:
    void push(int);
    void pop();
    int top() const;
    bool empty() const;

private:
    std::vector<int> data_;
};

The abstraction is "stack". The encapsulated detail is the vector representation.

Modularity

Good modules have:

  • High cohesion: related responsibilities stay together.
  • Low coupling: fewer dependencies on other modules.
  • Clear interfaces.
  • Hidden implementation details.

Bad modularity symptoms:

  • One class knows too much about another class's internals.
  • Changing one concept forces many unrelated changes.
  • Utility classes collect random behavior.

Object State, Behavior, Identity

Concept Meaning
State values stored in the object
Behavior operations the object can perform
Identity the object remains itself across state changes

Two objects may be equal by value but not identical.

std::string a = "x";
std::string b = "x";
// same value, different objects

Relationships

Relationship Meaning C++ hint
Association objects know/use each other pointer/reference/member
Dependency temporary use function parameter/local variable
Aggregation weak whole-part, part can outlive whole non-owning pointer/reference
Composition strong ownership, part dies with whole value member/unique pointer
Inheritance is-a relationship public base class

Aggregation vs composition:

Question Aggregation Composition
Who owns the part? separate owner whole object
Can part outlive whole? yes usually no
C++ representation reference/raw observer pointer value member or unique_ptr

Design by Contract

A contract states expectations:

  • Preconditions: what must be true before a function is called.
  • Postconditions: what is true after it returns.
  • Invariants: what always stays true for a valid object.
class Fraction {
    int n_;
    int d_; // invariant: d_ != 0
};

Interfaces expose contracts. Implementations fulfill them.

Finding Classes

Useful but imperfect method:

  • Nouns suggest candidate classes or attributes.
  • Verbs suggest responsibilities.
  • Adjectives/adverbs suggest constraints or states.
  • Repeated rules suggest policy objects or subclasses.

Then filter:

  • Is this a real concept or just a field?
  • Does it own state?
  • Does it have behavior?
  • Does it protect an invariant?
  • Is it stable under likely changes?

UML Overview

UML is a modeling language. Use only the diagram that answers the current design question.

Diagram Main use
Use case actors and goals
Class static structure and relationships
Sequence time-ordered object interactions
Communication object links and messages
Activity workflow/control flow
State machine lifecycle states and transitions
Interaction overview high-level interaction flow

Structural diagrams describe what exists. Behavioral diagrams describe what happens.

Use Case Diagrams

Use case diagrams capture external actors and system goals.

Elements:

  • Actor: external user/system.
  • Use case: goal the system provides.
  • System boundary: what is inside the system.
  • Association: actor participates in a use case.
  • Include: mandatory reused behavior.
  • Extend: optional/conditional behavior.
  • Generalization: specialized actor/use case.

Interview trap: use cases are not functions. They represent user-visible goals.

Class Diagrams

Class diagram contents:

  • Class name.
  • Attributes.
  • Operations.
  • Visibility: + public, - private, # protected.
  • Relationships: association, inheritance, aggregation, composition, dependency.
  • Multiplicity: 1, 0..1, *, 1..*.

Keep class diagrams at the right abstraction level. Do not dump every private helper into a design discussion unless it matters.

Sequence Diagrams

Sequence diagrams show messages over time.

Elements:

  • Lifeline: object/participant.
  • Activation: period of execution.
  • Message: call, return, create, destroy, async signal.
  • Time flows top to bottom.

Use sequence diagrams to validate responsibility placement:

  • Who initiates the operation?
  • Which object has the required data?
  • Are there too many messages crossing module boundaries?

Communication Diagrams

Communication diagrams emphasize object links and message numbering rather than vertical time.

Use when structure of collaboration matters more than precise timing.

Activity Diagrams

Activity diagrams model workflows.

Elements:

  • Initial node.
  • Action.
  • Decision and merge.
  • Fork and join.
  • Final node.
  • Swimlanes for responsibility.

Good for business processes, validation flows, and conditional workflows.

State Machine Diagrams

State machines model lifecycle behavior for one object.

Elements:

  • State.
  • Transition.
  • Event.
  • Guard.
  • Action.
  • Initial/final pseudostates.

Use state machines when valid operations depend heavily on current state, such as order processing or ticket lifecycle.

Protocol State Machines

A protocol state machine specifies valid operation sequences on an interface.

Use it to answer:

  • Which method calls are valid now?
  • What call changes the object's state?
  • Which calls are illegal in this state?

Compact LMS Modeling Case

The leave-management slide case is useful as a compact OOAD drill.

Core domain concepts:

  • Employee with roles such as individual contributor, lead, and manager.
  • Leave request.
  • Leave type with different rules.
  • Approval decision.
  • Balance or entitlement.
  • Supporting document for special leave.
  • Unauthorized absence as a derived case.

Candidate classes:

  • Employee
  • Approver
  • LeaveRequest
  • LeavePolicy
  • LeaveBalance
  • ApprovalWorkflow
  • Document
  • Calendar

Responsibilities:

  • Employee: holds identity, reporting relationship, balances.
  • LeaveRequest: dates, type, status, applicant.
  • LeavePolicy: validates rules for a leave type.
  • ApprovalWorkflow: routes request to the correct approver.
  • Calendar: counts working days and holidays.

Relationships:

  • Employee reports to another employee: association.
  • Leave request belongs to an employee: association/composition depending on persistence model.
  • Leave policy is used by validation: dependency or strategy composition.
  • Specific policies such as casual, earned, medical, parental, and unpaid can be modeled through Strategy or subclasses.

Placement-oriented design improvement:

  • Avoid a giant if leave_type == ... block.
  • Put variable rules behind a LeavePolicy interface.
  • Keep approval routing separate from leave-balance validation.

Checkpoints

  • What is the difference between abstraction and encapsulation?
  • What is high cohesion? What is low coupling?
  • Aggregation vs composition?
  • Association vs dependency?
  • When is inheritance a poor choice?
  • How do use case and sequence diagrams differ?
  • What does multiplicity mean in a class diagram?
  • When would you use a state machine diagram?
  • How would you model different leave rules without a large switch?
