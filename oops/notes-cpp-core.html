C++ Core

8 min read · source: oops_cpp/generated_notes/01_cpp_core.md

Mental Model

C++ is a compiled, statically typed language that keeps C's low-level control while adding stronger abstraction tools: references, overloading, classes, RAII, templates, and the standard library.

Interview focus:

  • Know the build pipeline.
  • Know object lifetime and memory regions.
  • Know how declarations, definitions, linkage, and headers interact.
  • Prefer standard library containers and algorithms over manual arrays and loops.

Build Pipeline

  1. Preprocessing: expands #include, macros, and conditional compilation.
  2. Compilation: parses each translation unit and generates object code.
  3. Assembly: emits machine-level instructions.
  4. Linking: combines object files and libraries, resolving symbols.

Key terms:

  • Source file: usually .cpp; contains definitions.
  • Header file: usually .h, .hpp, .hxx; contains declarations, templates, inline definitions.
  • Translation unit: one source file after preprocessing, including all headers it pulled in.
  • Object file: compiled unit with machine code, symbol table, relocation data, and unresolved external references.
  • Linker: connects unresolved references across translation units and libraries.

Headers, Linkage, and ODR

Declarations tell the compiler that a name exists. Definitions allocate storage or provide function bodies.

// declaration
int add(int, int);

// definition
int add(int a, int b) {
    return a + b;
}

The One Definition Rule says an entity should have exactly one definition across the program, except for allowed cases such as inline functions and templates with identical definitions in multiple translation units.

Header rules:

  • Put declarations in headers.
  • Put non-template function definitions in source files.
  • Put template definitions in headers because instantiation needs the full body.
  • Use include guards or #pragma once.
  • Avoid global variables in headers unless they are inline variables in modern C++ or extern declarations with one source definition.
#ifndef FRACTION_H
#define FRACTION_H

struct Fraction {
    int num;
    int den;
};

#endif

Linkage:

Construct Meaning
extern int x; Declaration for a variable defined elsewhere
static at file scope Internal linkage, visible only in that translation unit
anonymous namespace Modern way to give internal linkage to names
extern "C" Disable C++ name mangling for C interop

Name Mangling

C++ supports overloading, so symbol names encode type information.

void f(int);
void f(double);

These two functions cannot have the same linker symbol. Name mangling makes them distinct. extern "C" is used when linking with C libraries that do not use C++ mangling.

Memory Layout

Typical process memory:

Region Stores
text/code compiled instructions
read-only data string literals, some constants
data initialized globals and statics
BSS zero-initialized globals and statics
heap dynamic allocation via new, malloc, containers
stack function calls, local automatic variables

Important:

  • A local variable normally lives on the stack.
  • A static local variable has block scope but program lifetime.
  • Dynamically allocated objects live until released or owned by an object that releases them.
void calls() {
    int x = 0;          // automatic lifetime
    static int seen = 0; // program lifetime
    ++seen;
}

C Recap That Still Matters

Core C tools remain relevant in C++:

  • Primitive types: char, integer types, floating-point types, bool.
  • Operators: arithmetic, relational, logical, bitwise, operator=.
  • Control flow: if, switch, loops, break, continue, return.
  • Arrays decay to pointers in many expressions.
  • Structs group fields.
  • Pointers store addresses and may be null.
  • Functions use pass-by-value unless a pointer/reference is passed.

Common traps:

  • Array bounds are not checked.
  • Signed integer overflow is undefined behavior.
  • Using an uninitialized local variable is undefined behavior.
  • Dereferencing a null or dangling pointer is undefined behavior.
  • sizeof(array) only works as array size in the same scope before decay.

Arrays, Vectors, and Strings

Prefer std::vector over raw dynamic arrays.

#include <vector>
#include <string>

std::vector<int> a = {3, 1, 4};
a.push_back(2);

std::string s = "placement";
s += " prep";
Need Prefer
Fixed compile-time size std::array<T, N>
Dynamic contiguous sequence std::vector<T>
Text std::string
Raw buffer ownership std::vector<std::byte> or smart pointer

Raw arrays are useful for interop and low-level code, but they are easy to misuse.

Standard Library Algorithms

Sorting and searching should usually use the algorithm library.

#include <algorithm>
#include <vector>

std::vector<int> v = {4, 1, 3, 2};
std::sort(v.begin(), v.end());

bool has_three = std::binary_search(v.begin(), v.end(), 3);
auto it = std::find(v.begin(), v.end(), 4);

Interview angle:

  • std::sort is generally introsort, average O(n log n).
  • std::find is linear.
  • Binary search requires a sorted range.
  • Iterators are generalized pointers into containers/ranges.

const, constexpr, and volatile

const means the value cannot be modified through that name.

int x = 5;
const int y = x; // OK, y is read-only after initialization

constexpr means the value or function can be evaluated at compile time when inputs allow it.

constexpr int square(int x) {
    return x * x;
}

constexpr int n = square(5);

volatile tells the compiler the value may change outside normal program flow. It is for memory-mapped I/O and similar low-level cases, not for thread synchronization. Use atomics or locks for concurrency.

Pointer Constness

Read declarations right-to-left around *.

Declaration Meaning
const int* p pointer to const int; cannot modify *p through p
int const* p same as above
int* const p const pointer to mutable int
const int* const p const pointer to const int 
int a = 10;
int b = 20;

const int* p = &a;
p = &b;       // OK
// *p = 30;   // error

int* const q = &a;
*q = 30;      // OK
// q = &b;    // error

Macros vs Inline Functions

Macros are textual substitution. They do not respect types, scopes, or evaluation safety.

#define SQUARE(x) ((x) * (x))

Problems:

  • SQUARE(i++) increments twice.
  • Debugging is harder.
  • No type checking.

Prefer:

inline int square(int x) {
    return x * x;
}

Modern meaning of inline: multiple identical definitions across translation units are allowed. It does not force machine-code inlining.

References

A reference is an alias for an existing object.

int x = 10;
int& r = x;
r = 20; // x becomes 20

Pointer vs reference:

Feature Pointer Reference
Null possible yes no, in normal valid code
Rebinding yes no
Syntax explicit dereference alias syntax
Good for optional target yes no
Good for required parameter yes, but reference is cleaner

Pass-by-reference avoids copies and allows modification.

void increment(int& x) {
    ++x;
}

void print(const std::string& s) {
    // no copy, cannot modify s
}

Return-by-reference is only safe when the referred object outlives the function call.

int& first(std::vector<int>& v) {
    return v[0];
}

// Bad: returns reference to a dead local.
// int& bad() { int x = 0; return x; }

Function Defaults and Overloading

Default arguments are filled by the caller at compile time.

void log(std::string msg, int level = 1);

Rules:

  • Defaults must appear from right to left.
  • Usually put defaults in the declaration, not repeated across definitions.
  • Defaults do not create a different function signature.

Function overloading chooses among functions with the same name and different parameter lists.

void print(int);
void print(double);
void print(const std::string&);

Overload resolution prefers:

  1. Exact match.
  2. Promotion.
  3. Standard conversion.
  4. User-defined conversion.
  5. Variadic fallback.

Avoid overload sets that require subtle conversions. They make code hard to predict.

Dynamic Memory

new allocates memory and constructs an object. delete destroys the object and frees memory.

int* p = new int(5);
delete p;

malloc allocates raw memory only. It does not call constructors. free does not call destructors.

void* raw = std::malloc(sizeof(int));
std::free(raw);

C++ code should usually avoid owning raw new/delete directly. Use RAII:

  • std::vector<T> for dynamic arrays.
  • std::string for text.
  • std::unique_ptr<T> for exclusive heap ownership.
  • std::shared_ptr<T> only for shared ownership with clear semantics.

Undefined Behavior

Undefined behavior means the language gives no guarantee. The compiler may optimize assuming it never happens.

Common cases:

  • Out-of-bounds access.
  • Use-after-free.
  • Double delete.
  • Dereferencing null.
  • Returning reference to a local.
  • Signed integer overflow.
  • Data race.
  • Reading uninitialized storage.
