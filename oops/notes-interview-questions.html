Interview Questions

9 min read · source: oops_cpp/generated_notes/07_interview_questions.md

Use these for active recall. Try to answer in 60-120 seconds, then write a small snippet where relevant.

C++ Core

  1. What is a translation unit?

    • A source file after preprocessing, including all headers pulled in. Each translation unit is compiled separately.

  2. Why are templates usually defined in headers?

    • The compiler needs the full template definition when instantiating it for a specific type.

  3. What is the One Definition Rule?

    • An entity must have exactly one definition across the program, with special allowances for identical inline/template definitions.

  4. What does static mean at file scope?

    • Internal linkage. The name is visible only inside that translation unit.

  5. What does static mean inside a function?

    • The variable has program lifetime but block scope.

  6. Explain const int* p, int* const p, and const int* const p.

    • Pointer to const int, const pointer to int, const pointer to const int.

  7. Is const int x = runtime_value; allowed?

    • Yes. const means read-only after initialization, not necessarily compile-time.

  8. Why is constexpr stronger than const?

    • constexpr requires compile-time evaluability when used in constant contexts.

  9. What does std::move do?

    • It casts to an rvalue reference. The actual move happens only if a move constructor or move operator= is selected.

  10. Why avoid raw owning pointers?

    • Ownership and cleanup become error-prone. RAII types express ownership and clean up automatically.

OOP and Lifetime

  1. What is RAII?

    • Tie resource lifetime to object lifetime. Constructor acquires, destructor releases.

  2. Rule of 3, Rule of 5, Rule of 0?

    • If managing a resource manually, define copy/destructor operations; with moves, define five. Better: use RAII members and define none.

  3. Copy constructor vs copy operator=?

    • Copy constructor creates a new object. Copy operator= changes an existing object.

  4. Why prefer initializer lists?

    • They construct members directly and are required for const/reference members.

  5. Why should a base class destructor be virtual?

    • If deleting derived objects through base pointers, the derived destructor must run.

  6. What is object slicing?

    • Passing/copying a derived object by value into a base object loses the derived part.

  7. What is the difference between overloading and overriding?

    • Overloading selects among same-name functions with different parameters. Overriding replaces virtual base behavior in a derived class.

  8. When is inheritance wrong?

    • When the relationship is not substitutable or when composition would express ownership/use more clearly.

  9. Why use override?

    • The compiler verifies that a derived function actually overrides a virtual base function.

  10. What is a pure virtual function?

    • A virtual function declared with = 0; it makes the class abstract unless implemented in a derived class.

Polymorphism and Casting

  1. How does virtual dispatch usually work?

    • A polymorphic object has a hidden vptr pointing to a vtable. A virtual call uses the vtable to select the runtime function.

  2. What is static binding?

    • Compile-time binding, used for non-virtual calls and overload resolution.

  3. What is dynamic binding?

    • Runtime dispatch through virtual functions.

  4. When use dynamic_cast?

    • For checked downcasting across polymorphic types when redesign cannot avoid it.

  5. Why is reinterpret_cast dangerous?

    • It treats bits as another type with almost no safety checks.

  6. What is the diamond problem?

    • A derived class receives duplicate base subobjects through two inheritance paths.

  7. What does virtual inheritance solve?

    • It shares one common base subobject in a diamond hierarchy.

Exceptions

  1. What happens during stack unwinding?

    • Destructors of fully constructed local objects run while an exception propagates.

  2. Why catch exceptions by const reference?

    • Avoids slicing and copying while preserving dynamic type.

  3. What is the strong exception guarantee?

    • Operation either succeeds or leaves state unchanged.

  4. Should destructors throw?

    • Generally no. Throwing during unwinding can terminate the program.

  5. What does noexcept communicate?

    • The function is not expected to throw; it also enables optimizations and safer moves.

STL and Templates

  1. Why does std::sort not work on std::list?

    • std::sort needs random-access iterators; list has bidirectional iterators.

  2. Vector vs list?

    • Vector has contiguous storage and fast random access. List has stable node insertion/erasure but poor locality and no random access.

  3. Ordered map vs unordered map?

    • map is ordered, usually tree-backed, O(log n). unordered_map is hash-backed, average O(1), no order.

  4. What is iterator invalidation?

    • Operations on a container may make existing iterators/references invalid.

  5. What is a functor?

    • An object with operator(), usable like a function and capable of storing state.

  6. Functor vs function pointer?

    • Functor can hold state and is often easier to inline.

  7. What is a predicate?

    • Callable returning bool, used to customize algorithms such as find_if.

  8. Explain erase-remove idiom.

    • remove_if partitions unwanted elements to the end and returns new logical end; erase physically removes them.

Code Reading Traps

Trap 1: Returning a Reference to Local

int& bad() {
    int x = 10;
    return x;
}

Answer: dangling reference. The local dies when the function returns.

Trap 2: Missing Virtual Destructor

struct Base {};
struct Derived : Base {
    std::vector<int> v;
};

Base* p = new Derived;
delete p;

Answer: undefined behavior because base destructor is not virtual.

Trap 3: Macro Double Evaluation

#define SQR(x) ((x) * (x))

int i = 3;
int y = SQR(i++);

Answer: increments twice and has unsafe side effects. Use an inline function.

Trap 4: Object Slicing

struct B { virtual void f(); };
struct D : B { void f() override; };

void call(B b) {
    b.f();
}

Answer: passing by value slices D into B; virtual dispatch will not reach D.

Trap 5: std::move Misunderstanding

const std::string s = "abc";
std::string t = std::move(s);

Answer: likely copies because moving from const cannot bind to a typical non-const move constructor.

DE Shaw-Style Prompts

These are meant to test reasoning depth, not memorized definitions.

  1. A function takes const Base& and calls a virtual method. Which type decides the method body: static type or dynamic type?

    • Dynamic type, because the call is virtual and made through a base reference.

  2. A class owns a raw pointer and defines a destructor but not copy operations. What breaks?

    • The compiler-generated copy may copy the pointer value, causing double delete or shared accidental ownership.

  3. Why can vector beat list even when inserting in the middle seems expensive?

    • Cache locality and lower allocation overhead often dominate theoretical relinking cost.

  4. Why can catch (std::exception e) be wrong?

    • It copies and slices derived exception objects. Use catch (const std::exception& e).

  5. When would a switch over object type be a design smell?

    • When behavior varies by type and new types are expected. Prefer polymorphism or a strategy/policy object.

  6. Why is 100 percent statement coverage not enough?

    • It may miss branch outcomes, boundary cases, invalid states, and weak assertions.

  7. If two classes depend on each other's private fields, what design issue is likely?

    • High coupling and poor encapsulation. Move behavior to the class that owns the data or introduce a clearer collaboration.

  8. Why should a public API avoid exposing std::vector internals if representation may change?

    • It leaks implementation details and makes future changes source-breaking.

  9. What makes a state machine better than scattered boolean flags?

    • It names valid states and transitions explicitly, reducing impossible state combinations.

  10. What is the first thing to clarify before coding from a vague requirement?

    • Observable behavior and acceptance criteria, including edge cases and non-functional constraints.

OOAD and UML

  1. Abstraction vs encapsulation?

    • Abstraction exposes essential behavior. Encapsulation hides implementation details.

  2. Aggregation vs composition?

    • Aggregation is weak whole-part; part can outlive whole. Composition is owning whole-part; part usually dies with whole.

  3. Association vs dependency?

    • Association is a structural relationship. Dependency is temporary use.

  4. What is high cohesion?

    • A module's responsibilities are closely related.

  5. What is low coupling?

    • Modules depend on each other through minimal stable interfaces.

  6. Use case diagram vs sequence diagram?

    • Use case shows actors and goals. Sequence shows time-ordered object interactions.

  7. When use a state machine?

    • When an object's valid operations depend on lifecycle state.

  8. How would you model many leave types with different rules?

    • Use a LeavePolicy interface and concrete policy classes or strategy objects.

Design Patterns

  1. Iterator intent?

    • Traverse elements without exposing container internals.

  2. Command intent?

    • Encapsulate a request/action as an object.

  3. Singleton tradeoff?

    • Ensures one instance but creates hidden global state and makes tests harder.

  4. Factory Method vs Abstract Factory?

    • Factory Method creates one product through an overridable method. Abstract Factory creates families of related products.

  5. Visitor tradeoff?

    • Easy to add operations, hard to add new element types.

  6. Which pattern helps avoid a large switch over leave type?

    • Strategy is the most direct, though policy objects may be implemented with factories. From the slide pattern set, Factory can centralize creation of policy objects.

  7. Which pattern supports undo?

    • Command, with enough saved state to reverse the action.

Testing and SDLC

  1. Verification vs validation?

    • Verification: building the product right. Validation: building the right product.

  2. Black-box vs white-box testing?

    • Black-box tests external behavior. White-box uses implementation knowledge.

  3. Boundary value analysis?

    • Test values at, just below, and just above boundaries where bugs often occur.

  4. Statement vs branch coverage?

    • Statement coverage checks executed lines/statements. Branch coverage checks decision outcomes.

  5. What is regression testing?

    • Testing that previously working behavior still works after changes.

  6. Severity vs priority?

    • Severity is impact. Priority is scheduling urgency.

  7. Why are requirements important for tests?

    • Tests need expected behavior; requirements provide the oracle.

  8. What changes in AI systems?

    • Data quality, model drift, evaluation, reproducibility, monitoring, and human review become central.

Rapid Fire

  • Prefer const T& for large read-only parameters.
  • Prefer pass by value for cheap scalar types.
  • Prefer unique_ptr for exclusive ownership.
  • Prefer shared_ptr only for real shared ownership.
  • Prefer composition unless substitutability is clear.
  • Prefer RAII over manual cleanup.
  • Prefer algorithms over hand-written loops when intent becomes clearer.
  • Prefer small interfaces.
  • Prefer tests that assert behavior, not implementation details.
  • Prefer diagrams that answer a design question; do not draw UML for decoration.
