Design Patterns

5 min read · source: oops_cpp/generated_notes/05_design_patterns.md

Pattern Mindset

A design pattern is a reusable solution shape for a recurring design problem.

Use patterns to communicate intent and manage change. Do not force a pattern where a simple function or class is enough.

For interviews, always explain:

  • Problem it solves.
  • Structure.
  • Tradeoff.
  • When not to use it.

Quick Map

Pattern Main idea Common use
Iterator traverse without exposing representation containers, trees, custom collections
Command wrap a request as an object undo, queues, logging, GUI actions
Singleton one shared instance process-wide config, registry, logger
Factory Method subclass decides product type framework hooks, pluggable creation
Abstract Factory create families of related objects UI themes, platform families
Visitor add operations over stable object structure ASTs, document trees

Iterator

Intent: provide sequential access to elements without exposing internal representation.

std::vector<int> v = {1, 2, 3};

for (auto it = v.begin(); it != v.end(); ++it) {
    std::cout << *it << '\n';
}

Benefits:

  • Decouples traversal from container internals.
  • Enables generic algorithms.
  • Multiple traversal strategies can coexist.

Tradeoffs:

  • Iterator invalidation rules matter.
  • Mutating a container while iterating can be dangerous.

Interview angle:

  • vector iterators can be invalidated by reallocation.
  • list iterators usually remain valid across unrelated insert/erase.
  • Algorithms work on iterator ranges, not containers directly.

Command

Intent: represent an action as an object.

class Command {
public:
    virtual void execute() = 0;
    virtual ~Command() = default;
};

class PasteCommand : public Command {
public:
    void execute() override {
        // paste
    }
};

Uses:

  • Undo/redo.
  • Task queues.
  • Macro recording.
  • Logging requests.
  • Decoupling UI buttons from operations.

Tradeoffs:

  • More classes/objects.
  • State needed for undo can be non-trivial.

Modern C++ note:

  • For simple cases, std::function<void()> can replace a full hierarchy.
  • Use a class when the command needs identity, metadata, undo, or serialization.

Singleton

Intent: ensure exactly one instance and provide global access.

class Logger {
public:
    static Logger& instance() {
        static Logger logger;
        return logger;
    }

    void write(std::string_view msg);

    Logger(const Logger&) = delete;
    Logger& operator=(const Logger&) = delete;

private:
    Logger() = default;
};

Benefits:

  • Centralizes a truly unique resource.
  • Local static initialization is thread-safe in modern C++.

Tradeoffs:

  • Hidden global state.
  • Harder testing.
  • Lifetime-order issues across globals.
  • Can become a dumping ground.

Interview answer: Singleton is not always bad, but dependency injection is often more testable.

Factory Method

Intent: define an interface for object creation while letting subclasses or specific factories decide the concrete product.

class Document {
public:
    virtual void render() = 0;
    virtual ~Document() = default;
};

class PdfDocument : public Document {
public:
    void render() override {}
};

class App {
public:
    virtual std::unique_ptr<Document> create_document() = 0;
    virtual ~App() = default;
};

class PdfApp : public App {
public:
    std::unique_ptr<Document> create_document() override {
        return std::make_unique<PdfDocument>();
    }
};

Uses:

  • Frameworks where base code calls a creation hook.
  • Product type depends on subclass or runtime choice.
  • Creation logic should not be spread across client code.

Tradeoffs:

  • More indirection.
  • Can be overkill when a simple constructor call is enough.

Abstract Factory

Intent: create families of related objects without naming their concrete classes.

class Button {
public:
    virtual void draw() = 0;
    virtual ~Button() = default;
};

class Checkbox {
public:
    virtual void draw() = 0;
    virtual ~Checkbox() = default;
};

class WidgetFactory {
public:
    virtual std::unique_ptr<Button> make_button() = 0;
    virtual std::unique_ptr<Checkbox> make_checkbox() = 0;
    virtual ~WidgetFactory() = default;
};

Uses:

  • Platform-specific UI components.
  • Theme families.
  • Related products that must be compatible.

Tradeoffs:

  • Easy to add new product families.
  • Harder to add a new product kind because every factory interface changes.

Factory Method vs Abstract Factory:

Point Factory Method Abstract Factory
Creates one product type family of related products
Variation subclass/factory method factory object
Good when one creation decision varies compatibility across products matters

Visitor

Intent: add operations to a stable object structure without modifying each class repeatedly.

class Number;
class Add;

class ExprVisitor {
public:
    virtual void visit(Number&) = 0;
    virtual void visit(Add&) = 0;
    virtual ~ExprVisitor() = default;
};

class Expr {
public:
    virtual void accept(ExprVisitor&) = 0;
    virtual ~Expr() = default;
};

Uses:

  • AST operations such as type checking, printing, optimization.
  • Document trees.
  • Object structures where new operations are added more often than new element types.

Tradeoffs:

  • Easy to add new operations.
  • Hard to add new element classes because every visitor must change.
  • Can expose too much internal structure.

Visitor vs virtual methods:

  • Put behavior as virtual methods when behavior is core to the object.
  • Use Visitor when many external operations must run over a stable hierarchy.

Design Pattern Selection

Ask:

  • What varies?
  • What must stay stable?
  • Is this object creation, traversal, behavior encapsulation, or operation extension?
  • Would a standard library tool solve it more simply?
  • Does the pattern improve testability or harm it?

Checkpoints

  • Why does Iterator enable generic algorithms?
  • What state is needed for undo in Command?
  • Why can Singleton hurt tests?
  • Factory Method vs Abstract Factory?
  • When is Visitor a good fit?
  • Which pattern can replace a large switch over leave types?
  • Which pattern can model a queue of deferred actions?
  • Which pattern helps traverse a custom tree?
