Design Patterns Flashcards

4 min read · source: oops_cpp/flashcards/05_design_patterns.md

P001

Tags: patterns

Prompt: What is a design pattern?

Answer: A reusable solution shape for a recurring design problem. It communicates intent and manages change, but should not be forced where simple code suffices.

P002

Tags: patterns, interview

Prompt: In interviews, what four points should you explain for any pattern?

Answer: The problem it solves, its structure, its tradeoff, and when not to use it.

P003

Tags: patterns, map

Prompt: Match pattern to core idea: Iterator, Command, Singleton, Factory Method, Abstract Factory, Visitor.

Answer: Iterator traverses without exposing representation. Command wraps a request as an object. Singleton provides one shared instance. Factory Method lets subclasses/factories decide one product type. Abstract Factory creates compatible families. Visitor adds operations over a stable object structure.

P004

Tags: iterator

Prompt: Intent of Iterator?

Answer: Provide sequential access to elements without exposing the collection's internal representation.

P005

Tags: iterator, stl

Prompt: Why does Iterator enable generic algorithms?

Answer: Algorithms operate on iterator ranges, so they do not need to know the concrete container representation.

P006

Tags: iterator, edge-case

Prompt: What iterator invalidation issue matters for vector?

Answer: Reallocation can invalidate all iterators, references, and pointers to elements. Insert/erase can also invalidate from the modification point onward.

P007

Tags: iterator, edge-case

Prompt: Why are list iterators often more stable than vector iterators?

Answer: Linked nodes are not moved during unrelated insert/erase, so iterators to other nodes often remain valid.

P008

Tags: command

Prompt: Intent of Command?

Answer: Represent an action/request as an object so it can be queued, logged, undone, retried, or decoupled from the invoker.

P009

Tags: command

Prompt: Common uses of Command?

Answer: Undo/redo, task queues, macro recording, logging requests, GUI button actions, and decoupling UI from operations.

P010

Tags: command, undo

Prompt: What state is needed for undo in Command?

Answer: Enough previous state or inverse-operation data to restore the receiver to its earlier state.

P011

Tags: command, modern-cpp

Prompt: When can std::function<void()> replace a Command hierarchy?

Answer: For simple fire-and-forget actions without identity, metadata, serialization, or rich undo state.

P012

Tags: singleton

Prompt: Intent of Singleton?

Answer: Ensure exactly one instance and provide controlled global access to it.

P013

Tags: singleton, cpp

Prompt: Why is function-local static Singleton preferred in modern C++?

Answer: Initialization is lazy and thread-safe since C++11, and it avoids manual heap allocation/deletion.

P014

Tags: singleton, tradeoff

Prompt: Singleton tradeoffs?

Answer: It centralizes a truly unique resource, but creates hidden global state, testing difficulty, lifetime-order issues, and risk of becoming a dumping ground.

P015

Tags: singleton, design

Prompt: Interview answer: is Singleton always bad?

Answer: No. It can be appropriate for truly unique process-wide resources, but dependency injection is often more testable and explicit.

P016

Tags: factory-method

Prompt: Intent of Factory Method?

Answer: Define an interface/hook for object creation while letting subclasses or concrete factories decide the product class.

P017

Tags: factory-method

Prompt: When is Factory Method useful?

Answer: Framework hooks, pluggable creation, runtime/subclass-dependent product choice, and avoiding scattered construction logic.

P018

Tags: factory-method, tradeoff

Prompt: Factory Method tradeoff?

Answer: It centralizes variation in creation but adds indirection and can be overkill when a direct constructor is clear.

P019

Tags: abstract-factory

Prompt: Intent of Abstract Factory?

Answer: Create families of related/compatible objects without naming their concrete classes.

P020

Tags: abstract-factory

Prompt: Where is Abstract Factory a natural fit?

Answer: Platform-specific UI widgets, theme families, device families, or any related products that must be compatible.

P021

Tags: abstract-factory, tradeoff

Prompt: Abstract Factory is easy to extend along which axis, and hard along which axis?

Answer: Easy to add new product families. Hard to add a new product kind because every factory interface and concrete factory must change.

P022

Tags: factory-method, abstract-factory

Prompt: Factory Method vs Abstract Factory?

Answer: Factory Method varies creation of one product type through a method/subclass. Abstract Factory provides an object that creates a family of related products.

P023

Tags: visitor

Prompt: Intent of Visitor?

Answer: Add operations to a stable object structure without repeatedly modifying each element class.

P024

Tags: visitor

Prompt: Where is Visitor commonly used?

Answer: ASTs, document trees, interpreters/compilers, expression trees, and stable hierarchies needing many external operations.

P025

Tags: visitor, tradeoff

Prompt: Visitor tradeoff?

Answer: Easy to add new operations; hard to add new element types because every visitor interface/implementation must change.

P026

Tags: visitor, design

Prompt: Visitor vs virtual methods?

Answer: Use virtual methods when behavior is core to the object. Use Visitor when many external operations must run over a stable hierarchy.

P027

Tags: pattern-selection

Prompt: Which questions help select a pattern?

Answer: What varies? What stays stable? Is this creation, traversal, behavior encapsulation, or operation extension? Would a standard library tool be simpler? Does it improve testability?

P028

Tags: pattern-selection

Prompt: Which pattern can model a queue of deferred actions?

Answer: Command.

P029

Tags: pattern-selection

Prompt: Which pattern helps traverse a custom tree?

Answer: Iterator for traversal; Visitor if the goal is to perform many operations over a stable tree structure.

P030

Tags: pattern-selection, lms

Prompt: Which pattern helps avoid a large switch over leave type?

Answer: Strategy/policy objects are the direct design; from the covered pattern set, Factory Method or Abstract Factory can centralize creation of the appropriate policy objects.
