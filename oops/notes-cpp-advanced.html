Inheritance

Inheritance models an is-a relationship when used publicly.

class Shape {
public:
    virtual double area() const = 0;
    virtual ~Shape() = default;
};

class Circle : public Shape {
    double r_;

public:
    explicit Circle(double r) : r_(r) {}
    double area() const override { return 3.14159 * r_ * r_; }
};

Access under inheritance:

Inheritance Public base members become Protected base members become
public public protected
protected protected protected
private private private

Use public inheritance for substitutability. Use composition when the relationship is "has-a" or "implemented using".

Overloading, Overriding, Hiding

Term Meaning
Overloading same name, different parameters in same scope
Overriding derived virtual function replaces base virtual behavior
Hiding derived declaration hides base overload set by name 
struct Base {
    virtual void f(int);
    void g(double);
};

struct Derived : Base {
    void f(int) override;
    void g(int); // hides Base::g(double)
};

Use override. It lets the compiler catch signature mistakes.

Static and Dynamic Binding

Static binding happens at compile time. Dynamic binding happens at runtime through virtual dispatch.

void print_area(const Shape& s) {
    std::cout << s.area(); // virtual call
}

Virtual dispatch requires:

  • A virtual function in the base class.
  • A call through pointer/reference to base.
  • A dynamic object whose most-derived type may differ.

VTable and VPtr

Typical implementation:

  • Each polymorphic class has a vtable containing function pointers.
  • Each polymorphic object stores a hidden vptr pointing to its class vtable.
  • A virtual call indexes the vtable at runtime.

This is an implementation model, not a standard-mandated layout, but it is the common interview explanation.

Costs:

  • One hidden pointer per polymorphic object in common implementations.
  • Indirect call overhead.
  • Some optimization opportunities become harder.

Virtual Destructors

If a class is meant to be deleted through a base pointer, give it a virtual destructor.

struct Base {
    virtual ~Base() = default;
};

struct Derived : Base {
    std::vector<int> data;
};

Base* p = new Derived;
delete p; // OK only if Base destructor is virtual

Without a virtual base destructor, deleting through a base pointer is undefined behavior.

Abstract Base Classes

A pure virtual function makes a class abstract.

class Reader {
public:
    virtual std::string read() = 0;
    virtual ~Reader() = default;
};

Abstract base classes are useful for interfaces. Keep interfaces small.

Object Slicing

Passing a derived object by value to a base parameter slices off the derived part.

void bad(Shape s);        // impossible here because Shape is abstract
void good(const Shape& s);

Use references or smart pointers for polymorphic objects.

Casts

Cast Use
static_cast safe compile-time conversions you intend
dynamic_cast checked downcast across polymorphic types
const_cast add/remove constness; dangerous if object is truly const
reinterpret_cast low-level bit reinterpretation; highly risky 
Base* b = new Derived;

if (auto* d = dynamic_cast<Derived*>(b)) {
    // checked success
}

Prefer designs that avoid downcasting. Downcasting often means the interface is weak.

RTTI and typeid

typeid(expr) gives runtime type information for polymorphic expressions.

if (typeid(*b) == typeid(Derived)) {
    // exact dynamic type check
}

Prefer virtual functions over explicit type checks.

Multiple Inheritance and Diamond Problem

Multiple inheritance can model combining interfaces, but concrete multiple inheritance is harder.

Diamond:

struct A { int x; };
struct B : public A {};
struct C : public A {};
struct D : public B, public C {}; // two A subobjects

Virtual inheritance shares one base subobject:

struct B : virtual public A {};
struct C : virtual public A {};
struct D : public B, public C {};

Guideline: multiple interface inheritance is usually fine; multiple stateful base classes require strong justification.

Exceptions

Exceptions separate error handling from normal control flow.

double divide(double a, double b) {
    if (b == 0.0) throw std::invalid_argument("division by zero");
    return a / b;
}

Catch by const reference:

try {
    work();
} catch (const std::exception& e) {
    std::cerr << e.what();
}

Stack unwinding:

  • When an exception propagates, destructors of fully constructed local objects run.
  • RAII makes cleanup safe during unwinding.
  • Destructors should not emit exceptions.

Exception safety levels:

Level Meaning
No guarantee object may be corrupted
Basic guarantee invariants hold; no leaks
Strong guarantee operation succeeds or state is unchanged
No-throw guarantee operation will not throw

Use noexcept for functions that must not throw, especially move operations when true.

Templates

Templates generate code for types used with them.

Function template:

template <typename T>
T max_value(const T& a, const T& b) {
    return b < a ? a : b;
}

Class template:

template <typename T>
class Box {
    T value_;

public:
    explicit Box(T value) : value_(std::move(value)) {}
    const T& get() const { return value_; }
};

Important:

  • typename and class are equivalent for type template parameters.
  • Templates are usually defined in headers.
  • Compilation errors can be long because instantiation happens at use sites.
  • Concepts in modern C++ can constrain template parameters.

STL Model

The STL separates:

  • Containers: store data.
  • Iterators: navigate ranges.
  • Algorithms: operate on iterator ranges.
  • Function objects/lambdas: customize behavior.
std::vector<int> v = {4, 1, 3, 2};
std::sort(v.begin(), v.end(), std::greater<int>{});

This design gives generic algorithms independent of specific container types.

Containers

Container Use Important complexity
vector dynamic contiguous array random access O(1), push back amortized O(1)
deque double-ended sequence push front/back O(1)
list linked list insert/erase with iterator O(1), no random access
map ordered key-value tree search/insert/erase O(log n)
unordered_map hash table average search/insert/erase O(1)
set ordered unique keys search/insert/erase O(log n)
stack LIFO adaptor top/push/pop
queue FIFO adaptor front/push/pop
priority_queue heap adaptor top O(1), push/pop O(log n)

Vector vs list:

Point vector list
Memory locality excellent poor
Random access yes no
Middle insert with known iterator shifts elements relinks nodes
Typical performance often better useful when stable iterators and frequent splices matter

Iterators

Iterator categories:

  • Input: read forward once.
  • Output: write forward once.
  • Forward: multi-pass forward.
  • Bidirectional: forward and backward.
  • Random access: arithmetic jumps.
  • Contiguous: memory-contiguous random access.

Algorithm requirements depend on iterator category. std::sort needs random-access iterators, so it works for vector but not list.

Algorithms and Predicates

std::vector<int> v = {1, 2, 3, 4, 5};

auto even = [](int x) { return x % 2 == 0; };
auto it = std::find_if(v.begin(), v.end(), even);
int sum = std::accumulate(v.begin(), v.end(), 0);

Useful algorithms:

  • find, find_if
  • sort, stable_sort
  • binary_search, lower_bound, upper_bound
  • copy, copy_if
  • remove, remove_if with erase-remove idiom
  • accumulate
  • transform
  • for_each

Erase-remove idiom:

v.erase(std::remove_if(v.begin(), v.end(), even), v.end());

Functors and Lambdas

A functor is an object callable with operator().

struct IsGreaterThan {
    int limit;

    bool operator()(int x) const {
        return x > limit;
    }
};

std::vector<int> v = {1, 4, 7};
auto it = std::find_if(v.begin(), v.end(), IsGreaterThan{5});

Why functors:

  • Can store state.
  • Can be inlined by the compiler.
  • Useful as policy objects.

Lambda equivalent:

int limit = 5;
auto it = std::find_if(v.begin(), v.end(),
                       [limit](int x) { return x > limit; });

Function pointer vs functor vs lambda:

Callable Stores state Type-safe Usually inline-friendly
function pointer no yes limited
functor yes yes yes
lambda yes, via capture yes yes
std::function yes yes type-erased overhead

Policy Parameterization

Algorithms can accept behavior as a parameter.

std::sort(names.begin(), names.end(),
          [](const std::string& a, const std::string& b) {
              return a.size() < b.size();
          });

This avoids hardcoded branching and lets one algorithm support many policies.
