Class, Object, State, Behavior, Identity

A class defines a type. An object is a runtime instance of that type.

  • State: data members.
  • Behavior: member functions.
  • Identity: two objects can have equal state but still be different objects.
class Counter {
    int value_ = 0;

public:
    void increment() { ++value_; }
    int value() const { return value_; }
};

Encapsulation and Information Hiding

Encapsulation groups data and operations. Information hiding prevents users from depending on internal representation.

Use private data members and expose a minimal interface:

class BankAccount {
    long long balance_ = 0;

public:
    explicit BankAccount(long long opening_balance)
        : balance_(opening_balance) {}

    void deposit(long long amount) {
        if (amount <= 0) return;
        balance_ += amount;
    }

    long long balance() const {
        return balance_;
    }
};

Avoid mindless get-set interfaces. If every field has a setter, invariants are not protected.

Access Specifiers

Specifier Accessible from
public everyone
protected class and derived classes
private class and friends

Default access:

  • class: private.
  • struct: public.

Use struct for passive data aggregates and class for invariant-owning types.

this Pointer

Every non-static member function receives a hidden this pointer.

class Box {
    int width_;

public:
    explicit Box(int width) : width_(width) {}

    bool wider_than(const Box& other) const {
        return this->width_ > other.width_;
    }
};

this is unavailable in static member functions because they are not called on an object.

Constructors

A constructor initializes an object.

class Fraction {
    int num_;
    int den_;

public:
    Fraction(int num, int den) : num_(num), den_(den) {
        if (den_ == 0) throw std::invalid_argument("zero denominator");
    }
};

Prefer member initializer lists:

  • Required for references and const data members.
  • Avoids default construction followed by later operator=.
  • Initializes members in declaration order, not list order.

Use explicit for single-argument constructors to prevent accidental conversions.

class Meter {
    int value_;

public:
    explicit Meter(int value) : value_(value) {}
};

Destructors and RAII

A destructor runs when an object lifetime ends.

RAII: resource acquisition is initialization. Tie a resource to an object so cleanup happens automatically.

class File {
    FILE* fp_;

public:
    explicit File(const char* path) : fp_(std::fopen(path, "r")) {
        if (!fp_) throw std::runtime_error("open failed");
    }

    ~File() {
        if (fp_) std::fclose(fp_);
    }

    File(const File&) = delete;
    File& operator=(const File&) = delete;
};

Use standard RAII types when possible:

  • std::fstream
  • std::lock_guard
  • std::unique_ptr
  • std::vector
  • std::string

Object Lifetime

Objects are constructed before use and destroyed at lifetime end.

Destruction order:

  • Local automatic objects: reverse order of construction.
  • Class members: reverse declaration order after destructor body runs.
  • Base classes: after derived members, reverse base construction order.
struct Trace {
    std::string name;
    explicit Trace(std::string n) : name(std::move(n)) {}
    ~Trace() { /* observe destruction order while debugging */ }
};

Interview trap: a destructor should usually not throw. Throwing during stack unwinding can terminate the program.

Copy Control

Compiler-generated operations:

  • Default constructor.
  • Destructor.
  • Copy constructor.
  • Copy operator=.
  • Move constructor.
  • Move operator=.

Rule of 3: if you manually manage a resource and define one of destructor, copy constructor, or copy operator=, you probably need all three.

Rule of 5: also consider move constructor and move operator=.

Rule of 0: best design is to use RAII members so compiler-generated special members are correct.

class Buffer {
    std::vector<int> data_;

public:
    explicit Buffer(std::size_t n) : data_(n) {}
};

Copy Constructor vs Copy operator=

class X {
public:
    X(const X& other);            // copy construction
    X& operator=(const X& other); // copy operator=
};

Difference:

  • Copy constructor creates a new object from an existing one.
  • Copy operator= replaces the state of an already-existing object.
  • operator= should return *this.
  • operator= must handle x = x or be naturally safe.

Copy-and-swap idiom:

class Widget {
    std::vector<int> data_;

public:
    void swap(Widget& other) noexcept {
        data_.swap(other.data_);
    }

    Widget& operator=(Widget other) {
        swap(other);
        return *this;
    }
};

Move Semantics

Move transfers resources from a temporary or expiring object.

class Holder {
    std::unique_ptr<int> p_;

public:
    Holder() : p_(std::make_unique<int>(0)) {}
    Holder(Holder&&) noexcept = default;
    Holder& operator=(Holder&&) noexcept = default;
};

std::move does not move by itself. It casts an expression to an rvalue reference so a move operation may be selected.

std::string a = "abc";
std::string b = std::move(a); // a is valid but unspecified

Const Objects and Const Member Functions

A const object can call only const member functions.

class Vector2 {
    double x_;
    double y_;

public:
    double length() const {
        return std::sqrt(x_ * x_ + y_ * y_);
    }
};

const after the function name means this is a pointer to const.

double length() const;

Use const member functions for observers. This allows use with const references and makes intent clear.

Const Data Members and mutable

Const data members must be initialized in the constructor initializer list.

class Id {
    const int value_;

public:
    explicit Id(int value) : value_(value) {}
};

mutable allows a data member to change even in a const member function. Use it for caches or synchronization primitives, not logical state.

class Expensive {
    mutable bool cached_ = false;
    mutable int value_ = 0;

public:
    int get() const {
        if (!cached_) {
            value_ = 42;
            cached_ = true;
        }
        return value_;
    }
};

Static Data and Static Member Functions

Static data members belong to the class, not an individual object.

class User {
    inline static int count_ = 0;

public:
    User() { ++count_; }
    ~User() { --count_; }

    static int count() {
        return count_;
    }
};

Static member functions:

  • Have no this.
  • Can directly access only static members.
  • Are useful for factories, counters, and utility behavior tied to the class.

Singleton

A Singleton restricts a type to one shared instance.

class Config {
public:
    static Config& instance() {
        static Config c;
        return c;
    }

    Config(const Config&) = delete;
    Config& operator=(const Config&) = delete;

private:
    Config() = default;
};

Use sparingly. It creates hidden global state and makes testing harder. Dependency injection is often cleaner.

Friend Functions and Friend Classes

friend grants access to private/protected members.

class Fraction {
    int num_;
    int den_;

public:
    Fraction(int n, int d) : num_(n), den_(d) {}

    friend std::ostream& operator<<(std::ostream& os, const Fraction& f);
};

std::ostream& operator<<(std::ostream& os, const Fraction& f) {
    return os << f.num_ << '/' << f.den_;
}

Use friend for tightly coupled operations such as symmetric operators or stream insertion. Do not use it to bypass design.

Operator Overloading

Operators are functions with operator syntax.

class Fraction {
    int n_;
    int d_;

public:
    Fraction(int n, int d) : n_(n), d_(d) {}

    Fraction operator+(const Fraction& rhs) const {
        return Fraction(n_ * rhs.d_ + rhs.n_ * d_, d_ * rhs.d_);
    }
};

Rules:

  • You cannot create new operators.
  • You cannot change precedence or associativity.
  • At least one operand must be user-defined for most overloads.
  • Some operators should be members: =, [], (), ->.
  • Stream operators are usually non-member functions.

Good overloads match natural meaning:

Operator Good use
+ value-like addition
== equality of observable value
< strict ordering
<< stream output
[] indexing
() function object

Bad overloads surprise the reader. Do not make + mutate, or * mean "save to database".

Namespaces

Namespaces group names and avoid collisions.

namespace math {
class Fraction {};
}

math::Fraction f;

Guidelines:

  • Do not put using namespace std; in headers.
  • Prefer qualified names in shared interfaces.
  • Namespace aliases are useful for long names.
namespace fs = std::filesystem;

OOP Design Checklist

For each class, ask:

  • What invariant does it protect?
  • Who owns each resource?
  • Are constructors forcing valid state?
  • Does copying make sense? If not, delete it.
  • Does moving make sense? If yes, mark move operations noexcept when possible.
  • Which functions are observers and should be const?
  • Is inheritance needed, or would composition be simpler?

Checkpoints

  • Difference between class and object?
  • What is the hidden this pointer?
  • Why prefer initializer lists?
  • What is RAII?
  • What are Rule of 3, Rule of 5, and Rule of 0?
  • When is a destructor virtual?
  • What does std::move actually do?
  • Why should stream operators often be friends or non-members?
  • Why is uncontrolled Singleton usage risky?
