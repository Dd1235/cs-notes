try/catch & Function Templates

3 min read · source: oops_cpp/self_notes/19.md
  • Exceptions thrown will also result in stack frame being cleaned like when return will result in stack frame cleanup
#include <exception>

class MyException : public exception {};
class MyClass { ~MyClass() {} };

void h(){

    MyClass h_a;
    // throw 1;
    // throw 2.5;
    // throw MyException();
    // throw exception();
    // throw MyClass();
}
// stack unwind, destructor h_a.~MyClass() called
// passes on all exceptions

void g(){
    MyClass g_a;
    try {
        h();
        bool okay = true; // not executed
    }
    catch (int){
        ...
    }catch(double){
        ...
    }catch (...){ // default
        throw; // rethrow, the guy that catches this may catch MyException, exception etc. no copy, same object passed on
    }
} // stack unwind
  • try block: consolidate areas that might throw exceptions
  • tries an exact match, no implicit conversion allowed, can match with the public base class.
class B {};
class D : public B {};

try {
    throw D{};
}
catch (B b) {
    cout << "Caught B\n";
}
catch (D d) {
    cout << "Caught D\n";
}

Caught B, catch(D) is never reached.

  • more specialized ones have to come earlier. that's why catch all has to come at the end

  • destructor-savvy

  • unobstrusive

  • precise

  • native and Standard

  • scalable

  • fault tolerant

exception is a virtual base class from standard library available to us.

common exception classes already exist.

Templates

  • generic programming / meta programming
  • write algorithms wo knowing the types of some of the components

say a max function for int, double, char *, Complex.

  • we could do it in overloaded manner. for int and double, it is the same algorithm but different operator
int Max..{return x > y ? x : y; }
double Max..{return x > y ? x : y; }
..

could use a macro

#define Max(x, y) (((x) > (y)) ? x : y)
but macros have side effects like with a++ and b++

Function templates

  • describes how a function should be built.

template<class T> // templatized function
T Max(T x, T y){
    return x > y ? x : y
}
template<> // template specialization for char * type
char *Max<char *>(char* x, char* y){
    return strcmp(x, y) > 0 ? x : y;
}

iMax = Max<int>(a, b);
dMax = Max<double>(c, d);
sMax = Max<char *>(s1, s2); // won't generate function, will use templaet specialized algorithm

compiler internally generated two overloaded functions. "Macro side effects" problems solved.

can skip adding the type, and it will infer the types. Max(a,b)

template<class T> T Max(T x, T y);
template<> char* Max<char *>(char *x, char* y);
template <class T, int size> Type Max(T x[size]);

int main(){
    ...
    Max<int, 7>(arr)
}

compiler sees T* x
cannot so sizeof(x) inside because x is really a pointer.

To deduce size

template <typename T, size_t N>
T Max(const T (&x)[N])
{
    T m = x[0];

    for(size_t i = 1; i < N; i++)
        if(x[i] > m)
            m = x[i];

    return m;
}

int main(){
    int arr[5] = {1,2,3,4,5};
    cout << Max(arr);
}

typename keyword:

template <class T> f(T x){
    T::name * p;
}

is T::name a type and p is a pointer to that type? or both are variables and this is a multiplication?

it would be interpretted as multiplication unless typename T::name * p;

  • keywords class and typename have almost same meaning in a template parameter. Tell compiler that an expression is a type expression.
