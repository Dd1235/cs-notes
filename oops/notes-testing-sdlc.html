Testing, SDLC & SE

6 min read · source: oops_cpp/generated_notes/06_testing_sdlc_se.md

Why Software Engineering Matters

Software projects fail because of:

  • Unclear requirements.
  • Poor communication.
  • Underestimated complexity.
  • Weak design boundaries.
  • Inadequate testing.
  • Changing requirements without control.
  • Schedule pressure that creates technical debt.

Engineering response:

  • Understand requirements.
  • Model before building.
  • Design for change.
  • Test continuously.
  • Manage scope, time, cost, and quality.

SDLC Stages

Typical stages:

  1. Requirement gathering and analysis.
  2. Design.
  3. Implementation.
  4. Testing.
  5. Deployment.
  6. Maintenance.

The stages may be sequential or iterative depending on the process model.

Requirements

Requirement qualities:

  • Clear.
  • Testable.
  • Consistent.
  • Complete enough for current iteration.
  • Traceable to design and tests.
  • Prioritized.

Types:

Type Meaning
Functional what the system does
Non-functional quality constraints such as latency, security, reliability
Domain constraint rule from the business or environment
Interface requirement interaction with external systems

Bad requirement: "system should be fast".

Better: "search should return results within 200 ms for 95 percent of requests under target load".

SDLC Models

Model Idea Works best when Risk
Waterfall sequential phases requirements are stable late feedback
Iterative-incremental build in increments partial delivery is useful architecture drift
V-model development paired with test levels high assurance systems can be rigid
RAD rapid prototyping UI/business apps needing fast feedback weak architecture if rushed
Spiral risk-driven cycles high-risk complex systems management overhead
Agile adaptive iterations changing requirements weak if discipline is missing
TDD tests before code clear unit behavior poor tests can mislead

Agile and Scrum

Agile values:

  • Working software.
  • Customer collaboration.
  • Responding to change.
  • Individuals and interactions.

Scrum concepts:

  • Product backlog: ordered work list.
  • Sprint backlog: selected work for the sprint.
  • Sprint planning: decide what to build.
  • Daily sync: surface blockers.
  • Sprint review: inspect product increment.
  • Retrospective: improve process.
  • Burndown chart: remaining work trend.

Important distinction:

  • Agile is not "no planning".
  • Agile is short-feedback planning with frequent inspection.

Verification and Validation

Term Question
Verification Are we building the product right?
Validation Are we building the right product?

Verification includes reviews, static analysis, and tests against spec.

Validation checks whether the result solves the user's real problem.

Testing Vocabulary

Term Meaning
Error human mistake
Fault/bug defect in artifact
Failure observed incorrect behavior
Test case inputs, steps, expected result
Test suite collection of test cases
Oracle way to decide pass/fail
Regression previously working behavior breaks

Black-Box Testing

Black-box testing uses external behavior without looking at implementation.

Techniques:

  • Equivalence partitioning.
  • Boundary value analysis.
  • Decision table testing.
  • State transition testing.
  • Use-case based testing.
Technique Use
Equivalence partitioning group inputs expected to behave similarly
Boundary value analysis test edges such as min, max, just inside/outside
Decision table test combinations of conditions and actions
State transition test valid/invalid state changes

Sample: if valid age is 18..60, test 17, 18, 60, 61, and a normal middle value.

White-Box Testing

White-box testing uses implementation knowledge.

Coverage types:

  • Statement coverage.
  • Branch/decision coverage.
  • Condition coverage.
  • Path coverage.
  • Loop boundary coverage.

Coverage is useful but not sufficient. A test can execute code without asserting meaningful behavior.

Test Levels

Level Focus
Unit small function/class
Integration interactions between components
System whole system behavior
Acceptance user/business expectations
Regression protect previous behavior
Smoke quick build sanity
Performance latency, throughput, resource use
Security abuse cases and vulnerabilities

Test Plan

A good test plan states:

  • Scope.
  • Features under test.
  • Features not under test.
  • Test strategy.
  • Test data.
  • Environment.
  • Entry and exit criteria.
  • Risks.
  • Responsibilities.

For interview answers, connect test cases to requirements and risks.

Debugging

Practical debugging flow:

  1. Reproduce the failure.
  2. Minimize the input.
  3. Form a hypothesis.
  4. Inspect logs/state.
  5. Add a failing test if possible.
  6. Fix the root cause.
  7. Keep the regression test.

Avoid random changes. Debugging is controlled hypothesis testing.

Bug Tracking

Useful bug report:

  • Title.
  • Steps to reproduce.
  • Expected behavior.
  • Actual behavior.
  • Environment.
  • Severity and priority.
  • Logs/screenshots where relevant.

Severity is impact. Priority is scheduling.

Maintenance

Maintenance types:

Type Meaning
Corrective fix defects
Adaptive adjust to changed environment
Perfective improve performance/usability/features
Preventive refactor to reduce future risk

Most real software cost happens after initial release. Design for maintainability matters.

Version Control and Release

Version control supports:

  • History.
  • Collaboration.
  • Branching.
  • Code review.
  • Rollback.
  • Traceability.

Release discipline:

  • Versioning.
  • Changelog.
  • Automated checks.
  • Deployment plan.
  • Rollback plan.
  • Monitoring after deployment.

Project Management

The core tradeoff triangle:

  • Scope.
  • Time.
  • Cost.

Quality is affected by all three. If scope grows without changing time or cost, quality usually suffers.

Project manager responsibilities:

  • Clarify goals.
  • Track progress and risks.
  • Manage dependencies.
  • Communicate status.
  • Support people.
  • Resolve tradeoffs.

People management matters because communication failures become engineering failures.

Smart Software Engineering

AI in software engineering:

  • Code search and completion.
  • Test generation.
  • Bug triage.
  • Requirements clustering.
  • Static-analysis assistance.
  • Documentation assistance.

Software engineering for AI:

  • Data quality and versioning.
  • Model evaluation.
  • Bias and drift monitoring.
  • Reproducibility.
  • Explainability.
  • Deployment and rollback.

AI engineering:

  • Treat model, data, prompts, evaluation, and product behavior as one engineered system.
  • Use benchmarks and human review.
  • Monitor behavior in production.

Checkpoints

  • Verification vs validation?
  • Waterfall vs iterative development?
  • Why is boundary value analysis effective?
  • Statement coverage vs branch coverage?
  • What belongs in a test plan?
  • Severity vs priority?
  • Corrective vs adaptive maintenance?
  • What happens if scope grows but time and cost stay fixed?
  • What extra engineering concerns appear in AI systems?
