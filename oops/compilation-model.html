The C++ Compilation Model

5 min read · source: oops_cpp/self_notes/cpp_notes.md

  • Source file (.cpp) → your human-written C++ code

  • Header file (.h/.hpp) → declarations/templates/macros shared across files

  • Preprocessor runs before compilation

    • handles #include, #define, conditional compilation
    • basically a giant text-copying machine 🧩

  • #include <file>

    • literally pastes file contents into current file before compilation
    • no magic module system in old-school C++

  • Translation Unit

    • result after preprocessing a .cpp file
    • one .cpp + all recursively included headers
    • compiler compiles each translation unit independently

  • Compilation pipeline

    1. Preprocessing
    2. Lexing/tokenization
    3. Parsing
    4. AST generation
    5. Semantic analysis/type checking
    6. Optimization
    7. Assembly generation
    8. Object file generation
    9. Linking

  • AST (Abstract Syntax Tree)

    • tree representation of program structure
    • compiler uses this for semantic analysis/optimizations
    • example: a + b * c becomes a tree with + root and * subtree

  • Object file (.o/.obj)

    • compiled machine code but not complete executable yet
    • may contain unresolved symbols (printf, functions from other files)
    • contains:
      • machine instructions
      • symbol table
      • relocation info
      • debug info (optional)

  • Linker

    • combines object files + libraries into executable
    • resolves symbol references between translation units
    • decides final memory layout

  • Static linking

    • library code copied into executable
    • bigger executable, fewer runtime deps

  • Dynamic linking

    • executable loads shared libraries at runtime (.so, .dll)
    • smaller executable

  • Symbol

    • named entity compiler/linker tracks
    • functions, globals, statics, etc.

  • Name mangling

    • C++ encodes function signatures into symbol names
    • needed for function overloading

  • extern "C"

    • disables C++ name mangling
    • useful when interfacing with C libraries

  • static at global/file scope

    • gives internal linkage
    • symbol visible only inside that translation unit
    • avoids linker conflicts
static int x = 5; // only this file can see x

  • static function at file scope

    • callable only inside that .cpp file

  • static local variable inside function

    • lifetime = entire program
    • scope = only inside function
    • stored in static/global data region, NOT stack
void foo() {
    static int cnt = 0;
}

  • normal local variable

    • stored on stack
    • recreated every function call

  • Function code usually lives in text/code segment of process memory

  • Memory layout

    • text/code segment → instructions
    • data segment → initialized globals/statics
    • BSS → uninitialized globals/statics
    • heap → dynamic allocation (new/malloc)
    • stack → function frames/local vars

  • const

    • means value cannot be modified through that variable
    • not necessarily compile-time constant
int x = 5;
const int y = x; // allowed
  • constexpr
    • must be evaluatable at compile time
int x = 5;
constexpr int z = x; // error

  • consteval (C++20)

    • function MUST run at compile time

  • ODR (One Definition Rule)

    • entity should have only one definition across program
    • violating it causes linker chaos ☄️

  • Inline function

    • allows multiple identical definitions across translation units
    • modern meaning is more about ODR/linkage than performance

  • Templates are usually defined in headers

    • compiler needs full template definition during instantiation

  • Header guards / #pragma once

    • prevent duplicate inclusion
#ifndef MY_HEADER
#define MY_HEADER
#endif

  • VTable

    • table of virtual function pointers for polymorphism

  • VPtr

    • hidden pointer inside polymorphic objects pointing to vtable

  • Virtual dispatch

    • resolved at runtime using vtable lookup

  • Pure virtual function

virtual void foo() = 0;

  • makes class abstract

  • Static member variable in class

    • shared across all objects

  • Static member function

    • no this pointer
    • can access only static members directly

  • This pointer

    • hidden pointer passed to non-static member functions

  • RAII

    • resource tied to object lifetime
    • constructor acquires, destructor releases

  • Constructor

    • initializes object

  • Destructor

    • cleanup when object dies

  • Copy constructor vs move constructor

    • copy duplicates resources
    • move transfers ownership

  • Rule of 3/5/0

    • if managing resources manually, special member functions matter

  • Lvalue

    • has persistent identity/address

  • Rvalue

    • temporary object

  • std::move

    • casts to rvalue reference
    • does NOT move by itself

  • Stack unwinding

    • destructors called automatically during exception propagation

  • new

    • allocates memory + constructs object

  • delete

    • destroys object + frees memory

  • malloc/free

    • raw memory only, no constructors/destructors

  • Undefined Behavior (UB)

    • compiler allowed to assume impossible things
    • can produce spooky action at a distance 👻

Examples:

  • out-of-bounds access

  • use-after-free

  • signed integer overflow

  • dereferencing null

  • Segmentation fault

    • OS kills process for invalid memory access

  • Compiler optimizations (-O2/-O3)

    • reorder/remove code assuming no UB

  • Dead Code Elimination

    • removes code with no observable effect

  • Constant Folding

    • compile-time evaluation of expressions

  • Inlining

    • replace function call with function body

  • ABI (Application Binary Interface)

    • calling conventions, object layout, binary compatibility rules

  • Calling convention

    • defines:
      • argument passing
      • return value handling
      • stack cleanup

  • Stack frame

    • function call bookkeeping:
      • return address
      • locals
      • saved registers

  • Recursion

    • creates new stack frame per call

  • Stack overflow

    • too many stack frames

  • Heap fragmentation

    • memory becomes split into tiny unusable chunks

  • Smart pointers

    • unique_ptr → exclusive ownership
    • shared_ptr → reference counted ownership
    • weak_ptr → non-owning observer

  • volatile

    • prevents certain compiler optimizations
    • NOT for thread synchronization

  • atomic

    • lock-free synchronized operations between threads

  • Data race

    • unsynchronized concurrent access with at least one write
    • UB in C++

  • Mutex

    • mutual exclusion lock for critical sections

  • Cache line

    • smallest unit CPU cache transfers (typically 64B)

  • False sharing

    • unrelated variables on same cache line causing contention

  • Syscall

    • transition from user mode → kernel mode

  • Context switch

    • OS switches running thread/process

  • Process vs thread

    • process = isolated memory space
    • threads share process memory

  • ELF / PE / Mach-O

    • executable formats on Linux / Windows / macOS respectively

  • a.out

    • traditional default executable name on Unix-like systems

  • nm, objdump, readelf, otool

    • useful binary inspection tools 🔬
