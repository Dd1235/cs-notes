Virtual Functions & Dynamic Binding

2 min read · source: oops_cpp/self_notes/14.md

  • Staty type of the object is the type declared for the object while writing the code

  • Compiler sees static type

  • Dynamic type is determined by the type of hte objet to which it currently refers. Compiler does not see dynamic type.

Animal* a;

if (x > 0)
    a = new Dog();
else
    a = new Cat();

a->speak();
  • In the above code, static type of a is Animal* but dynamic type of a is Dog* or Cat* depending on the value of x.
graph LR
    A["Animal* a"] --> OBJ["Dog object<br/>[ vptr | Dog data ]"]
    OBJ -->|"vptr"| VT["Dog vtable"]
    VT --> S1["slot 0: &Dog::speak"]
    VT --> S2["slot 1: &Dog::~Dog"]
    S1 -.->|"a->speak() = two loads + indirect call"| CODE["Dog::speak() code"]

  • if the destructor is not virtual in a polymorphic hierarchy, it leads to slicing. Destructor must be declared virtual in the base class.

  • shape -> polygon and closed conics
    triangle quadrilateral -> polygon
    ellipse circle -> closed conics

we want to have a polymorphic draw() function for the hierarchy

0 draw() vritual everywhere, we cannot define the body of draw() in root, as we don't have an algorithm to draw an arbitrary shape.

=> pure virtual function, has a signature but not body

vritual void draw() = 0;

  • if even a child doesn't have implementation, simply don't override it will you get in the inheritance hierarchy where draw() is implemented.

  • Abstract base class - at least one pure virtual class, cannot have an object of this class.

  • for a collection to loop over, say the shapes i < sizeof(arr) / sizeof(Shapes *)

  • dispatching based on dynamic type

  • a pure virtual function can still have a body. Because this can be called in a derived class. Can put common code there.

virtual void draw() = 0 {
    cout << "Shapes: init brush\n";
}
// now call using Shape::draw() in derived class
  • not all compilers have the support for this body. If is it not, just define it outside the class
