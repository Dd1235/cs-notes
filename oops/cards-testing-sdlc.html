Testing, SDLC & SE Flashcards

6 min read · source: oops_cpp/flashcards/06_testing_sdlc_se.md

S001

Tags: software-engineering

Prompt: Why do software projects fail?

Answer: Unclear requirements, poor communication, underestimated complexity, weak design boundaries, inadequate testing, uncontrolled change, and schedule pressure that creates technical debt.

S002

Tags: software-engineering

Prompt: What is the engineering response to software complexity?

Answer: Understand requirements, model before building, design for change, test continuously, and manage scope/time/cost/quality.

S003

Tags: sdlc

Prompt: Typical SDLC stages?

Answer: Requirement gathering/analysis, design, implementation, testing, deployment, and maintenance.

S004

Tags: requirements

Prompt: Qualities of good requirements?

Answer: Clear, testable, consistent, complete enough for the iteration, traceable, and prioritized.

S005

Tags: requirements

Prompt: Functional vs non-functional requirement?

Answer: Functional describes what the system does. Non-functional describes quality constraints such as latency, security, reliability, or usability.

S006

Tags: requirements

Prompt: Turn "system should be fast" into a testable requirement.

Answer: Example: "Search returns results within 200 ms for 95 percent of requests under target load."

S007

Tags: requirements

Prompt: What are domain constraints and interface requirements?

Answer: Domain constraints are business/environment rules. Interface requirements describe interactions with external systems.

S008

Tags: sdlc-models

Prompt: Waterfall: idea, best fit, risk.

Answer: Sequential phases; best when requirements are stable; risk is late feedback.

S009

Tags: sdlc-models

Prompt: Iterative-incremental: idea, best fit, risk.

Answer: Build in increments; best when partial delivery is useful; risk is architecture drift if design discipline is weak.

S010

Tags: sdlc-models

Prompt: V-model: idea, best fit, risk.

Answer: Development phases paired with corresponding test levels; best for high-assurance systems; risk is rigidity.

S011

Tags: sdlc-models

Prompt: RAD: idea, best fit, risk.

Answer: Rapid prototyping and quick feedback; best for UI/business apps; risk is weak architecture if rushed.

S012

Tags: sdlc-models

Prompt: Spiral model: idea, best fit, risk.

Answer: Risk-driven cycles; best for high-risk complex systems; risk is management overhead.

S013

Tags: sdlc-models

Prompt: Agile: idea, best fit, risk.

Answer: Adaptive iterations and frequent feedback; best for changing requirements; risk is weak execution if discipline is missing.

S014

Tags: tdd

Prompt: TDD: idea, strength, risk.

Answer: Write tests before code; helps clarify unit behavior and regression safety; poor tests can mislead and overfit implementation.

S015

Tags: agile, scrum

Prompt: Agile values emphasized in the notes?

Answer: Working software, customer collaboration, responding to change, and individuals/interactions.

S016

Tags: scrum

Prompt: Scrum artifacts/events to know.

Answer: Product backlog, sprint backlog, sprint planning, daily sync, sprint review, retrospective, and burndown chart.

S017

Tags: agile

Prompt: Correct the misconception: "Agile means no planning."

Answer: Agile means short-feedback planning with frequent inspection and adaptation, not absence of planning.

S018

Tags: verification-validation

Prompt: Verification vs validation?

Answer: Verification asks "are we building the product right?" Validation asks "are we building the right product?"

S019

Tags: testing-vocabulary

Prompt: Error, fault/bug, and failure?

Answer: Error is a human mistake. Fault/bug is a defect in an artifact. Failure is observed incorrect behavior during execution/use.

S020

Tags: testing-vocabulary

Prompt: Test case, test suite, oracle, regression?

Answer: A test case has inputs/steps/expected result. A test suite is a collection of tests. An oracle decides pass/fail. Regression is previously working behavior breaking.

S021

Tags: black-box-testing

Prompt: What is black-box testing?

Answer: Testing based on external behavior/specification without using implementation knowledge.

S022

Tags: black-box-testing

Prompt: Equivalence partitioning?

Answer: Group inputs expected to behave similarly and test representative values from each group.

S023

Tags: black-box-testing

Prompt: Boundary value analysis?

Answer: Test edges such as min, max, just inside, just outside, and typical middle values because defects often occur at boundaries.

S024

Tags: black-box-testing

Prompt: If valid age is 18..60, what values are useful boundary tests?

Answer: 17, 18, 60, 61, plus a normal middle value such as 30.

S025

Tags: black-box-testing

Prompt: Decision table testing?

Answer: Test combinations of conditions and the corresponding actions, useful when behavior depends on multiple boolean/rule inputs.

S026

Tags: black-box-testing

Prompt: State transition testing?

Answer: Test valid and invalid transitions between states, including events that should be rejected in a given state.

S027

Tags: white-box-testing

Prompt: What is white-box testing?

Answer: Testing with knowledge of implementation structure, control flow, paths, and code branches.

S028

Tags: coverage

Prompt: Name white-box coverage types.

Answer: Statement coverage, branch/decision coverage, condition coverage, path coverage, and loop boundary coverage.

S029

Tags: coverage

Prompt: Why is high coverage not sufficient?

Answer: Code can execute without meaningful assertions or without covering important input combinations and failure modes.

S030

Tags: test-levels

Prompt: Unit, integration, system, acceptance testing?

Answer: Unit tests small functions/classes. Integration tests component interactions. System tests whole-system behavior. Acceptance tests user/business expectations.

S031

Tags: test-levels

Prompt: Regression, smoke, performance, and security testing?

Answer: Regression protects old behavior. Smoke checks build sanity. Performance measures latency/throughput/resources. Security tests abuse cases/vulnerabilities.

S032

Tags: test-plan

Prompt: What belongs in a test plan?

Answer: Scope, features under/not under test, strategy, test data, environment, entry/exit criteria, risks, and responsibilities.

S033

Tags: debugging

Prompt: Practical debugging flow?

Answer: Reproduce, minimize, form hypothesis, inspect logs/state, add failing test if possible, fix root cause, keep regression test.

S034

Tags: debugging

Prompt: Why avoid random changes during debugging?

Answer: Debugging should be controlled hypothesis testing; random edits can hide the root cause and create new defects.

S035

Tags: bug-tracking

Prompt: What makes a useful bug report?

Answer: Title, steps to reproduce, expected behavior, actual behavior, environment, severity/priority, and logs/screenshots when relevant.

S036

Tags: bug-tracking

Prompt: Severity vs priority?

Answer: Severity is impact. Priority is scheduling/urgency.

S037

Tags: maintenance

Prompt: Corrective, adaptive, perfective, preventive maintenance?

Answer: Corrective fixes defects. Adaptive adjusts to environment changes. Perfective improves performance/usability/features. Preventive refactors to reduce future risk.

S038

Tags: maintenance

Prompt: Why does maintainability matter?

Answer: Most real software cost occurs after initial release, so designs must support change, testing, and comprehension.

S039

Tags: version-control

Prompt: What does version control support?

Answer: History, collaboration, branching, code review, rollback, and traceability.

S040

Tags: release

Prompt: What belongs in release discipline?

Answer: Versioning, changelog, automated checks, deployment plan, rollback plan, and post-deployment monitoring.

S041

Tags: project-management

Prompt: What is the project management triangle, and where does quality fit?

Answer: Scope, time, and cost trade off. Quality is affected by all three; if scope grows while time/cost stay fixed, quality usually suffers.

S042

Tags: project-management

Prompt: Responsibilities of a project manager?

Answer: Clarify goals, track progress/risks, manage dependencies, communicate status, support people, and resolve tradeoffs.

S043

Tags: smart-se

Prompt: Examples of AI in software engineering.

Answer: Code search/completion, test generation, bug triage, requirements clustering, static-analysis assistance, and documentation help.

S044

Tags: ai-engineering

Prompt: Extra engineering concerns for AI systems?

Answer: Data quality/versioning, model evaluation, bias/drift monitoring, reproducibility, explainability, deployment/rollback, benchmarks, human review, and production monitoring.

S045

Tags: ai-engineering

Prompt: What does "treat model, data, prompts, evaluation, and product behavior as one engineered system" mean?

Answer: AI behavior is not only code. Data, model choice, prompt/configuration, evaluation, monitoring, and user-facing behavior must be versioned and tested together.
