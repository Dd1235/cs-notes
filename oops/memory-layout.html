Memory Layout, Alignment & Padding

2 min read · source: oops_cpp/self_notes/memory_layout.md

Systems companies ask sizeof questions not for trivia but to check you can see memory. Two pictures to hold: the process layout, and the inside of a struct.

Picture 1 — the process

From low to high addresses: text (code, read-only) → data (initialized globals) → bss (zero-initialized globals — costs nothing in the binary) → heap (grows up, new/malloc) → shared library mappings → stack (grows down, frames with locals + return addresses).

  • Stack allocation is a pointer bump — effectively free; heap allocation is a data structure search (dynamic memory notes). This asymmetry is half of C++ performance folklore.
  • Stack overflow = recursing past the guard page. Dangling stack pointer = returning &local — the frame is gone.
  • Where does static int x inside a function live? data/bss, not the stack — one instance, program lifetime.

Picture 2 — inside a struct

Hardware fetches a T fastest at addresses divisible by alignof(T) (int → 4, double → 8). The compiler inserts padding so every member is aligned, and the struct's total size is a multiple of its strictest member's alignment (so arrays stay aligned).

struct Bad  { char a;  // 1 byte + 3 padding
              int  b;  // 4
              char c;  // 1 + 3 tail padding
            };         // sizeof = 12

struct Good { int  b;  // 4
              char a, c; // 2 + 2 tail padding
            };         // sizeof = 8

Rule: order members largest-first (or group same-size) to minimize holes. sizeof(Bad) = 12, sizeof(Good) = 8 — same data, 33% smaller, and that ratio survives into arrays of millions.

Extras that score: empty struct has sizeof == 1 (distinct addresses guarantee); adding a virtual function adds a vptr (+8 on 64-bit, and alignment jumps to 8 — vtable notes); #pragma pack(1) kills padding but slows access (some rare unaligned accesses can crash on non-x86, e.g. older ARM); alignas(64) pins to cache lines.

Why quant/networking firms care

  • Cache lines are 64 bytes — a 12-byte struct fits 5 per line, an 8-byte one 8 per line: fewer misses in hot loops, and misses are ~100× register math.
  • False sharing: two threads writing different variables in the same line ping-pong it between cores — alignas(64) on per-thread counters is the fix (Citadel loves this).
  • Packets and wire formats are packed structs — Arista asks struct-of-headers questions naturally.

Drill

  1. struct { char a; double d; int i; } — sizeof? Reorder for minimum. (24 → 16.)
  2. Why does sizeof of a class gain 8 (not 1) when you add its first virtual function on x86-64?
  3. Two atomics on one cache line, two threads hammering them — what happens and what's the one-word name? (False sharing.)
