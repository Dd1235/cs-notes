Operator Overloading

5 min read · source: oops_cpp/self_notes/11.md

Operator function

  • global function A operator+(const A& a, const A& b)
  • member function A operator+(const A& other) const
  • friend function friend A operator+(const A& a, const A& b)
Complex a(1,2);
Complex b(3,4);

Complex c = a + b;

// now how will this be allowed? I need to overload to allow adding any float to complex number, here these should be added to the real part of the complex number
Complex d = 2 + a;
Complex e = a + 2;
  • cout is an stream object, cin is an istream object
  • cin >> d, cout << d should also work out.
  • we can extend these operators.

explicit before the constructor, I don't want to allow explicit conversion.

  • say in constructor I pass in default values Complex(float r = 0, float i = 0), then if I do d3 = d1 + d2, and d1 is a float, it can just be converted to a complex. Don't do that, the + has been extended output to add a double d to a complex number.

3 separate overloading are provided so use that.

Global is not the right solution by keeping things public.

I can overload the operator as a member function, but I can't do 2 + a, because 2 is not an object of class A, so I need to make it a friend function.

friend Complex operator+(double d, const Complex& c) { return Complex(d + c.real, c.imag); }

Overloading IO operators

cout << a << b; // (cout << a) << b; // left associative

Result of the streaming as to be ostream objecct itself;

int a, b, c = 10; // b = c happens first, result of assignment expression is reference to b, a = (reference to b)

(a = b) = c;

here b does not change, a gets finally assigned value of c.

Right associative

=, +=, -=, *=

Left associative

+, -, *, /, <<, >>

conceptually, a - b - c has to be a - b first, then -c, not a - (b - c)

same for a / b / c

expr => expr + term so naturall builds left associative parse trees

// global function
ostream& operator << (ostream& os, const Complex& a){

}

// member function in ostream, this is not acceptable
ostream& ostream::operator<<(const Complex& a);

// member function in complex, invocation of streaming will change to d << cout; left operand is invoking object
ostream& Complex::operator<<(ostream& os);

object to write is passed by constant reference, return by reference for ostream object for chaining.

io operators cannot be overloaded by member functions.
operator output stream cannot be written as member function.

why not global function? it breaks encapsulation, a.re and a.im need to be made public so that os << a.re << " + " << a.im << "i"; return os

Must stick to parameter passing conventions ( built in types by value and udt by constant reference)

  • decide on return type base on natural semantics, like pre incrementn and post increment

  • effect of casting on operands

by built in types by value? because of how compiler works. reference still has to alloctate memory. any reference has overhead in terms of memory. majority of built in type, like int type, are sized to fit into register. call by value? mostly no parameter copying, just reuse same register. for user defined types, they are usually larger than built in types, so pass by reference to avoid copying.
