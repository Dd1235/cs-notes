Sharding vs Partitioning

3 min read · source: dbms/sharding.md

The terms are often used interchangeably in industry, but there is a useful distinction.

Partitioning

Partitioning means splitting data into smaller pieces.

You can partition:

  • within one database server
  • across multiple database servers

Example:

Users

1
2
3
4
5
6

Partition by user ID:

Partition A: 1,2,3
Partition B: 4,5,6

The application may not even know partitions exist.

Vertical vs Horizontal Partitioning

Vertical Partitioning

Split by columns.

Users

id
name
email
profile_picture
bio

becomes:

UsersBasic
id
name
email

UsersProfile
id
profile_picture
bio

Useful when some columns are large or rarely accessed.

Horizontal Partitioning

Split by rows.

Users

1
2
3
...
1000000

becomes:

Partition 1: ids 1-100000
Partition 2: ids 100001-200000
...

This is what most people mean by "partitioning".

Sharding

Sharding is a special case of horizontal partitioning where partitions live on different machines.

Shard 1 (Server A)
Users 1-1,000,000

Shard 2 (Server B)
Users 1,000,001-2,000,000

Shard 3 (Server C)
Users 2,000,001-3,000,000
graph TD
    APP["application"] --> RT["router / shard map<br/>shard = f(user_id)"]
    RT --> S1["Server A<br/>users 1 – 1M"]
    RT --> S2["Server B<br/>users 1M – 2M"]
    RT --> S3["Server C<br/>users 2M – 3M"]
    S1 -.X.- S2
    S2 -.X.- S3
    linkStyle 4,5 stroke:#B05730,stroke-dasharray:4

The dashed links are the catch: queries that need two shards (joins, transactions, aggregates) no longer happen inside one database.

Each shard owns part of the data.

Simple analogy

Partitioning:

One cupboard
├── drawer A
├── drawer B
└── drawer C

Sharding:

House A → drawer A
House B → drawer B
House C → drawer C

Why shard?

Single machine limitations:

CPU
RAM
Disk
Network

Eventually one database server cannot hold:

10 TB
100 million users

So data is distributed.

Sharding strategies

Range-based

1-1000000   -> Shard A
1000001-... -> Shard B

Easy but can cause hotspots.

Hash-based

hash(user_id) % N

Example:

hash(123)%4 = 2

goes to shard 2.

More balanced.

Directory-based

Lookup service:

user123 -> shard5
user456 -> shard2

Flexible but requires metadata management.

Example: PostgreSQL

Within a single PostgreSQL instance:

PARTITION BY RANGE(order_date)

creates partitions:

orders_2024
orders_2025
orders_2026

This is partitioning.

Still one database server.

Example: Instagram / Facebook style

DB Server 1
DB Server 2
DB Server 3
DB Server 4

Users distributed by hash.

This is sharding.

Interview answer

A concise answer:

Partitioning means splitting a table into smaller pieces for manageability or performance. It can happen within a single database instance. Sharding is horizontal partitioning where each partition is stored on a different machine, allowing the database to scale beyond the resources of a single server.

A common follow-up:

Is every shard a partition?

Yes.

Sharding ⊂ Horizontal Partitioning

Every shard is a partition, but not every partition is a shard.
