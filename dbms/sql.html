For SQL interview purposes, the logical query processing order is:

SELECT ...
FROM ...
JOIN ...
WHERE ...
GROUP BY ...
HAVING ...
ORDER BY ...
LIMIT ...

is not the actual execution order.

The logical order is:

  1. FROM
  2. JOIN
  3. WHERE
  4. GROUP BY
  5. HAVING
  6. SELECT
  7. DISTINCT
  8. ORDER BY
  9. LIMIT/OFFSET

Example

Table: Employees

id dept salary
1 Eng 100
2 Eng 200
3 HR 150
4 HR 250

Query:

SELECT dept, AVG(salary) AS avg_sal
FROM Employees
WHERE salary > 100
GROUP BY dept
HAVING AVG(salary) > 180
ORDER BY avg_sal DESC;

Step 1: FROM

Start with all rows.

id dept salary
1 Eng 100
2 Eng 200
3 HR 150
4 HR 250

Step 2: WHERE

WHERE salary > 100

Removes rows before grouping.

id dept salary
2 Eng 200
3 HR 150
4 HR 250

Step 3: GROUP BY

GROUP BY dept

Creates groups:

Eng -> [200]
HR  -> [150, 250]

Step 4: HAVING

Compute aggregates and filter groups.

HAVING AVG(salary) > 180
Eng avg = 200
HR  avg = 200

Both survive.

Step 5: SELECT

SELECT dept, AVG(salary)

Result:

dept avg_sal
Eng 200
HR 200

Step 6: ORDER BY

ORDER BY avg_sal DESC

Sorts final rows.

WHERE vs HAVING

This is one of the most common interview questions.

WHERE

Filters rows before grouping.

SELECT dept, COUNT(*)
FROM Employees
WHERE salary > 100
GROUP BY dept;

Only employees with salary > 100 participate in grouping.

HAVING

Filters groups after grouping.

SELECT dept, COUNT(*)
FROM Employees
GROUP BY dept
HAVING COUNT(*) > 10;

All rows participate in grouping first, then groups with ≤ 10 employees are removed.

Why this is invalid

SELECT dept, COUNT(*)
FROM Employees
WHERE COUNT(*) > 10
GROUP BY dept;

WHERE runs before grouping, so COUNT(*) does not exist yet.

Use:

SELECT dept, COUNT(*)
FROM Employees
GROUP BY dept
HAVING COUNT(*) > 10;

Interview One-Liner

  • WHERE filters rows.
  • GROUP BY creates groups.
  • HAVING filters groups.

Think:

Rows --> WHERE --> Groups --> HAVING --> Final Result
