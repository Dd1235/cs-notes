Concurrency Control Basics

1 min read · source: dbms/concurrecy_control.md

Pessimistic vs optimistic

  • Pessimistic — assume conflicts will happen: take a lock before touching the data (SELECT ... FOR UPDATE, 2PL). Nobody else can interfere while you hold it; the cost is blocking and possible deadlocks.
  • Optimistic — assume conflicts are rare: read freely, and at commit time validate that what you read hasn't changed; if it has, just roll back and retry. Often done at the application level with a version column: UPDATE ... SET v = v+1 WHERE id = ? AND v = ? — zero rows updated means someone beat you.
  • Rule of thumb: if you can give up some consistency (tolerate retries), go with optimistic — it wins under low contention because nobody ever waits.

The integrity taxonomy

  1. Entity integrity — every row is uniquely identifiable; achieved using a primary key.
  2. Referential integrity — relationships between tables remain valid; foreign keys.
  3. Domain integrity — column values are constrained; achieved using NOT NULL, CHECK, DEFAULT, UNIQUE.
  4. Transactional integrity — correct even on crash; ACID.
  5. Data consistency vs integrity — consistency is about the current data obeying the state (no contradictions right now); integrity is about the standing rules, like "salary must be positive".

Interview one-liners

  • Pessimistic = lock then work; optimistic = work then check.
  • Optimistic control turns conflicts into retries — great for web apps where two users rarely edit the same row; terrible for hot counters.
  • MVCC (see its own note) is the storage trick that lets readers skip both: they read a snapshot and never conflict with anyone.
