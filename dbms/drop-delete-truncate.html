DROP vs DELETE vs TRUNCATE

4 min read · source: dbms/drop_del_truncate.md

These three are among the most common SQL interview questions.

Command Removes Data Removes Table Can Use WHERE Rollback?* Resets Identity
DELETE Yes No Yes Usually Yes No
TRUNCATE Yes (all rows) No No DB-dependent Usually Yes
DROP Yes Yes No Usually No N/A
  • Exact rollback behavior depends on the DBMS.

DELETE

Removes selected rows.

Example

DELETE FROM Employees
WHERE EmployeeID = 5;

Before:

ID Name
1 Alice
5 Bob
7 Charlie

After:

ID Name
1 Alice
7 Charlie

Delete all rows

DELETE FROM Employees;

Table remains:

Employees

but contains zero rows.

Characteristics

  • DML (Data Manipulation Language)
  • Row-by-row deletion
  • Supports WHERE
  • Fires DELETE triggers
  • Generates more logs
  • Slower for huge tables

Typical use case

DELETE FROM Sessions
WHERE ExpiryTime < NOW();

TRUNCATE

Removes all rows very efficiently.

Example

TRUNCATE TABLE Employees;

Before:

ID Name
1 Alice
2 Bob

After:

Employees
(empty)

Characteristics

  • DDL in most databases
  • Removes all rows
  • No WHERE clause
  • Usually deallocates pages instead of deleting rows one by one
  • Much faster than DELETE

Invalid

TRUNCATE TABLE Employees
WHERE ID = 5;

Not allowed.

Identity reset

Suppose:

INSERT INTO Employees VALUES (...);

created:

1
2
3

Then:

TRUNCATE TABLE Employees;

Next insert often gets:

1

again.

With DELETE:

4

usually continues.

(Database dependent but common interview answer.)

DROP

Removes the entire object.

Example

DROP TABLE Employees;

Before:

Employees

After:

Table does not exist

Characteristics

  • Removes data
  • Removes indexes
  • Removes constraints
  • Removes metadata
  • Removes table definition

Everything is gone.

After DROP

SELECT * FROM Employees;

fails because the table no longer exists.

Visual analogy

Suppose a notebook.

DELETE

Erase some pages.

Notebook still exists.

TRUNCATE

Rip out all written pages.

Notebook still exists.

DROP

Throw away the notebook.

Notebook gone.

Logging difference

DELETE

DELETE FROM Orders;

Database may log:

Delete row 1
Delete row 2
Delete row 3
...
Delete row 1,000,000

Lots of work.

TRUNCATE

Database often logs something closer to:

Deallocate these pages

Much cheaper.

Foreign Key behavior

Suppose:

Customers
Orders

and:

Orders.CustomerID
    references Customers

Then:

TRUNCATE Customers;

often fails because child rows exist.

Similarly:

DROP TABLE Customers;

may fail unless constraints are removed first.

Interviewers like this follow-up.

Most common interview answer

DELETE

DELETE FROM Employees WHERE ID=5;
  • Removes selected rows
  • Supports WHERE
  • Slower
  • Table structure remains

TRUNCATE

TRUNCATE TABLE Employees;
  • Removes all rows
  • No WHERE
  • Faster
  • Table structure remains
  • Often resets identity

DROP

DROP TABLE Employees;
  • Removes entire table
  • Data + schema removed
  • Must recreate table to use it again

One-line summary

DELETE   -> remove rows
TRUNCATE -> remove all rows quickly
DROP     -> remove the table itself

That's usually the exact level expected in SDE interviews, with bonus points if you mention:

  • DML vs DDL,
  • logging/page deallocation,
  • identity reset,
  • foreign-key restrictions.

any of these may fail because of referential integrity

ON DELETE CASCASDE
ON DELETE SET NULL
DROP TABLE Customers CASCADE;
needed

referential integrity violation.
