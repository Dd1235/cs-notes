ACID one-liners

  • Atomicity — a transaction is all-or-nothing; rollback undoes partial work.
  • Consistency — every commit moves the DB from one valid state to another (constraints hold).
  • Isolation — concurrent transactions behave as if alone; degree is tunable (isolation levels).
  • Durability — once committed, survives crash/power loss (WAL + fsync).

Normal forms, fast

  • 1NF — atomic values, no repeating groups.
  • 2NF — no partial dependency on part of a composite key.
  • 3NF — no transitive dependency (non-key → non-key).
  • BCNF — every determinant is a candidate key.
  • Why: kill insertion/update/deletion anomalies. Why denormalize: read speed, fewer joins.

Indexing

  • B+ tree: high fan-out, all data in leaves, leaves linked → great for range scans; O(log n) point lookups; used by nearly every RDBMS (and MongoDB).
  • Clustered (primary) index: table rows physically ordered by the key — one per table.
  • Secondary index: separate structure pointing into the heap/PK — in Postgres all indexes are secondary.
  • Unique index: index + uniqueness constraint enforced at write time.

Concurrency control

  • Pessimistic: lock first (2PL — growing then shrinking phase; strict 2PL holds till commit to avoid cascading aborts).
  • Optimistic (OCC): run, validate at commit, abort on conflict — wins when conflicts are rare.
  • Timestamp ordering, SSI, or just single-threaded (Redis).
  • MVCC: keep multiple versions — readers never block writers, writers never block readers. Postgres, MySQL InnoDB, Oracle; also LSM-based NoSQL where old versions are natural.
  • MVCC is a storage scheme, still paired with a concurrency protocol (Postgres = MVCC + SSI, etc.).

WAL — why it works

  • Rule: log the change before the data page hits disk; commit = log flushed.
  • Sequential appends are cheap; data pages can be flushed lazily.
  • Crash recovery: redo committed, undo uncommitted.
  • Same idea everywhere: DB WAL, filesystem journals, Kafka, event sourcing.

Scale

  • Partitioning: split a table within one database. Sharding: split data across machines; app/router must route by shard key.
  • Sharding costs: cross-shard joins, transactions, rebalancing, hot keys. Do it last.
  • N+1 problem: 1 query for a list + N queries for children. Fix: JOIN, IN (...) batch, or ORM eager loading (select_related, includes).

Drill questions

  1. Which anomaly does snapshot isolation not prevent? Walk through the on-call doctors example.
  2. Why can a B+ tree do a range scan cheaply but a hash index cannot?
  3. Why is DELETE slow and TRUNCATE fast, mechanically?
  4. Explain "readers don't block writers" — what does a reader see mid-update?
  5. Why must the WAL record reach disk before the page? What breaks otherwise?
