Primary vs Secondary Indexes

2 min read · source: dbms/pk_vs_seckey.md

Tuples have no inherent order — a relation is a set. Physical order on disk is a storage decision, and that's exactly the difference between these two kinds of index.

Clustered (primary) index

If you have a primary key in a clustered engine (MySQL InnoDB, Oracle index-organized tables), you cluster the table around that index on disk: the B+ tree's leaves are the rows, physically sorted by key. A primary-key lookup lands directly on the row. There can be only one clustered index — rows can only be in one physical order.

Secondary index

With a secondary index, your table is a mess (a heap — rows live wherever there was space), but there is a separate index like a B-tree on the side: search it, find which pages and tuples you have to go to, then read the actual rows from there. That extra hop is the price of every secondary-index lookup.

graph LR
    subgraph clustered ["Clustered — InnoDB"]
        PK["B+ tree on PK"] --> ROWS["leaves = the rows themselves"]
    end
    subgraph secondary ["Heap + secondary — Postgres"]
        IDX["B+ tree on any column"] --> TIDS["leaf: (page, offset) pointers"] --> HEAP["heap pages, unordered rows"]
    end

Postgres: all indexes are secondary — there is no primary index. Even the primary key is just a unique secondary index over an unordered heap.

Consequences worth saying in an interview

  • In InnoDB, secondary indexes store the primary key as their pointer — so a secondary lookup walks two B+ trees, and a fat primary key bloats every other index.
  • In Postgres, an update that moves a row must update every index (mitigated by HOT updates); in InnoDB only the changed indexes.
  • Covering / index-only scans: if the index already contains every column the query needs, the heap hop disappears — that's why INCLUDE columns exist.
  • Range scans over the clustering key are sequential I/O; over a secondary index they can degenerate into random I/O on the heap.
