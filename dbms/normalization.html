Normalization

9 min read · source: dbms/normalization.md

  • data integrity (eg customer id with two dob)

  • normalized tables are protected from insertion, update, and deletion anamolies.

  • a functional dependency is a constraint between two sets of attributes in a relation.

  • trivial fd, non trivial fd, ffd, partial fd, transitive fd

  • Superkey - set of all keys that determine a row

  • Candidate key - minimal superkey

  • primary keyi s a candidate unique (unique not null)

  • alternate key : all candidate keys which are not pk

  • foreign key - point to pk of another table.

  • composite key - combination of two or more columns

  • partial dependencies only arise on composite canddiate key which violate 2nf

  • surrogate key, aurogenerated on natural comb

can find superkey, and candidate key from functional dependencies.

Canonical cover or minimal cover of a set of functional dependencies - each fd has a single attribute on rhs, and minimal set on lhs

insertion anamoly - certain attributes cannot be inserted without others being present (eg. student-course table student cannot be added unless course is present)

update anamoly - duplicated data inconsistency, changing course name requires changing it for all students

deletion anamoly - deleting student enrollment causes course to also be deleted.

Normalization organizes data to minimize redundancy and avoid anomalies.

For interviews, it helps to know both the formal textbook definition and the intuition.

1NF (First Normal Form)

Textbook Definition

A relation is in 1NF if:

  • All attributes contain atomic (indivisible) values.
  • No repeating groups or arrays.

Violates 1NF

StudentID Phones
1 9999,8888
2 7777

Phones contains multiple values.

Convert to 1NF

StudentID Phone
1 9999
1 8888
2 7777

Interview One-Liner

1NF eliminates repeating groups and requires atomic values.

2NF (Second Normal Form)

Textbook Definition

A relation is in 2NF if:

  1. It is in 1NF.
  2. Every non-prime attribute is fully functionally dependent on every candidate key.

Equivalent:

No partial dependencies.

Example

Enrollment(
    StudentID,
    CourseID,
    StudentName,
    Grade
)

Candidate Key:

(StudentID, CourseID)

FDs:

(StudentID, CourseID) -> Grade
StudentID -> StudentName

Violation:

StudentID -> StudentName

depends on only part of the key.

Decomposition

Students(
    StudentID,
    StudentName
)

Enrollments(
    StudentID,
    CourseID,
    Grade
)

Interview One-Liner

2NF removes partial dependencies on composite keys.

3NF (Third Normal Form)

Textbook Definition

A relation is in 3NF if:

  1. It is in 2NF.
  2. No transitive dependency of a non-prime attribute on a candidate key.

Equivalent formal definition:

For every FD:

X -> A

either:

  • X is a superkey, or
  • A is a prime attribute.

Example

Employee(
    EmployeeID,
    DepartmentID,
    DepartmentName
)

FDs:

EmployeeID -> DepartmentID
DepartmentID -> DepartmentName

Therefore:

EmployeeID -> DepartmentName

transitively.

Violation.

Decomposition

Employee(
    EmployeeID,
    DepartmentID
)

Department(
    DepartmentID,
    DepartmentName
)

Interview One-Liner

3NF removes transitive dependencies.

BCNF (Boyce-Codd Normal Form)

Textbook Definition

For every FD:

X -> Y

X must be a superkey.

Example

Teaching(
    Student,
    Course,
    Instructor
)

FDs:

(Student, Course) -> Instructor
Instructor -> Course

Candidate keys:

(Student, Course)
(Student, Instructor)

Notice:

Instructor -> Course

Instructor is not a superkey.

Violates BCNF.

Why 3NF allows it

Course is prime (part of a candidate key).

So 3NF is satisfied.

BCNF is stricter and rejects it.

Interview One-Liner

In BCNF every determinant must be a candidate key (or superkey).

4NF (Fourth Normal Form)

Textbook Definition

A relation is in 4NF if:

  1. It is in BCNF.
  2. It contains no non-trivial multivalued dependencies.

Example

Suppose:

Student(
    Student,
    Language,
    Hobby
)

A student can independently have:

Languages:
English
French

Hobbies:
Cricket
Chess

Stored as:

Student  Language  Hobby
A        English   Cricket
A        English   Chess
A        French    Cricket
A        French    Chess

Problem:

Student ->-> Language
Student ->-> Hobby

(Multivalued dependencies)

Decomposition

StudentLanguage(
    Student,
    Language
)

StudentHobby(
    Student,
    Hobby
)

Interview One-Liner

4NF removes independent multivalued facts.

5NF (Fifth Normal Form)

Textbook Definition

A relation is in 5NF if every join dependency is implied by candidate keys.

Practical Example

Rarely asked.

Imagine:

Supplier
Part
Project

A table can be losslessly decomposed into:

Supplier-Part
Supplier-Project
Part-Project

and reconstructed.

5NF deals with such join dependencies.

Interview One-Liner

5NF removes redundancy caused by join dependencies.

What Most Interviews Actually Expect

Usually:

NF What to Remember
1NF Atomic values
2NF No partial dependency
3NF No transitive dependency
BCNF Every determinant is a superkey
4NF No multivalued dependency
5NF No join dependency

If asked to normalize a schema in an SDE interview, it is almost always a 1NF → 2NF → 3NF → BCNF problem. 4NF and 5NF are much more common in DBMS theory courses than in software engineering interviews.
Yes. A common interview question is something like:

Given the following table, identify the normal form and normalize it to 3NF/BCNF.

Starting Schema

OrderDetails(
    OrderID,
    CustomerID,
    CustomerName,
    CustomerAddress,
    ProductID,
    ProductName,
    UnitPrice,
    Quantity
)

Sample data:

OrderID CustomerID CustomerName CustomerAddress ProductID ProductName UnitPrice Quantity
1001 C1 Alice Bangalore P1 Laptop 50000 1
1001 C1 Alice Bangalore P2 Mouse 1000 2
1002 C2 Bob Hyderabad P1 Laptop 50000 1

Step 1: Find Candidate Key

Can OrderID uniquely identify a row?

No.

1001 appears twice

Can ProductID?

No.

The candidate key is:

(OrderID, ProductID)

Step 2: Find Functional Dependencies

From business rules:

CustomerID -> CustomerName, CustomerAddress

ProductID -> ProductName, UnitPrice

OrderID -> CustomerID

(OrderID, ProductID) -> Quantity

Check 1NF

Everything is atomic.

✓ 1NF

Check 2NF

Candidate key:

(OrderID, ProductID)

Look for attributes depending on only part of the key.

Product side

ProductID -> ProductName
ProductID -> UnitPrice

Depends only on ProductID.

Partial dependency.

Order side

OrderID -> CustomerID

Depends only on OrderID.

Partial dependency.

Therefore:

✗ Not in 2NF

Remove Partial Dependencies (2NF)

Split product information:

Products(
    ProductID,
    ProductName,
    UnitPrice
)

Split order information:

Orders(
    OrderID,
    CustomerID
)

Remaining:

OrderItems(
    OrderID,
    ProductID,
    Quantity
)

Current structure:

Orders
OrderItems
Products

Check 3NF

Consider Orders:

Orders(
    OrderID,
    CustomerID,
    CustomerName,
    CustomerAddress
)

FDs:

OrderID -> CustomerID

CustomerID -> CustomerName, CustomerAddress

This implies:

OrderID -> CustomerName
OrderID -> CustomerAddress

through CustomerID.

That's a transitive dependency.

Therefore:

✗ Not in 3NF

Remove Transitive Dependency

Create:

Customers(
    CustomerID,
    CustomerName,
    CustomerAddress
)

and:

Orders(
    OrderID,
    CustomerID
)

Final 3NF Schema

Customers(
    CustomerID,
    CustomerName,
    CustomerAddress
)

Products(
    ProductID,
    ProductName,
    UnitPrice
)

Orders(
    OrderID,
    CustomerID
)

OrderItems(
    OrderID,
    ProductID,
    Quantity
)

How an interviewer expects you to think

Usually:

Step 1

Find candidate key.

(OrderID, ProductID)

Step 2

Write FDs.

OrderID -> CustomerID
CustomerID -> CustomerName, Address
ProductID -> ProductName, Price
(OrderID, ProductID) -> Quantity

Step 3

Find partial dependencies.

OrderID -> CustomerID
ProductID -> ProductName
ProductID -> Price

Remove them → 2NF.

Step 4

Find transitive dependencies.

OrderID -> CustomerID
CustomerID -> Name, Address

Remove them → 3NF.

This order-management example is probably the most realistic normalization problem you'll see in SDE interviews because it resembles an actual e-commerce schema and exercises both partial and transitive dependency removal.

a foreign key establishes a relationship between two tables.
Column Type: The FK column in the child table must have the same data type as the primary key column in the parent table.
Referential Integrity: The FK ensures data consistency by either rejecting the change, setting it to NULL, or cascading the changes.
Joining Tables: Utilizing the foreign key in SQL queries helps combine related data from multiple tables.

Denormalization is the process of reducing normalization (typically for performance reasons) by introducing redundant data into one or more tables. While normalization ensures data integrity, denormalization mainly focuses on improving query performance by eliminating complex JOIN operations.

Benefits and Drawbacks
CTEs: With multiple Common Table Expressions, maintaining benefits of a normalized schema can be achieved in a denormalized setup, potentially eliminating the drawbacks.
Maintainability: Denormalization can make data harder to manage and keep consistent.
Query Performance: Queries may become more efficient, as data required from different normalized tables is now available in a single denormalized table.
Storage Efficiency: Queries that access multiple smaller tables require less disk I/O and can fit into the memory more easily than queries accessing a few larger tables.
Data Integrity: Redundant data can lead to inconsistencies if not managed properly.
Case-by-Case Considerations
Lookup Tables: Tables with relatively static data (like Status types or Location information) that are frequently joined with other tables in your system, are strong candidates for denormalization. This can make frequent lookups faster.

Reporting and Analytics: Systems that are skewed towards read operations over write operations benefit from denormalization. When designing systems for reporting or analytics, it's common practice to have dedicated read replicas that are denormalized for improved read performance.

Partitioning: In large systems, denormalization can be used to partition data across different tables or clusters to distribute load. For example, customer data can be segregated into one cluster, and order data into another, linked via denormalization.
