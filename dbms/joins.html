Joins — Semantics & Algorithms

2 min read · source: dbms/joins.md

Two separate interview questions hide under "explain joins": what rows come back (semantics), and how the engine computes them (algorithms). Answer whichever was asked.

Semantics — what comes back

Think "keep or drop the non-matches", not Venn diagrams:

  • INNER — only rows that matched. Non-matches from both sides dropped.
  • LEFT — every left row survives; unmatched ones carry NULLs for the right columns. (RIGHT is LEFT with the tables swapped; FULL keeps both sides' orphans.)
  • CROSS — every pair, no condition: n×m rows. Usually a bug wearing a feature's clothes.
  • Anti-join idiom: LEFT JOIN … WHERE right.id IS NULL = "left rows with no match" — the interview favourite (customers who never ordered).

The trap they always set: a WHERE right.col = 'x' on a LEFT JOIN silently turns it INNER (NULLs fail the filter) — move the condition into ON to keep left rows.

Algorithms — how it's computed

  • Nested loop — for each outer row, scan/probe inner. O(n·m) naive; fine when the inner probe is an index lookup (n log m). The default for small-driving-table joins.
  • Hash join — build a hash table on the smaller side, probe with the bigger. ~O(n+m), equality joins only, wants memory (spills to disk in "grace" partitions). The workhorse for big unordered inputs.
  • Merge join — both sides sorted on the key, zip them. O(n+m) after sorting — free if indexes/previous sorts already provide order; the reason ORDER BY on the join key sometimes makes the plan faster.

Rule of thumb the interviewer wants: small×indexed → nested loop; big×big equality → hash; both pre-sorted → merge.

The 30-second worked example

Orders (1M) ⋈ Customers (10k) ON customer_id: hash the 10k customers (small side!) in memory, stream the million orders through it — one pass each. Picking the big side to hash is the classic wrong answer.

Drill

  1. Customers with zero orders — write it two ways (anti-join; NOT EXISTS) and say which the optimizer likes. (Usually identical plans; NOT IN is the dangerous one — NULL semantics.)
  2. Why can't a hash join serve ON a.ts BETWEEN b.start AND b.end? What runs instead? (Hashing needs equality; nested loop / merge on ranges.)
  3. EXPLAIN shows nested loop on two million-row tables and it's slow — name two likely causes. (Missing index on inner key; optimizer misestimating row counts.)
