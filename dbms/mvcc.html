Write skew is the anomaly that makes people realize:

Snapshot Isolation is not the same as Serializable.

The key point is:

Two transactions read the same data, make decisions based on that data, but update different rows.

Since they update different rows, there is no write-write conflict for MVCC to detect.

The Classic Doctor Example

Constraint:

At least one doctor must be on call.

Table:

doctor on_call
Alice true
Bob true

Transaction T1 (Alice goes off call)

Reads:

doctor on_call
Alice true
Bob true

T1 thinks:

Bob is still on call.
Safe for me to leave.

Transaction T2 (Bob goes off call)

At the same time reads:

doctor on_call
Alice true
Bob true

T2 thinks:

Alice is still on call.
Safe for me to leave.

Updates

T1:

UPDATE doctors
SET on_call = false
WHERE doctor = 'Alice';

T2:

UPDATE doctors
SET on_call = false
WHERE doctor = 'Bob';

Notice:

T1 writes Alice row
T2 writes Bob row

Different rows.

No write conflict.

Both commit.

Final state:

doctor on_call
Alice false
Bob false

Constraint violated:

Nobody is on call.

Why MVCC Allows This

MVCC only notices:

Did two transactions modify the same row?

Here:

T1 -> Alice
T2 -> Bob

No overlap.

So both are allowed.

The problem is that the conflict was logical, not physical.

Both transactions depended on:

"someone else remains on call"

but the database only sees row updates.

Compare With Lost Update

Initial:

balance = 100

T1:

100 -> 110

T2:

100 -> 120

Both modify the same row.

MVCC detects:

write-write conflict

and aborts one.

So lost updates are prevented.

Write Skew = Read-Write Dependency Cycle

Think of it like:

T1 reads Bob
T1 writes Alice

T2 reads Alice
T2 writes Bob

Graph:

T1 ----reads----> Bob
 ^                |
 |                |
 |                v
Alice <---writes-- T2

T2 ----reads----> Alice
 ^                |
 |                |
 |                v
Bob   <---writes-- T1

Each transaction made a decision based on data the other changed.

Another Example: Bank Approval

Rule:

At least one manager must approve a loan.

Table:

manager approved
A true
B true

Both managers withdraw approval simultaneously.

Each sees:

Other manager approved.

Both set:

approved = false

Final:

No approvals.

Again:

Different rows.
No write conflict.
Constraint broken.

Why Serializable Prevents It

Serializable asks:

Could these transactions have happened one after another in some order?

Try:

T1 then T2

After T1 commits:

| Alice | false |
| Bob | true |

Now T2 would read:

Alice = false

and should not proceed.

So the execution cannot be serialized.

Database aborts one transaction.

Interview Explanation

A good answer is:

Write skew occurs under Snapshot Isolation when two transactions read overlapping data, make decisions based on that snapshot, and update different rows. Because there is no write-write conflict, both transactions can commit even though a business constraint is violated. Serializable isolation prevents this by detecting the dependency cycle and aborting one transaction.

The key distinction:

Lost Update:
    same row written twice

Write Skew:
    different rows written,
    shared constraint violated

That sentence alone is often enough to impress an interviewer because many candidates can explain lost updates but not write skew.

MVCC is not specific to relational databases.

It's a concurrency-control technique that can be used by any database that needs:

Concurrent readers
Concurrent writers
Consistent snapshots

Relational Databases Using MVCC

Common examples:

  • PostgreSQL
  • MySQL
  • Oracle Database
  • CockroachDB

These are the databases where interviewers most often discuss MVCC.

NoSQL Databases Using MVCC

Many NoSQL systems use MVCC or something very close to it.

Examples:

  • CouchDB
  • FoundationDB
  • TiKV
  • Apache HBase (versioned cells)
  • Apache Cassandra (timestamped versions, not classic MVCC)

For example, FoundationDB is essentially:

Key-value store
+
MVCC
+
Snapshot transactions

despite not being a traditional relational database.

Why MVCC Fits Naturally with LSM Systems

LSM trees already create immutable data structures:

MemTable
↓ flush
SSTable
↓ compaction
New SSTable

Old versions naturally exist for a while.

Many distributed KV stores exploit this to provide snapshot reads.

Examples:

  • TiKV
  • CockroachDB
  • YugabyteDB

What About Redis?

Classic Redis:

Single-threaded command execution

doesn't need MVCC.

A command runs:

GET
SET
INCR

to completion before the next command.

So concurrency control is largely avoided.

What About MongoDB?

Modern versions of:

MongoDB

support snapshot reads and multi-document transactions using MVCC-like techniques internally.

So even document databases increasingly adopt MVCC concepts.

Why People Associate MVCC with SQL

Historically, MVCC became famous through:

  • PostgreSQL
  • Oracle
  • InnoDB

and because it is heavily discussed when teaching:

ACID
Isolation levels
Transactions

which are traditionally relational database topics.

Interview Perspective

If asked:

Is MVCC only for relational databases?

A strong answer is:

No. MVCC is a concurrency-control mechanism rather than a relational feature. It is used by many SQL databases such as PostgreSQL and InnoDB, but also by distributed key-value stores and document databases. The core idea is maintaining multiple versions of data so readers can access a consistent snapshot without blocking writers.

A useful mental model:

B+ Tree      -> storage structure
LSM Tree     -> storage structure

MVCC         -> concurrency control

WAL          -> durability

Replication  -> distribution

These are mostly orthogonal design choices. A database can be:

B+ Tree + MVCC + WAL

(PostgreSQL)

or

LSM + MVCC + WAL

(CockroachDB/TiKV-like systems)

or

LSM + WAL + no MVCC

depending on its design goals.

Historically, before MVCC became dominant, databases mainly used lock-based concurrency control.

There are several alternatives:

1. Two-Phase Locking (2PL)

Classic textbook solution.

Transactions acquire locks:

T1:
Read A
→ shared lock(A)

Write B
→ exclusive lock(B)

Locks are held until commit.

Example:

T1 reads row X
T2 wants to update row X

T2 waits

Readers and writers can block each other.

Pros

  • Serializable
  • Conceptually simple

Cons

Reader blocks writer
Writer blocks reader
Deadlocks

This is why MVCC became popular.

2. Strict Two-Phase Locking (Strict 2PL)

Used heavily in older database systems.

Rule:

Hold write locks until commit.

Provides:

Conflict Serializability
Recoverability

Many interview questions about locking assume Strict 2PL.

3. Timestamp Ordering

Every transaction gets a timestamp.

T1 = 10
T2 = 20

Database enforces:

All operations appear in timestamp order.

Example:

If T2 writes a row first:

x = 5

and later T1 tries to write:

x = 4

T1 is aborted because it violates timestamp order.

Pros

No deadlocks.

Cons

Many aborts under contention.

4. Optimistic Concurrency Control (OCC)

Assumption:

Conflicts are rare.

Transactions run without locks.

At commit:

Validate reads/writes.

If conflict exists:

Abort transaction.

Example:

Read A = 10
Read B = 20
Compute
Commit

At commit:

Did somebody change A or B?

If yes:

Abort

Where OCC Appears

Common in:

  • Distributed databases
  • In-memory databases
  • Application-level versioning

Example pattern:

UPDATE account
SET balance = 200,
    version = version + 1
WHERE id = 1
  AND version = 5;

If affected rows = 0:

Conflict detected

This is optimistic concurrency control.

5. Serializable Snapshot Isolation (SSI)

Interesting hybrid.

Used by:

PostgreSQL

Internally:

MVCC snapshots
+
Dependency tracking
+
Transaction aborts

Not purely locking.

Not purely MVCC.

Not purely OCC.

6. Single-Threaded Execution

Used by:

Redis (classic design)

Idea:

No concurrency
No concurrency bugs

Commands execute:

SET
GET
INCR

one at a time.

No need for MVCC.

7. Event Sourcing / Append-Only Logs

Some systems avoid in-place updates entirely.

Example:

Deposit +100
Withdraw -50

instead of:

balance = 50

Conflicts are resolved at a higher layer.

Used in:

  • Kafka-based architectures
  • Some distributed systems

Not really a direct replacement for MVCC, but often changes concurrency requirements.

Why MVCC Won

Suppose:

1000 readers
10 writers

With locking:

Readers may block writers.
Writers may block readers.

With MVCC:

Readers read old versions.
Writers create new versions.

Much higher concurrency.

This is why nearly every modern general-purpose database uses some form of MVCC.

Interview Summary

If asked:

What are alternatives to MVCC?

Answer:

Technique Main Idea
Two-Phase Locking Use read/write locks
Strict 2PL Hold locks until commit
Timestamp Ordering Enforce timestamp order
OCC Validate at commit time
SSI MVCC + dependency checking
Single-threaded execution Avoid concurrency entirely

The most important comparison is:

2PL:
    wait for locks

MVCC:
    read old versions

OCC:
    detect conflicts at commit

Those three form the core concurrency-control families you'll encounter in database papers and interviews.

  • with mvcc dbms maintains multiple physical versions of a single logical object
  • when a transactionw rites, creates a new version
  • when a transaction reads, reads the newest version that existed when the transaction started
  • writers don't block readers and readers don't block writers.
  • can easily support time travel queries

design decisions:

  1. concurrency control protocol -> 2pl, time stamp ordering, optimistic concurrency control
  2. version storage
  3. garbage collection
  4. index management

snapshot isolation - providing a transaction with a consistent snapshot of the database when the transaction started. data values from the snapshot consist of only values from committed transactions and the transaction operates in complete isolation from other transactions until it finishes.

There are two ways people use the term MVCC

Simplified textbook/interview usage

People often say:

2PL vs MVCC

meaning:

Lock-based systems
vs
Snapshot-based systems

This is the common PostgreSQL vs old lock-manager discussion.

More precise systems usage (CMU notes)

MVCC is not itself a concurrency protocol.

MVCC is:

Multiple versions
+
Snapshot visibility rules
+
Version storage
+
Garbage collection
+
Index management

Then you still need a protocol to decide:

Who commits?
Who aborts?
Who wins conflicts?

That's where:

2PL
Timestamp Ordering
OCC

come in.

This is what Andy Pavlo is getting at.

Think of it like this

Suppose I tell you:

I store multiple versions of rows.

You still don't know:

Can two transactions commit?
Should one abort?
What if they conflict?

Those are concurrency-control questions.

MVCC alone doesn't answer them.

Example 1: MVCC + Timestamp Ordering

Many research databases do:

Version A (ts=10)
Version B (ts=20)
Version C (ts=30)

Readers choose versions by timestamp.

Writers obey timestamp-ordering rules.

This is literally:

MVCC + TO

Example 2: MVCC + OCC

A transaction:

Reads snapshot
Creates tentative versions

At commit:

Validate read set

If validation succeeds:

Commit

Otherwise:

Abort

This is:

MVCC + OCC

Example 3: PostgreSQL

PostgreSQL roughly uses:

MVCC
+
Row-level locks
+
Serializable Snapshot Isolation

for conflict resolution.

Not pure TO.

Not pure OCC.

Not pure 2PL.

It's a hybrid.

Why the confusion exists

Historically:

Old systems

2PL

stored only one version.

Readers blocked writers.

Writers blocked readers.

New systems

MVCC

stored many versions.

Readers stopped blocking writers.

So people started talking as if:

MVCC replaced 2PL

which is true at a very high level.

But internally:

MVCC

did not eliminate the need for a concurrency-control protocol.

It changed how reads work.

Another important correction

You asked:

So MVCC isn't the best for consistency?

Not exactly.

MVCC itself says almost nothing about consistency.

You can have:

MVCC + Read Committed

which is relatively weak.

Or:

MVCC + Serializable

which is the strongest standard SQL isolation level.

For example:

PostgreSQL uses MVCC and can provide Serializable isolation.

So MVCC is not inherently weaker.

The layered view

This is probably the cleanest interview model:

Recovery Layer
    WAL

Storage Layer
    B+Tree / LSM

Versioning Layer
    MVCC

Concurrency Protocol
    2PL / TO / OCC / SSI

Isolation Level
    RC / RR / Serializable

The CMU notes are essentially teaching that:

MVCC = versioning architecture

2PL / TO / OCC = concurrency protocol

and modern systems mix and match them.

That's the more advanced and technically precise view.

Lock granularity is about:

How big a thing does a lock protect?

The smaller the granularity, the more concurrency you get, but the more locks the DBMS must manage.

Database Hierarchy

Think of a database as:

Database
 └── Table
      └── Page
           └── Row

You can place locks at any level.

Database-Level Lock

LOCK DATABASE

One lock protects everything.

T1 locks database
T2 waits
T3 waits
T4 waits

Pros

  • Simple

Cons

  • Terrible concurrency

Almost never used for normal transactions.

Table-Level Lock

LOCK TABLE users

Example:

T1 updates user #5

but

T2 updates user #1000

must wait because entire table is locked.

Pros

  • Fewer locks

Cons

  • Unnecessary blocking

Page-Level Lock

Suppose a page contains:

Rows 1-100

Locking the page means:

Any row in page
→ protected by same lock

Historically used more often.

Nowadays row-level locking is preferred.

Row-Level Lock

Most common.

Example:

UPDATE accounts
SET balance = balance - 100
WHERE id = 5;

Locks only:

row id=5

Other rows remain available.

Pros

Maximum concurrency.

Cons

Potentially millions of locks.

Why Not Always Use Row Locks?

Imagine:

UPDATE orders
SET status='closed';

affecting:

10 million rows

Row locking means:

10 million lock objects

Huge overhead.

Sometimes a table lock is cheaper.

Lock Escalation

Classic interview topic.

Suppose:

T1 acquires:
1000 row locks
2000 row locks
5000 row locks

Eventually DBMS decides:

Forget individual locks.
Lock entire table.

This is:

Lock escalation

Used by:

  • Microsoft SQL Server

Shared vs Exclusive Locks

At every granularity you can have:

Shared (S)

Read lock.

Many readers allowed.

Exclusive (X)

Write lock.

Only one writer.

Example:

S + S  = allowed
S + X  = blocked
X + X  = blocked

The Big Problem

Suppose:

T1 locks row #5

and

T2 wants table lock

How does T2 know whether rows inside are locked?

Need a hierarchy.

This leads to:

Intention Locks

Intention Locks

Interview favorite.

Types:

IS = Intention Shared
IX = Intention Exclusive
S
X

Example:

T1 wants:

Row #5 X lock

Before doing that:

Table gets IX
Row gets X

Meaning:

"I intend to take exclusive locks below."

Now if T2 asks:

Table S lock

DBMS immediately sees:

IX exists

and can reject it.

No need to inspect every row.

Compatibility Matrix

You don't usually memorize the whole matrix for SDE interviews.

Know:

IS compatible with IS

IX compatible with IX

S conflicts with IX

X conflicts with everything

is enough.

Example Interview Question

Why use table locks?

Answer:

Lower memory overhead.
Fewer lock manager operations.

Why use row locks?

Answer:

Higher concurrency.
Less blocking.

Why intention locks?

Answer:

Allow hierarchical locking efficiently.
Avoid scanning all child locks.

Connection to MVCC

Modern databases often use:

MVCC for reads
Locks for writes

Example:

In PostgreSQL:

SELECT

usually doesn't acquire traditional read locks because MVCC provides snapshots.

But:

UPDATE ...
DELETE ...
SELECT FOR UPDATE ...

still use row-level locks.

So lock granularity remains important even in MVCC systems.

A plain-English explanation that still sounds technically strong is:

The database lets transactions lock at different levels of the hierarchy—database, table, page, or row. Intention locks are like putting a sign on a parent object saying "I plan to take locks somewhere underneath this." This allows the database to quickly determine whether a table-level lock is compatible with existing row-level locks without scanning every row lock in the table.

Then explain the two important ones:

IS (Intention Shared)

Means:

I am not locking this table itself.
I plan to take shared/read locks on some rows inside it.

Example:

Read row 5

Before taking:

S(Row 5)

the transaction takes:

IS(Table)

which tells everyone:

"I'm reading some rows in this table."

IX (Intention Exclusive)

Means:

I am not locking this table itself.
I plan to take exclusive/write locks on some rows inside it.

Example:

Update row 5

Before taking:

X(Row 5)

the transaction takes:

IX(Table)

which tells everyone:

"I'm modifying some rows in this table."

Why Not Just Lock The Row?

Suppose:

T1 updates Row 5

and therefore holds:

X(Row 5)

Now:

T2 wants S(Table)

meaning:

Read entire table

Without intention locks, the DBMS would need to ask:

Are any rows locked?
Check row 1...
Check row 2...
Check row 3...
...
Check row 10 million...

That's impractical.

Instead:

T1 already placed IX(Table)

so T2 immediately sees:

IX exists
→ cannot take S(Table)

Done in O(1).

The Most Important Compatibility Rule

People often think:

IX means writing
IX means writing
Therefore conflict

But that's wrong.

Example:

T1:
IX(Table)
X(Row 1)

T2:
IX(Table)
X(Row 2)

Both are allowed.

Why?

Because:

They are only announcing
their intention to write.

The actual conflict check happens at the row.

Row 1 ≠ Row 2

so both proceed.

This is the entire reason intention locks exist: many writers can coexist on different rows while still preventing someone from suddenly taking a table-wide lock.

What S(Table) Actually Means

When you take:

S(Table)

you're saying:

I want the entire table
to behave as one read-locked object.

Therefore:

S(Table)

cannot coexist with:

IX(Table)

because somebody is modifying rows.

Interview Version (30 seconds)

Intention locks are metadata locks used in multi-granularity locking. They tell the database that a transaction intends to acquire row-level locks underneath a table or page. This lets the lock manager determine whether a table-level lock conflicts with existing row-level activity without scanning every row lock. IS means "I intend to read rows below," IX means "I intend to write rows below." Multiple IX locks are allowed because the actual conflicts are checked at the row level, while a table-level shared or exclusive lock conflicts with IX because it affects the whole table.

That's roughly how a database engineer would explain it rather than reciting the compatibility matrix.

MVCC = multiple versions + snapshot reads

1. MVCC + 2PL / locking

Used in practical systems like PostgreSQL/InnoDB-style engines.

What happens

READ:
    read from snapshot
    usually no read lock

WRITE:
    take row/write lock
    create new version
    commit makes version visible

Example:

T1 reads x=10
T2 updates x=20
T1 still sees x=10

But if:

T1 updates row A
T2 updates row A

one waits or aborts.

Guarantees

Depends on isolation level:

Read Committed:
    each statement gets fresh snapshot

Repeatable Read:
    whole transaction gets one snapshot

Serializable:
    needs extra machinery

Key point

MVCC helps reads. Locks still handle writes.

2. MVCC + Timestamp Ordering

Common in distributed SQL / KV-style systems.

What happens

Every transaction/version has a timestamp.

x=10 @ ts=5
x=20 @ ts=9
x=30 @ ts=15

A transaction at timestamp 10 reads:

x=20

because it is the newest version ≤ 10.

Writes are checked against timestamp-order rules.

Guarantees

Can provide serializable isolation if timestamp ordering is enforced correctly.

Key point

Timestamps decide both:

which version you see
whether your write is allowed

3. MVCC + OCC

Used often in distributed/in-memory systems.

What happens

Transaction does:

1. Read from snapshot
2. Buffer writes privately
3. At commit, validate
4. If conflict: abort
5. Else: publish new versions

Example:

T1 reads A, B
T2 modifies A
T1 commits

At validation:

Did A change after my snapshot?
Yes → abort T1

Guarantees

Can provide serializable isolation if validation checks the full read/write set.

Can provide weaker snapshot isolation if it only checks write-write conflicts.

Key point

No waiting during execution; conflicts are paid for at commit time.

4. MVCC + Snapshot Isolation

This is the common “MVCC behavior” people talk about.

What happens

Transaction gets a fixed snapshot.

T1 starts
T1 sees database as of time 100

Even if other transactions commit later, T1 keeps seeing time 100.

Writes are usually checked by:

first-writer-wins

If two transactions update same row, one fails.

Guarantees

Prevents:

dirty reads
non-repeatable reads
many phantom-like effects

But allows:

write skew

So it is not always serializable.

Key point

Snapshot isolation is strong for reads, but not fully serializable.

5. MVCC + SSI

Serializable Snapshot Isolation.

Used by PostgreSQL for Serializable isolation.

What happens

Transactions still read MVCC snapshots.

Additionally, DB tracks dependencies:

T1 read something T2 changed
T2 read something T1 changed

If it detects a dangerous cycle, it aborts one transaction.

Guarantees

Serializable.

Prevents:

write skew
phantoms
non-serializable schedules

Key point

Looks like snapshot isolation to the programmer, but aborts transactions that would break serializability.

Summary Table

Combination Reads Writes Conflict handling Typical guarantee
MVCC + Locks Snapshot reads Row locks + new version Wait/abort RC/RR/Serializable with extras
MVCC + TO Read version by timestamp Timestamp-checked writes Abort/restart Can be Serializable
MVCC + OCC Read snapshot Buffer writes Validate at commit SI or Serializable
MVCC + SI One snapshot per txn First-writer-wins Abort same-row conflicts Not fully Serializable
MVCC + SSI Snapshot reads Track dependencies Abort dangerous cycles Serializable

Interview answer

MVCC itself means the database stores multiple versions and lets transactions read a consistent snapshot. It is not the whole isolation algorithm. Real systems combine MVCC with locks, timestamp ordering, optimistic validation, or SSI to decide which transactions can commit. The guarantees depend on the isolation level: Read Committed uses fresh snapshots per statement, Repeatable Read/Snapshot Isolation uses one snapshot per transaction, and Serializable requires extra conflict detection such as 2PL, timestamp ordering, OCC validation, or SSI.
