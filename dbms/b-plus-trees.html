B+ Trees

2 min read · source: dbms/b+tree.md

Why not a flat file?

If you store records in a file sequentially, everything is O(n):

  • you cannot insert efficiently in the middle
  • an update can overwrite in place only if it stays within the same width
  • finding one record is a linear scan
  • deleting means writing a new file without that entry/row

The B+ tree idea

The size of a node is chosen to be approximately the disk block size, because the OS reads an entire block at a time — say the block is 4 kB and one row is ~40 B, that's about 100 rows per node.

  • Every B+ tree is serialized and stored on disk; each node contains an offset, and that offset is the pointer.
  • Leaf nodes store the actual data (or pointers to it); non-leaf nodes hold only routing information (separator keys).
  • Databases care about disk page reads, not CPU comparisons — a shallow, wide tree wins even if each node needs a binary search inside.
graph TD
    R["root · keys: 40 | 70"]
    A["internal · 10 | 25"]
    B["internal · 40 | 55"]
    C["internal · 70 | 85"]
    L1["leaf 1–9"] --> L2["leaf 10–24"] --> L3["leaf 25–39"] --> L4["leaf 40–54"] --> L5["leaf 55–69"] --> L6["leaf 70–84"] --> L7["leaf 85–99"]
    R --> A & B & C
    A --> L1 & L2 & L3
    B --> L4 & L5
    C --> L6 & L7

Leaves are linked left-to-right — that chain is what makes range scans cheap.

B tree vs B+ tree

  • A B tree allows data in non-leaf nodes as well.
  • A B+ tree keeps data only in leaf nodes — this helps range queries, because actual records sit in sorted order at the leaf level and the leaf nodes are linked together: find the start of the range, then walk the sibling chain.
  • Keys-only internal nodes mean a B+ tree offers higher fanout than a B tree, which makes the tree very shallow — a 3–4 level tree can index billions of rows.

Clustered vs unclustered

  • PostgreSQL: heap table + B+ trees. Actual rows live in heap pages without any ordering, and the B+ tree is just index storage — an unclustered index.
  • MySQL (InnoDB) and Oracle use clustered / index-organized storage: the B+ tree is the table — leaves hold the full rows, physically ordered by primary key.
  • MongoDB also uses B+ trees (WiredTiger).

Interview one-liners

  • "Why B+ tree over hash index?" — hash gives O(1) point lookups but no range scans, no ordering, no prefix matches; B+ gives O(log n) everything.
  • "Why is fanout the number that matters?" — tree height = ceil(log_fanout N), and height = number of disk reads per lookup.
  • Insert into a full node → split, push the middle key up; splits cascading to the root are how the tree grows taller (rarely).
