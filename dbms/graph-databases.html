Graph Databases

1 min read · source: dbms/graphdb.md

Watch: graph databases lecture — todo. Primer below until then.

The model

A property graph: nodes and edges, both carrying key-value properties, edges typed and directed.

graph LR
    A(("person<br/>name: Asha")) -- "FOLLOWS since 2024" --> B(("person<br/>name: Ravi"))
    B -- "WROTE" --> P["post<br/>title: B+ trees"]
    A -- "LIKED" --> P

Why not just SQL?

A relationship in an RDBMS is a join through an index: find friends-of-friends and you're doing joins whose cost grows with the total table size. A native graph DB stores adjacency directly on the node (index-free adjacency): hopping an edge is a pointer dereference, so traversal cost scales with the size of the neighborhood you touch, not the whole dataset.

The rule of thumb: if your queries are shaped like "start here, walk 3+ hops, with conditions on the path" (recommendations, fraud rings, org charts, dependency graphs), graphs win. If your queries are aggregations over columns, stay relational.

Query language taste (Cypher / Neo4j)

// friends-of-friends who liked the same post
MATCH (me:Person {name: "Asha"})-[:FOLLOWS*2]->(fof)-[:LIKED]->(p:Post)<-[:LIKED]-(me)
RETURN DISTINCT fof.name

The same query in SQL is a self-join pyramid; at variable depth (*2..5) it needs recursive CTEs.

Names to drop

  • Neo4j — the default native graph DB, Cypher.
  • Amazon Neptune — managed, supports both property graph (Gremlin/openCypher) and RDF/SPARQL.
  • Postgres can fake small graphs with WITH RECURSIVE; that's often the honest answer for one graph-ish feature in a relational app.
