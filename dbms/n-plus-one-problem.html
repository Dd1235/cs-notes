The N+1 Query Problem

2 min read · source: dbms/n+1problem.md

The N+1 query problem occurs when you execute:

1 query to fetch N rows
+
N additional queries for related data

instead of fetching everything in a small number of queries.

It's extremely common in ORMs like Hibernate, Entity Framework, and Django ORM.

Example

Tables:

Users
-----
id
name

Orders
------
id
user_id
amount

Suppose you want:

Order 1 -> Alice
Order 2 -> Bob
Order 3 -> Alice

Bad Approach (N+1)

First query:

SELECT * FROM Orders;

Returns N orders.

For each order:

SELECT * FROM Users WHERE id = ?;

If there are 100 orders:

1 query for orders
100 queries for users

Total:

101 queries

This is the N+1 problem.

Why is it bad?

The issue is usually not SQL execution.

The issue is:

Network round trips
Connection overhead
Query parsing
Lock acquisition
Context switching

100 tiny queries are often much slower than 1 larger query.

Example in ORM

orders = Order.objects.all()

for order in orders:
    print(order.user.name)

Looks innocent.

Behind the scenes:

SELECT * FROM Orders;

Then:

SELECT * FROM Users WHERE id=1;
SELECT * FROM Users WHERE id=2;
SELECT * FROM Users WHERE id=1;
...

Potentially hundreds of queries.

Solution 1: JOIN

SELECT
    o.id,
    u.name,
    o.amount
FROM Orders o
JOIN Users u
ON o.user_id = u.id;

Now:

1 query

instead of:

N+1 queries

Solution 2: Eager Loading

ORMs provide:

Order.objects.select_related("user")

or

JOIN FETCH

which internally generates a JOIN.

Solution 3: Batch Fetching

Instead of:

SELECT * FROM Users WHERE id=1;
SELECT * FROM Users WHERE id=2;
SELECT * FROM Users WHERE id=3;

do:

SELECT *
FROM Users
WHERE id IN (1,2,3);

Now:

2 queries total

instead of:

N+1

Real-world example

Imagine a social media feed.

Fetch:

SELECT * FROM Posts LIMIT 100;

Then for each post:

SELECT * FROM Users WHERE id=?;

For 100 posts:

101 queries

A common production performance bug.

Interview Answer

The N+1 problem occurs when an application executes one query to fetch N records and then executes an additional query for each record to fetch related data. For N records this results in N+1 database queries, causing excessive round trips and poor performance. It is typically solved using JOINs, eager loading, or batch fetching.

Example:

1 query: fetch 100 orders
100 queries: fetch customer for each order
-----------------------------------------
101 total queries

A JOIN can reduce this to:

1 query

which is why interviewers often bring it up when discussing ORMs and database performance.
