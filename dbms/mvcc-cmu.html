MVCC — CMU Lecture Notes

1 min read · source: dbms/mvcc_cmulec.md

Core ideas from the CMU 15-445 multiversioning lecture (slides):

  • The DBMS maintains multiple physical versions of the logical tuples.
  • Writers do not block readers, readers do not block writers.
  • Use timestamps to determine visibility: a transaction sees the newest version whose begin-timestamp is ≤ its own read timestamp (and not deleted before it).
  • Allows time-travel queries if there is no garbage collection — you can ask "what did this row look like at 3pm?"
  • Financial systems would want this for auditing.

Version storage (where do old versions live?)

  • Append-only — every update writes a whole new tuple version into the same table space; versions chained by pointer (Postgres).
  • Time-travel storage — the main table keeps the newest version; old versions are copied out to a separate time-travel table.
  • Delta storage — the main table keeps the newest version; only the changed attributes (deltas) go to an undo log, and old versions are reconstructed by applying deltas backwards (MySQL InnoDB, Oracle rollback segments).

Garbage collection

Old versions that no active transaction can see must eventually be reclaimed:

  • Tuple-level — background vacuuming scans for dead versions (Postgres VACUUM).
  • Transaction-level — each transaction tracks its own old versions and hands them to GC on commit.
  • The classic failure mode: one long-running transaction pins the oldest visible timestamp, so nothing can be vacuumed and version chains grow — Postgres table bloat.

One-liner

MVCC is a storage/visibility scheme, not a full concurrency protocol — the engine still pairs it with 2PL, timestamp ordering, or OCC to order the writers (see "MVCC in Depth").
