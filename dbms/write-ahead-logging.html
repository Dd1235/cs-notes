Write-Ahead Logging

6 min read · source: dbms/wal.md

commit -> changes need to be flushed to the disk or any non volatile storage like S3. Os failure, power gone etc. also the changes are there.

engineer -> update row -> table update or index update

disk writes are complicated.
ram -> os cache -> disk cache -> disk

mode could be regular read only mode, or sync mode etc. sync mode bypass all cache and makes sure immediately flush is done so that there is no dataloss.

write ahead logging -> standard method for ensuring data integrity.

its a concept that can be applied to any database for ensuring reliability.

core idea: before making changes to actual datafiles, log these changes.

sync -> write to log
periodic -> actual change to disk (say heap pages or the b+ trees)

advantages:

  1. no need to flush entire data on every commit
  2. recover by replaying the logs
  3. reduce the number of disk writes
  4. point in time recovery is possible with wal, a point in time snapshot

wal file is sequential so cost of logging changes is significantly lower.

log -> one append
data file -> update table, update index, tree traversal

graph LR
    TX["COMMIT"] -->|"1. append record + fsync<br/>(sequential, cheap)"| WAL["WAL segment file"]
    TX -->|"2. modify in memory"| BUF["buffer pool<br/>dirty pages"]
    BUF -.->|"3. flushed lazily, later<br/>(checkpoint)"| DATA["heap pages / B+ tree files"]
    WAL -->|"crash? replay committed,<br/>undo uncommitted"| DATA

Data integrity in WAL

  • every individual record is CRC-32 protected (we could tell if record content are connrect crc is checked during crash recovery and replication). Cyclic redundancy checksum.

  • it is enabled by default in all persistant databases.

  • its an append only file. Each file is a segment (16MB). Insert Position -> log sequence number (LSN). Byte offset into logs.

  • byte offset as the sequence number optimization crc update + set k:v

WAL (Write-Ahead Log)

The core idea:

Before modifying the actual data structure, first append the change to a durable log.

Example:

UPDATE account SET balance = 100

Database does:

1. Append update record to WAL
2. fsync WAL
3. Apply update to B-tree/page/memtable

If the machine crashes:

Replay WAL -> recover state

WAL is primarily a durability and recovery mechanism.

LSM Trees

An LSM tree often uses a WAL.

Write path:

Client write
    ↓
WAL append
    ↓
MemTable (RAM)
    ↓
SSTables on disk

Examples:

  • RocksDB
  • LevelDB
  • Apache Cassandra
  • ScyllaDB

So:

LSM tree often contains a WAL
WAL does not imply LSM tree

A B+ tree database can also have a WAL.

Examples:

  • PostgreSQL
  • MySQL

Kafka

Kafka took the "append-only log" idea much further.

Instead of:

Log exists to recover database

Kafka says:

The log IS the database

A partition is basically:

event1
event2
event3
event4

stored sequentially on disk.

Consumers maintain offsets:

consumer A -> offset 100
consumer B -> offset 500

and replay whenever they want.

"The log is the source of truth"

This is the philosophy behind:

  • Apache Kafka
  • Event Sourcing
  • CQRS architectures

Instead of:

Database = source of truth
Log = recovery mechanism

you have:

Log = source of truth
Views/Indexes/Materialized state = derived data

Example:

UserCreated
AddressChanged
EmailChanged
SubscriptionCancelled

Current user state can be reconstructed by replaying events.

Evolution of the idea

You can think of it historically as:

WAL
  ↓
Databases realize logs are useful
  ↓
LSM trees use WAL + immutable files
  ↓
Kafka says keep only the log
  ↓
Event sourcing says derive everything from logs

A useful interview distinction:

System What is the source of truth?
PostgreSQL B+ tree tables/pages
PostgreSQL WAL Recovery log
RocksDB SSTables + MemTables
RocksDB WAL Recovery log
Kafka Append-only log itself
Event Sourcing Event log itself

So the slogan "the log is the database" is much closer to Kafka and event sourcing than to traditional WALs. WALs are usually just a mechanism to protect the real data structure; Kafka elevates the log to become the primary data structure.

CQRS stands for:

Command Query Responsibility Segregation

The idea is:

Separate the system that handles writes from the system that handles reads.

Instead of:

          +-----------+
Read  --->| Database  |<--- Write
          +-----------+

you have:

          Command Side
Write ---> Validation ---> Write Model
                                |
                                v
                            Event Log
                                |
                                v
                          Read Projections

Read -----------------> Read Model

Traditional CRUD

Suppose you're building an e-commerce site.

products
orders
users

Reads and writes both hit the same database.

Create Order
Update Order
Get Order
Search Orders
Dashboard
Analytics

all use the same tables.

This works for most applications.

Why CQRS?

Often read patterns and write patterns are completely different.

Example:

Write

Place Order
Cancel Order
Refund Order

Need:

  • Validation
  • Transactions
  • Correctness

Read

Show dashboard
Top customers
Revenue by month
Search orders

Need:

  • Fast queries
  • Denormalized views
  • Aggregations

The ideal schema for writes is usually terrible for reads.

CQRS Solution

Command Side

Handles:

CreateOrder
CancelOrder
UpdateAddress

Only writes.

May store:

Orders
OrderItems
Customers

in a normalized schema.

Query Side

Maintains separate read models.

Example:

CustomerDashboardView

{
    customer_id,
    total_orders,
    total_spent,
    latest_order
}

Precomputed.

Reads become:

SELECT *
FROM customer_dashboard
WHERE customer_id = ?

Very fast.

CQRS + Event Sourcing

This is where it gets interesting.

Instead of storing current state:

Order:
status = SHIPPED

store events:

OrderCreated
PaymentReceived
Packed
Shipped

The event log becomes the source of truth.

A read model is built from those events.

Event Log
     ↓
Projection Worker
     ↓
Read Database

This is common with:

  • Apache Kafka
  • Event-driven systems
  • High-scale financial systems

Example: Banking

Commands:

Deposit(100)
Withdraw(20)

Events:

MoneyDeposited(100)
MoneyWithdrawn(20)

Read model:

CurrentBalance = 80

If the read database dies:

Replay events
Rebuild balance

Advantages

Independent scaling

100 writes/sec
100,000 reads/sec

Scale read side separately.

Optimized schemas

Write DB:

Normalized
Transactional

Read DB:

Denormalized
Materialized views

Event replay

Rebuild projections anytime.

Useful for:

  • Analytics
  • Auditing
  • Debugging

Drawbacks

Complexity

Instead of:

App
 ↓
Database

you now have:

App
 ↓
Command Model
 ↓
Event Bus
 ↓
Projections
 ↓
Read Model

Much more infrastructure.

Eventual Consistency

After:

Place Order

the read side may not update immediately.

For a brief period:

Write succeeded
Read still shows old data

This surprises many developers.

Interview Perspective

Most systems do not need CQRS.

Good answer:

CQRS separates write operations (commands) from read operations (queries), allowing each side to use different models and storage optimized for its workload. It is often combined with event sourcing, where an append-only event log is the source of truth and read models are asynchronously built from those events. The tradeoff is additional complexity and eventual consistency.

For interviews, it's enough to understand:

CRUD:
    same model for reads and writes

CQRS:
    separate models for reads and writes

Event Sourcing:
    store events, not current state

CQRS + Event Sourcing:
    events drive read projections

Many candidates know Kafka; far fewer can clearly explain why CQRS exists and when the extra complexity is justified.
